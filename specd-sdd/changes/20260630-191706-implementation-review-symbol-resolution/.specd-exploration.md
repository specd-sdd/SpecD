# Problem statement

The user wants a new change to harden implementation review symbol resolution for TypeScript-heavy implementation links.

Current behavior in changes implementation review produces frequent false-positive stale diagnostics even when the graph is fresh and the code is still plausibly linked. The reported failure modes are:

- Ports/interfaces: the stored link points to a member signature such as `validateChangeAll`, but the graph often indexes the interface or port container instead of each member symbol.
- Rename mismatch: a link such as `toProjectStatusDto` should remain stale when the real code is now `toProjectStatusDtoFromSnapshot`.
- Barrels and re-exports: a link may point to `index.ts`, while the symbol is actually defined in a file like `host-context.ts` and only re-exported through the barrel.
- Simple symbol names: current fallback only helps for composed member notation such as `Change.transition`, `Foo#bar`, or `X::Y`; simple names do not get a second chance and are marked stale immediately.

The user explicitly framed the goal as reducing false-positive stale links without losing the signal for real stale links.

# Approach / solution outline

The brief proposes distinguishing among:

- `stale`: real breakage, symbol no longer exists or the file/target is wrong
- `unresolved`: the link is imprecise, but the implementation is still plausible
- optionally `ambiguous`: multiple plausible candidates exist and the CLI should not guess

The user proposed several design options:

## Option A: smarter CLI-side resolution

In `enrichImplementationTracking`, if same-file exact symbol matching fails:

- search same symbol name within the spec workspace, not just the exact file
- if exactly one candidate exists, treat it as resolved via workspace and not stale
- follow import/export or re-export chains when the stored file is a barrel
- for interface methods, if the linked member exists in an interface AST in the linked file, treat it as plausible

## Option B: richer code-graph indexing

- index interface members as resolvable symbols such as `PortChangesMutate.validateChangeAll`
- index re-export aliases so a barrel symbol can resolve back to its real definition

## Option C: richer link semantics

- add link kinds such as `interface-member`, `re-export`, or `definition`
- make review logic branch by link kind

## Option D: hygiene and UX only

- keep the algorithm conservative, but surface `stale` vs `unresolved` vs `ambiguous`
- improve `implementation add` UX by suggesting graph symbols and encouraging definition-file links instead of barrel files

The user recommendation was:

- prefer `A + D` for the first pass because it is less invasive and gives immediate review improvements
- consider `B` if accurate graph-level dependents and impact for ports/re-exports are also desired
