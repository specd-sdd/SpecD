# Exploration Context: deprecate-ladybug-store

## Problem Statement

The Ladybug graph store backend requires a DuckDB/native-based database setup. This has introduced process lifecycle issues (native threads preventing clean Node.js process exit, requiring forced `process.exit(0)` inside CLI graph commands) and redundant wrapper architecture (`withOpenGraphProvider` in the SDK and `withProvider` in the CLI).
Now that `sqlite` is the default and only active built-in store, Ladybug can be deprecated and deleted, allowing us to remove these lifecycle wrapper workarounds.

## Approach / Solution Outline

1. **Remove Ladybug**: Completely delete LadybugDB-specific store infrastructure, tests, and its corresponding specification file.
2. **Support Explicit Resource Management (ERM)**: Introduce `Symbol.asyncDispose` inside `CodeGraphProvider`, the abstract `GraphStore`, and `SQLiteGraphStore` in `@specd/code-graph`.
3. **Refactor Hosts & Orchestrators to `await using`**:
   - Refactor CLI graph subcommands to open/close the provider using ES `await using` blocks, eliminating the need for `withProvider`.
   - Refactor SDK orchestrators (`buildProjectStatusSnapshot` and `runIndexProjectGraph`) to use `await using` blocks.
4. **Remove Redundant Wrappers**:
   - Delete `withOpenGraphProvider` from `@specd/sdk`.
   - Delete `withProvider` from `@specd/cli`.

## Design Decisions

- **`Symbol.asyncDispose` Implementation**: Will delegate directly to the asynchronous `close()` method of the provider/store.
- **Idempotency**: `close()` and `[Symbol.asyncDispose]()` must be safe to call multiple times without throwing (already standard in `SQLiteGraphStore`).
- **Signal Handling**: Retained at the host level where necessary. Short-lived CLI commands do not require custom signal trapping anymore as Node's default exit handles empty event loops cleanly when SQLite is closed.
- **`withOpenGraphProvider` removal**: Fully remove it from the SDK since `await using` makes it redundant.

## Agreements Reached

- All changes will proceed outside the callback-wrapping lifecycle helpers, moving directly to TS-native resource management.
- Spec overlap warnings with other active changes (`file-impact-covering-specs`, `graph-change-context`, `decouple-composition-factories`) were acknowledged, and we agreed to proceed.
- Added specs that referenced Ladybug in their verify/spec documentation (`core:kernel-builder`, `core:kernel`, `code-graph:sqlite-graph-store`) to cleanly remove their Ladybug dependencies.
