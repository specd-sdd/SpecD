# Exploration: fix-validate-all-dag

## Problem statement

`specd changes validate --all` does not follow the artifact dependency DAG from the active schema. It iterates `change.specIds` in declaration order and, for each spec, invokes `ValidateArtifacts` without `--artifact`, which attempts to validate **all** schema artifacts every time.

This causes:

- Spurious **dependency-blocked** failures (e.g. `design` requires aggregate `specs`/`verify` complete across all specs, but only the current spec's deltas may be ready).
- Redundant re-validation of **change-scoped** artifacts (`proposal`, `design`, `tasks`) on every spec iteration.
- Redundant re-validation of files already **`complete`**, which can interact badly with **approval/signoff drift detection** and **invalidation policies** (`downstream` default).

The user suspected `--all` "does not follow the DAG" — confirmed by code review.

## Approach / solution outline

1. **`Schema.artifactDag()`** — single canonical API on `Schema` (user preference: one method, not scattered helpers). Returns a cached `ArtifactDag` value object built from `artifacts[].requires` (cycles already rejected at schema load in `build-schema.ts`).

   Proposed `ArtifactDag` surface:
   - `roots()` — artifact ids with no/requires empty
   - `childrenOf(id)` — direct dependents (inverse of `requires`)
   - `topologicalOrder()` — parents before children, stable tie-break using schema declaration order
   - `descendantsOf(ids)` — BFS descendants for invalidation expansion

   **Rejected:** separate `domain/services/artifact-dag.ts` module — DAG is purely schema structure; `Schema` already owns artifacts and indexes.

   **Rejected:** hardcoded DAG order (proposal → specs → verify → design → tasks) — must come from schema only.

2. **`changes validate --all`** — batch driver uses `schema.artifactDag().topologicalOrder()`:
   - **`scope: change`** — validate each artifact type **once** per batch.
   - **`scope: spec`** — validate for **each** `specId` in the change.
   - **Skip files with status `complete`** (file-level, not artifact aggregate) — do not re-read/re-`markComplete`; avoids unnecessary work and reduces risk of `artifact-drift` invalidation when spec approval/signoff is active.
   - **Refresh dependency state** between DAG steps in the same batch so children see parents marked `complete` in the same pass (today `artifactVerdicts` is computed once at start of `ValidateArtifacts.execute`).

3. **Refactor DAG consumers** to use `schema.artifactDag()` (same change scope):
   - `Change._findDagDescendants` (invalidation policy `downstream`)
   - `invalidate-change` → `orderArtifactsByTraversal` / reporting order
   - CLI `renderDag` in `packages/cli/src/commands/change/status.ts` — use dag for roots/children; keep ASCII formatting in CLI

## Design decisions (agreed)

| Decision                                           | Rationale                                                                              |
| -------------------------------------------------- | -------------------------------------------------------------------------------------- |
| DAG from **schema** `requires`, not fixed list     | Custom schemas must work                                                               |
| **`Schema.artifactDag()`** only entry point        | User preference; matches `artifact()`, `workflowStep()` style                          |
| **Skip file `complete`**                           | Re-validation can trigger drift invalidation under active approval; wasteful otherwise |
| Change-scoped **once**, spec-scoped **per specId** | Matches artifact semantics                                                             |
| Refactor existing DAG ad-hoc code                  | Today duplicated in `change.ts`, `invalidate-change.ts`, `status.ts`                   |

## Invalidation behavior (investigation — for verify scenarios)

**Normal validate (no active spec/signoff approval):** does **not** cascade-invalidate; only structural validation + `markComplete`.

**With active spec approval or signoff:** before validating, compares hashes of **all** tracked files vs approval/signoff `artifactHashes`. Mismatch → `invalidate('artifact-drift')` → transition to `designing` → applies `invalidationPolicy` (default **`downstream`**: targets + DAG descendants reopened).

Manual `changes invalidate`, transition to `designing` from later states, `updateSpecIds`, repository load drift — separate paths.

**Implication for `--all`:** skipping `complete` files is a safety/performance requirement, not optional nicety.

## Rejected alternatives

- **Separate `artifact-dag.ts` service** — thin wrapper over Schema; user chose method on Schema.
- **`getArtifactDag()` naming** — prefer `artifactDag()` consistent with `artifacts()`, `artifact(id)`.
- **Validate only via `--all --artifact specs` then verify** — valid workaround today but not a product fix; skills should not rely on agents knowing the workaround.
- **Hardcode std schema order** — breaks custom schemas.
