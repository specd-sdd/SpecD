# Exploration: file-impact-covering-specs

Generated: 2026-06-07

## Problem statement

`specd graph impact --file <path>` exists and returns structural impact today:
risk level, dependency counts, affected files, changed symbols, and per-symbol
breakdown. What it does not currently surface is the reverse spec view derived
from implementation coverage links:

- which specs cover the target file
- which specs cover the affected files
- optionally, which specs cover affected symbols when symbol-level links exist

The immediate goal of this change is to close that gap for file-scoped impact
output.

## Approach / solution outline

Extend file impact so the output includes covering specs derived from reverse
spec↔code coverage links already present in the graph model.

Current state:

- `graph impact --spec` already uses requirement-aware coverage through
  `COVERS_FILE` and `COVERS_SYMBOL`
- the graph store contract already exposes reverse lookups:
  `getCoveringSpecsForFile(filePath)` and `getCoveringSpecsForSymbol(symbolId)`
- `graph impact --file` does not currently project those reverse spec links in
  its result/output

Expected direction:

1. keep the existing file-impact traversal semantics
2. enrich the file-impact result with covering specs derived from the file
   itself and likely also from affected files/symbols
3. expose that data in CLI text/json/toon output
4. update the traversal and CLI specs to make this behavior explicit

## Design decisions

- Do not create a broader “impact by change” change here. Keep this scoped to
  the missing reverse spec projection for `graph impact --file`.
- Do not widen scope to task-list generation.
- Do not widen scope to execution flows / `affectedProcesses`; that remains
  aligned with issue #39.
- Start with existing reverse coverage queries instead of changing the graph
  persistence contract. The store already appears to support what is needed.

## Agreements reached

- The missing behavior is not “file impact does not exist”; it does exist.
- The missing behavior is specifically that file impact does not surface specs
  from reverse coverage links.
- This deserves its own specd change.
- The change should be created via `specd-new`, not as an ad hoc repo edit.

## Steps / plan defined

1. Create a narrow change for file-impact covering specs.
2. Update `code-graph:traversal` to define the reverse-spec projection for
   file impact.
3. Update `cli:graph-impact` to require rendering/specifying that data in
   output.
4. Implement the code changes after design artifacts are written.

## Rejected alternatives

- Expanding this change to solve full change-level impact input
  Reason: there is already an active `graph-change-context` change that targets
  that area more directly.

- Treating the lack of specs in `graph impact --file` as data absence
  Reason: the graph/store already model reverse spec coverage; the gap looks
  like output/projection logic, not missing persistence.

- Framing the remaining issue #19 gap as “task list generation”
  Reason: that is broader and more heuristic than the concrete missing file
  impact behavior being addressed here.

## Open questions

- Should file impact show only specs covering the target file, or also specs
  covering affected files/symbols in the wider blast radius?
- Should the result distinguish:
  - specs covering the input file directly
  - specs discovered transitively through affected files
  - specs discovered via symbol-level coverage
- Should `FileImpactResult` gain an `affectedSpecs` field, a richer grouped
  structure, or both?
- Should text output group specs separately for “target file coverage” and
  “blast-radius spec coverage”?

## Key codebase observations

- `specd graph impact --file` currently returns:
  - risk level
  - direct/indirect/transitive deps
  - affected files
  - changed symbols
  - per-symbol breakdown
- `specd graph impact --spec` already performs requirement-aware spec impact via
  `analyzeSpecImpact(...)`.
- Reverse coverage methods already exist in the graph-store surface:
  - `getCoveringSpecsForFile(filePath)`
  - `getCoveringSpecsForSymbol(symbolId)`
- The graph currently contains real coverage links in this repo:
  `COVERS_FILE` and `COVERS_SYMBOL` are populated in project status / graph
  stats.
- The file-impact output gap appears to be in traversal/result projection and
  CLI rendering, not in the underlying storage capability.
