# Exploration & Discovery Context: Repository Cache Optimizations

This file serves as the working memory of the discovery conversation for implementing the dual-level cache system for SpecD repositories.

## 1. Problem Statement

The SpecD codebase is being exposed via a long-lived HTTP API that will experience concurrent client polling (e.g. 500 users polling every 5 seconds). Currently:

- `FsSpecRepository` performs recursive directory walks (`_walk`) and parses markdown files on every `list()` or `count()` query.
- `FsArchiveRepository` scans directories to ensure index validity recursively.
- `FsChangeRepository` must scan active changes folder structure.
  This constant recursive I/O blocks the Node.js Event Loop, causes file descriptor pressure, and degrades performance. We must optimize file system reads to be as close to $O(1)$ in memory as possible under normal polling conditions, while supporting local Git-based developer workflows where specs/changes are edited directly in the IDE or via CLI commands in parallel.

## 2. Approach & Solution Outline (2-Level Cache)

We will implement a two-level cache (L1 in RAM, L2 on Disk) inside the filesystem repository adapters (`Fs*Repository`).

1.  **L1 Cache (RAM):** Process-scoped memory cache inside the repository instances. Used to serve concurrent API reads in $<0.1\text{ms}$ without touching the disk.
2.  **L2 Cache (Disk):** Persistent, project-scoped files located in `.specd/tmp/` (added to `.gitignore`). This ensures that the L1 cache can be bootstrapped instantly on process startup, and CLI invocations (which are short-lived Node.js processes) also benefit from cached queries.

### Cache File Structure (L2 Disk Cache)

Following the pattern established in the archive, cache files are separated into:

- `*-meta.json`: A tiny metadata file ($<1\text{KB}$) containing verification parameters.
- `*-index.jsonl`: A JSON Lines file containing the data. One JSON object per line. This avoids thread-blocking parses of large "fat" JSON files.

### Fine-Grained Partitioning

To allow incremental invalidations and lazy loading:

1.  **Specs (`SpecRepository`):** Scoped per-workspace (since the repository itself is instanced per workspace).
    - `specs-meta-${workspace}.json`
    - `specs-index-${workspace}.jsonl`
2.  **Working-state Changes (`ChangeRepository`):** Scoped per lifecycle phase.
    - `changes-meta.json` & `changes-index.jsonl` (Active changes)
    - `drafts-meta.json` & `drafts-index.jsonl` (Drafts)
    - `discarded-meta.json` & `discarded-index.jsonl` (Discarded changes)
3.  **Archive (`ArchiveRepository`):** Scoped to the archive root.
    - `.specd-index-meta.json`
    - `.specd-index.jsonl`

## 3. Port Contracts (`Repository` Base Class)

To unifier cache behavior across all delivery mechanisms, we elevate cache configurations to the [Repository](file:///Users/monki/Documents/Proyectos/specd/packages/core/src/application/ports/repository.ts) base class:

- `cacheTtlMs?: number` is added to `RepositoryConfig`. The CLI sets it to `0` (or undefined) for absolute real-time consistency. The HTTP server configures it (e.g. `5000`) for high performance under polling.
- `abstract getLastModifiedFingerprint(): Promise<string>` is defined on `Repository` to allow adapters to return a lightweight, tech-specific hash representing the current disk state.
- `invalidateCache(): void | Promise<void>` is defined as a virtual method on `Repository` (default no-op) to clear both L1 (RAM) and L2 (Disk) caches.

## 4. Minimum Disk Metadata (`meta.json` Schema)

The `*-meta.json` cache files must contain the minimum data necessary to verify cache freshness:

```json
{
  "totalCount": 234,
  "fingerprint": "combined-sha1-hash-of-disk-mtimes"
}
```

- **Total Count:** Allows count-only queries (like `count()`) to be answered instantly without reading data entries.
- **Fingerprint:** A single SHA-1 hash computed dynamically from the `mtimes` of files/directories. We **do not** store the list of individual files or timestamps in the metadata file to keep it tiny, avoiding file size bloat.

## 5. Query Execution & Invalidation Matrix

### Query Mapping

- **Bulk Queries (`list()`, `count()`, `project status`):** Beneficiaries of the cache. They check the `getLastModifiedFingerprint()` (gated by `cacheTtlMs`). If it matches the cached fingerprint, they read `.jsonl`/`meta.json`.
- **Point Queries (`get(name)`, `getSpec(id)`, `artifact()`):** Bypasses the cache entirely and reads directly from disk. This guarantees absolute consistency and prevents race conditions or operations on stale states prior to writes/mutations. Direct path reads are $O(1)$ and cheap anyway.

### Programmatic Invalidation

Use cases that perform writes explicitly call `invalidateCache()` on the modified repository:

- `ArchiveChange`: Invalidates the `ArchiveRepository` cache, and the `SpecRepository` caches of all workspaces that received deltas.
- `CreateChange`: Invalidates `ChangeRepository`'s `changes` cache.
- `EditChange` / `TransitionChange` / `DiscardChange` / `RestoreChange` / `DeleteChange`: Invalidate the caches of the affected phases.

### External Invalidation (IDE Edits / Git)

When checking the cache:

1.  If the time elapsed since the last check is less than `cacheTtlMs`, use the L1 RAM cache.
2.  If it exceeds `cacheTtlMs`, walk the directories to collect file names and `mtimes` to compute the current fingerprint hash.
3.  If the hash matches the cached fingerprint in `meta.json`, return cache data.
4.  If it does not match, run the full walk and parsing logic, and rewrite L2 files.

## 6. Rejected Alternatives

- **Database-backed Archive (`SqlArchiveRepository`):** Rejected for the default implementation because specd must operate locally against the user's workspace, maintaining Git as the absolute source of truth.
- **Storing individual `mtimes` on disk:** Rejected because walking the directory tree recursively to fetch current `mtimes` takes only $\approx 2-5\text{ms}$ in Node.js (VFS page cache). Doing 750 individual `fs.stat` calls based on a saved list is actually slower. Keeping the metadata file tiny ($<100\text{ bytes}$) is superior.
- **Global Change Cache File:** Rejected because combining changes, drafts, and discarded changes in one file would slow down active change queries as the historical discarded logs grow.

## 7. Key Codebase Observations

- `FsSpecRepository` walks the tree and instantiates full `Spec` entities on every listing.
- `FsArchiveRepository` has index files but lacks directory `mtime` verification (scans directories recursively to verify).
- Low-level file utilities are already isolated in `packages/core/src/infrastructure/fs/` (`manifest-change-loader.ts`, `write-atomic.ts`, `dir-name.ts`, `path-confinement.ts`), facilitating implementation without code duplication.

## 8. Conversation Flow

We analyzed the separation of `ChangeRepository` and `ArchiveRepository`, then evaluated performance under high-concurrency HTTP API polling. We designed the L1/L2 cache partitions and aligned on the `.jsonl` / `meta.json` dual-file layout for both specs and changes. We simplified the fingerprint checking by discarding individual `mtimes` logs in favor of a single hash, and finally established the contract boundaries on the base `Repository` class.
