# Exploration Context

## Problem statement

The user wants Studio/UI-side work to stay in this worktree, while shared platform fixes go
to `main`.

The broader symptom observed in production usage was:

- Studio Desktop open
- active change in advanced lifecycle state (`implementing`, `verifying`, or later)
- repeated drift detections on `tasks.md`
- lifecycle rollback to `designing`

Investigation confirmed that the root cause of the false `tasks.md` drift lives in shared
`core` / `sdk` project-status composition and therefore belongs to `main`.

This change exists only for host-specific mitigation or hardening around Studio/UI polling
and read patterns that amplify the shared bug or make similar loops more likely.

## Scope of this change

Initial attached specs:

- `ui:hooks-project`
- `ui:hooks-changes-read`
- `ui:shell-layout`
- `client:port-project`
- `studio-desktop:desktop-local-data-adapter`

The intended boundary is:

- shared bug fix in `../../specd`
- Studio/Desktop/UI/client-side mitigation in this repo only

Do not move shared `core` drift logic into this change unless `/specd-design` later finds a
strictly host-local requirement that cannot live upstream.

## What is already known

### Confirmed root cause lives elsewhere

The false drift was reproduced through:

```bash
node packages/cli/dist/index.js project status --format toon
```

That command alone was enough to mutate the manifest and invalidate a healthy change.
Therefore the primary defect is not Studio-exclusive.

The main fix is tracked separately in `../../specd` as:

- `fix-project-status-artifact-drift`

### Why Studio still matters

Studio amplifies the shared bug because it polls project status automatically.

Key anchors:

- `packages/ui/src/hooks/use-project-poll.ts`
- `apps/specd-studio-desktop/src/main/ipc-handlers.ts`
- `packages/client/src/port-project.ts`
- `packages/ui/src/hooks/use-changes-read.ts`
- `packages/ui/src/hooks/use-change-artifact-list.ts`
- `packages/ui/src/shell/ShellLayout.tsx`

Observed host path:

1. `useProjectPoll()` sets a 2.5s polling interval while focused.
2. That hook calls `port.getProjectStatus()`.
3. In desktop, IPC `getProjectStatus` builds a project-status snapshot.
4. That snapshot currently flows into the shared buggy project-summary path.

This means Studio open + focused is enough to trigger repeated calls into the shared bug.

## Candidate host-specific work

These are exploration candidates, not final design commitments.

### Candidate A: reduce poll pressure

Revisit whether the shell should always poll full project status every 2.5s while focused,
especially when an active change is open and other per-change polling is already happening.

Possible directions:

- lower frequency
- scope polling by visible view
- pause project-status polling while change-focused workflows are active
- fetch a lighter host-specific status surface for chrome warnings

### Candidate B: avoid duplicate read churn

The change inspector already has its own reads:

- `useChangesRead`
- `useChangeArtifactList`

Review whether the shell is redundantly driving:

- global project status polling
- change status polling
- artifact list polling

If the same UI tick causes multiple status-style reads, this change may define cleaner host
coordination so the desktop shell does less work and has fewer chances to amplify backend
state churn.

### Candidate C: isolate desktop-local project status usage

Review whether `studio-desktop:desktop-local-data-adapter` or its IPC call sites should
request a safer or more explicit host-oriented status shape rather than blindly mirroring the
project-summary path used by CLI/HTTP.

This could be temporary mitigation until the upstream fix lands, or permanent API hygiene if
the desktop host has different needs.

### Candidate D: document host expectations

If Studio depends on project-status polling for badges, warnings, or shell chrome, the specs
may need to say more clearly:

- what must be polled
- when it is safe to poll
- what read-only guarantees hosts rely on

## Important non-goals

- Do not fix core drift hashing here.
- Do not duplicate shared lifecycle logic in desktop/UI code.
- Do not add UI-only workarounds that hide real drift after the upstream fix.
- Do not broaden this into generic polling refactors unrelated to the regression.

## Design decisions already made

- The user explicitly wants `main` changes for shared behavior and this worktree only for
  Studio/UI concerns.
- The exploration should preserve that split clearly so `/specd-design` does not merge both
  fixes into one artifact set.
- The temporary investigation copy of `studio-graph-health-warnings` was cleaned up after the
  bug was reproduced; this change should not depend on that copied change still existing.

## Key findings from the investigation

- `useProjectPoll()` polls every 2500ms while the window is focused.
- Desktop IPC `getProjectStatus` currently routes through project-status snapshot building.
- The root invalidation bug reproduces even without Studio, so Studio is an amplifier, not
  the sole cause.
- After the temporary repro change was removed, this change no longer overlaps with the old
  `studio-graph-health-warnings` investigation target.

## Rejected alternatives

## Open questions for /specd-design

1. After the upstream fix lands, is any Studio-specific mitigation still necessary?
2. Should the desktop shell stop polling full project status when the user is deep in a
   change workflow?
3. Is there value in a lighter `project chrome status` concept for host/UI consumers?
4. Are `useProjectPoll`, `useChangesRead`, and `useChangeArtifactList` currently doing
   avoidable overlapping reads during active change views?
5. Should `client:port-project` be clarified to state whether `getProjectStatus()` is
   expected to be read-only and cheap enough for frequent polling?
