# Exploration: graph-change-context

Generated: 2026-06-05

## Problem Statement

Create a new CLI command `specd graph change-context <name>` that gathers all
code-graph intelligence about a change in a single command. Currently, agents
must manually compose multiple CLI calls to understand the graph context of a
change:

- `changes status` to get specIds
- `graph impact --spec <id>` per spec
- `graph search` to find related symbols
- `graph hotspots` to identify risk

A single consolidated command would save tokens, reduce latency, and provide a
richer context bundle for agents.

## Approach / Solution Outline

New CLI command: `specd graph change-context <name>`

It takes a change name, looks up the change's specIds via the kernel (change
repo), then queries the code graph for each spec:

1. **Spec deps & dependents** — via `getSpecDependencies` / `getSpecDependents`
2. **Covered files** — via `getCoveredFiles`
3. **Covered symbols** — via `getCoveredSymbols`
4. **Spec impact** — via `analyzeSpecImpact` per spec
5. **Combined impact** — aggregate across all change specs
6. **Hotspots scoped** — filter hotspots to files/symbols touched by change specs

Output: consolidated context bundle with risk levels, affected files/symbols,
and per-spec breakdowns.

Options:

- `--direction` (dependents|dependencies|both, default: both)
- `--depth <n>` (default: 3)
- `--format text|json|toon` (default: text)

## Affected Areas

### Packages

- **@specd/code-graph** — new domain service `analyzeChangeContext` or the
  logic can live at CLI level by composing existing CodeGraphProvider methods
- **@specd/cli** — new command file `commands/graph/change-context.ts` +
  registration in `index.ts`

### No new code-graph domain service needed

The CLI command composes existing `CodeGraphProvider` methods. No new store
queries or traversal functions required — the value is in the aggregation and
change-scoped filtering.

## Design Decisions

- **No new GraphStore queries needed** — the command composes existing
  `getSpecDependencies`, `getCoveredFiles`, `getCoveredSymbols`,
  `analyzeSpecImpact` methods from `CodeGraphProvider`.
- **Change lookup via kernel** — the CLI command uses `resolveCliContext` to
  get the kernel, then reads the change manifest to get `specIds`. This means
  the code-graph package does NOT need to know about changes.
- **Output format** follows the same pattern as `graph impact` (text/json/toon)
  for consistency.

## Key Codebase Observations

- CLI graph commands live in `packages/cli/src/commands/graph/`
- Registration happens in `packages/cli/src/index.ts` (line 225-230)
- `withProvider` helper wraps CodeGraphProvider lifecycle
- `resolveGraphCliContext` resolves config for graph commands
- `CodeGraphProvider` in `@specd/code-graph` exposes: `getSpec`, `getSpecDependencies`,
  `getSpecDependents`, `getCoveredFiles`, `getCoveredSymbols`, `analyzeSpecImpact`,
  `analyzeImpact`, `analyzeFileImpact`, `getHotspots`
- Change data accessed via kernel: `kernel.changes.getStatus.execute({ name })`
  returns specIds in the change manifest
- `ImpactResult` type: `target`, `directDependents`, `indirectDependents`,
  `transitiveDependents`, `riskLevel`, `affectedFiles`, `affectedSymbols`,
  `affectedProcesses`
- `SpecImpactResult` extends `ImpactResult` with `affectedSpecs`
- No existing method in CodeGraphProvider takes a list of specIds
