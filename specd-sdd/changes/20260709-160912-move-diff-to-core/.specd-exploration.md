# Exploration: move-diff-to-core

## Problem statement

Unified diff generation (`createTwoFilesPatch` from the `diff` npm package) currently lives exclusively in the CLI layer (`packages/cli/src/commands/change/spec-preview.ts`). The upcoming HTTP API and MCP hosts also need to expose diffs in spec previews, and duplicating the diff logic across hosts is unacceptable.

## Approach / solution outline

Move diff generation from the CLI into the `PreviewSpec` core use case (`packages/core/src/application/use-cases/preview-spec.ts`):

1. Add a `diff` field (optional `string`) to `PreviewSpecFileEntry` in core
2. Generate the unified diff inside `PreviewSpec.execute()` when both `base` and `merged` differ (status `merged`)
3. The CLI consumes the pre-computed `diff` field instead of calling `createTwoFilesPatch` itself
4. Add `diff` as a production dependency of `@specd/core`

## Design decisions

- **`diff` as optional field**: `base` and `merged` remain always present. `diff` is computed only for `merged` files (not for `no-op` or `missing`), and omitted (undefined) otherwise. This keeps the interface backward-compatible: existing callers that don't need the diff pay no cost.
- **Context lines**: Use 3 lines of context (matching current CLI behavior).
- **`diff` npm package**: Already in the monorepo (used by CLI). Moving it to core is a simple `pnpm add diff` in the core package. Types are available via `@types/diff`.
- **No colorization in core**: Core generates plain unified diff strings. Colorization stays in the CLI (presentation layer concern).

## Rejected alternatives

- **Keep diff in CLI, wrap in a shared utility**: Each host (HTTP, MCP) would still need to call the same utility, requiring either a separate shared package or duplicating the import. Moving to `PreviewSpec` is the natural home since it already owns the `base`/`merged` data.
- **Add diff generation as a separate use case**: Over-engineering — the diff is a simple string transformation of data already held by `PreviewSpec`. No complex orchestration needed.

## Open questions

- Should `diff` be eagerly computed for all merged files, or lazily? Decision: eager — diff is cheap for typical spec content sizes, and lazy adds complexity for no measurable benefit.
- Should `diff` return `undefined` or `null` when not applicable? Decision: `undefined` — idiomatic TypeScript for optional fields.
