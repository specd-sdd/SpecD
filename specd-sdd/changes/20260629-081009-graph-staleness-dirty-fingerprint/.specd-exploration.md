# Problem statement

The user wants to improve code-graph staleness detection in two related ways:

1. Avoid re-running the full expensive staleness-check flow when the graph was already determined to be stale after the last successful index and no reindex has happened since then.
2. Detect repository changes that happen after the indexed VCS ref without requiring a new commit, so uncommitted worktree changes that affect graph-visible files also mark the graph stale.

During exploration, an additional architectural gap became clear: the current design only persists and compares a single VCS ref resolved from `projectRoot`, while specd supports external workspace `codeRoot` values that may live in different repositories or may not be under VCS at all.

# Approach / solution outline

The explored direction is to separate graph freshness into:

- Derivation/configuration freshness: existing `graphFingerprint`-style signal
- Content freshness: new signal based on persisted VCS/content scope state

The preferred content-freshness model is:

- Group workspaces by detected shared VCS root at index time
- Persist one content-freshness entry per logical scope
- Use `kind: 'vcs-group'` for any group backed by VCS, even if it contains only one workspace
- Use `kind: 'workspace'` for a workspace with no resolvable VCS

For VCS-backed scopes, the intended freshness check is:

1. Compare `currentRef` with the persisted indexed `baseRef`
2. If refs differ, stale immediately
3. If refs match, compare a persisted `diffFingerprint` against the current diff fingerprint relative to that same persisted `baseRef`
4. If diff fingerprints differ, stale

The user explicitly preferred `ref + diffFingerprint` over any full-worktree fingerprint recomputation.

# Key codebase observations

- Current `GetGraphHealth` resolves `currentRef` through `createVcsAdapter(input.config.projectRoot)` and computes `stale` only via `isGraphStale(stats.lastIndexedRef, currentRef)`.
- Current `isGraphStale()` is a pure ref comparison and has no persisted memory of "already known stale since last index".
- Current `runIndexProjectGraph()` resolves exactly one `vcsRef` from `projectRoot`, not per workspace or per external VCS root.
- Current persisted graph statistics include `lastIndexedRef: string | null` and `graphFingerprint: string | null`, but no persisted content-scope entries.
- Current graph-derivation fingerprinting already includes workspace/code-root/configuration-derived state, but not committed/uncommitted content state.
- `discoverFiles()` contains the real visibility logic for `.gitignore`, `excludePaths`, `allowedPaths`, and default exclusions. Any diff-based freshness logic should reuse this effective selection logic rather than reimplementing a parallel path filter.
- `VcsAdapter` already exposes:
  - `ref()`
  - `rootDir()`
  - `modifiedFiles(baseRef)`
  - `show(ref, filePath)`
  - `isClean()`
- `GitVcsAdapter.modifiedFiles(baseRef)` currently uses `git diff --name-only --diff-filter=ACMR <baseRef>` plus untracked files. Deletions are not currently included, and this was identified as a likely gap for correct diff fingerprinting.

# Design decisions

## Freshness dimensions remain separate

The explored model keeps:

- derivation/configuration mismatch as a separate concern
- content stale/fresh/unknown as a separate concern

The user discussion converged on preserving this distinction instead of collapsing everything into one fingerprint.

## No full-worktree fingerprint for VCS-backed scopes

The user explicitly pushed back on recomputing a full `currentWorktreeFingerprint` for every health check. The agreed direction is:

- for VCS-backed scopes, use `baseRef + diffFingerprint`
- do not recompute a full tree fingerprint when only a small diff is relevant

## Scope model: `vcs-group` vs `workspace`

The user preferred this split over "standalone workspace" or cardinality-driven naming:

- `kind: 'vcs-group'` means "this freshness scope is backed by VCS", even if it contains only one workspace
- `kind: 'workspace'` means "this workspace has no resolvable VCS"

This is semantically about freshness source, not group size.

## Privacy constraint: do not persist user filesystem paths

The user explicitly rejected persisting user paths in graph freshness metadata because it could leak into public repositories.

Accepted consequences:

- do not persist absolute `codeRoot`
- do not persist VCS root paths
- do not derive persisted IDs from absolute paths

Preferred persisted identity:

- `scopeId = sha256(sorted(workspaceNames).join('\n'))`

This preserves deterministic identity without storing local filesystem paths.

## Avoid path-based persisted keys

A discarded idea was persisting keys such as `git:/abs/path/to/repo`. This was rejected for privacy reasons.

# Agreements reached

- The change should proceed with the explored direction; the user explicitly said we can move on to creating the change after exploration.
- The content-freshness scope model should use:
  - `kind: 'vcs-group'` for VCS-backed grouped scopes, even singleton ones
  - `kind: 'workspace'` for no-VCS scopes
- Persisted IDs must not expose user filesystem paths.
- `scopeId` should be derived from sorted workspace names rather than filesystem paths.
- For VCS-backed scopes, the preferred freshness signal is `baseRef + diffFingerprint`, not a full worktree fingerprint.
- The existing `cli:graph-stats` overlap with change `12-cli-mcp-sdk-migration` should remain out of the initial scope unless later design work decides that explicit CLI contract changes are necessary.

# Persisted field contract discussed

The final explored metadata shape before handing off to design was:

```ts
interface PersistedGraphMetadata {
  readonly lastIndexedAt: string | null
  readonly graphFingerprint: string | null
  readonly contentFreshnessVersion: 1
}

interface PersistedContentFreshnessEntry {
  readonly scopeId: string
  readonly kind: 'vcs-group' | 'workspace'
  readonly workspaceNames: readonly string[]

  readonly baseRef: string | null
  readonly diffFingerprint: string | null

  readonly lastIndexedAt: string
  readonly knownStaleSinceLastIndex: boolean
  readonly resolverFingerprint: string
}

interface PersistedGraphFreshnessState {
  readonly metadata: PersistedGraphMetadata
  readonly entries: readonly PersistedContentFreshnessEntry[]
}
```

And the preferred scope-id derivation rule became:

```ts
scopeId = sha256(sorted(workspaceNames).join('\n'))
```

# Rejected alternatives

## Persist "last time checked and found stale"

The original user idea included possibly storing "last indexed time" and "last time checked and found stale". During exploration, this shifted toward a simpler, more direct boolean:

- `knownStaleSinceLastIndex: boolean`

This was treated as a cleaner runtime shortcut than relying primarily on timestamps.

## Full worktree fingerprint as the primary content signal

Rejected because:

- expensive
- unnecessary when a base ref plus diff already describes the effective VCS-backed state
- user explicitly preferred avoiding full-tree recalculation

## Path-derived persisted identity

Rejected because:

- it would persist user-local filesystem paths into repo-stored metadata
- the user explicitly called out privacy concerns if the repo becomes public

## Cardinality-driven naming for content scopes

An earlier wording implied that a singleton VCS-backed scope might be represented as a standalone workspace-style entry. The user explicitly improved this:

- `vcs-group` means VCS-backed regardless of cardinality
- `workspace` means no VCS
