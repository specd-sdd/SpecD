# Implementation Snapshot — Exploration Context

Generated: 2026-07-07

## Problem Statement

When working with AI agents on specd changes, there is no safety net if the agent corrupts or destroys files in the working tree. Additionally, the current `RefreshImplementationTracking` auto-detects _all_ files modified since the VCS baseline, including files that were already dirty before the change entered `implementing` (pre-existing modifications unrelated to the change). Both problems stem from the same root cause: there is no snapshot of the working tree state at the moment the change enters `implementing`.

## Solution Outline

Add an **implementation snapshot** capability with two complementary parts:

### Part A: mtime+hash baseline (for filtering pre-existing files)

When the change transitions to `implementing`, capture a snapshot of every modified/untracked file's `{mtime, hash}`. During subsequent `RefreshImplementationTracking` runs:

1. `stat()` each candidate file — if mtime matches the snapshot, skip immediately
2. If mtime differs, compute `git hash-object` — if hash matches, update mtime in snapshot and skip
3. If hash differs, auto-track as a new implementation file

This prevents files that were already dirty before `implementing` from being falsely detected as change-related.

### Part B: working tree patch (for restore safety net)

When the change transitions to `implementing`, persist a unified diff of all uncommitted changes (including staged and unstaged) to a sidecar file. This diff can be used to restore the exact working tree state if an AI agent corrupts files.

The patch is saved via `VcsAdapter.writeSnapshot()` which writes directly to disk (never fully loaded into memory). An indexed `PatchFile` reader provides `listFiles()`, `diffFor(file)`, and `apply()` without loading the entire diff into heap.

## Design Decisions

1. **Baseline strategy**: mtime+hash combination — stat for fast skip, hash for deterministic comparison. This avoids false positives from `git checkout`, `touch`, or build tools that change mtime without changing content.
2. **Where baseline lives**: In the `Change` entity as `_implementationBaseline: Map<file, {mtime: number, hash: string}>`, serialized in the change manifest JSON (same pattern as `trackedImplementationFiles`).
3. **Where snapshot lives**: Sidecar file `changes/<name>/impl-snapshot.diff` — the Change entity only stores `{baselineRef, patchFile, fileCount}`. The diff content never lives in heap, only on disk.
4. **Patch reader**: Indexed (not full-parse). Scans headers (`diff --git`) to build an offset index. `listFiles()` is O(1), `diffFor(file)` reads only that file's bytes from disk.
5. **VCS abstraction**: `VcsAdapter` port gets two new methods — `diff(file, baseRef?)` for per-file diffs, and `writeSnapshot(path, baseRef?, files?)` for bulk write-to-disk. Each VCS adapter (git, hg, svn, null) implements with its native commands.
6. **No external diff library**: The `diff` npm package exists in `packages/cli` but isn't needed for this feature. A streaming parser with index is simpler and more memory-efficient.
7. **Patch file format**: Standard unified diff (same format produced by `git diff`, `hg diff`, `svn diff`). Compatible with `git apply`, `hg import`, `svn patch`.

## Agreements Reached

- The snapshot should be VCS-agnostic via `VcsAdapter` port, not hardcoded to git
- mtime+hash baseline and patch snapshot are complementary but independent features — can be implemented together in one change
- The diff is saved to a sidecar file, never in the manifest JSON
- The patch reader uses index-based scanning, no full parse
- No external diff library needed

## Open Questions

- Whether to capture snapshot on every transition to `implementing` or only on first entry
- Whether `apply()` should be a CLI command or a use case
- Whether snapshot should be auto-cleaned on archive or kept as historical record
- CLI command naming: `specd changes implementation snapshot <name> [--take] [--restore] [--list] [--show <file>]`

## Rejected Alternatives

- **Pure hash baseline** — rejected in favor of mtime+hash for performance (stat fast-path)
- **Pure mtime baseline** — rejected due to false positives from `touch`, `git checkout`, build tools
- **Full parse of patch** — rejected in favor of indexed/streaming patch reader
- **Putting diff in manifest JSON** — rejected due to memory and serialization cost
- **Forcing clean working tree on transition** — rejected as too restrictive
- **Using `diff` npm library** — rejected; raw streaming parser is simpler
