# Exploration Context

## Problem statement

The user wants a new change for multiple `code-graph` improvements centered on graph search and path handling ergonomics.

Primary problems discussed:

- `specd graph search "core:change"` does not prioritize the exact existing spec `core:change`.
- Exact spec identifiers are not treated as first-class search keys in spec search ranking.
- File-oriented graph commands still require canonical `workspace:relativeToWorkspacePath` selectors in too many places, even though users naturally have normal project paths.
- CLI graph outputs currently show canonical workspace-prefixed file paths where the user wants project-relative paths based on `projectRoot`.
- Some graph entry points that conceptually accept a symbol name do not yet support progressively more specific symbol selectors or full symbol identifiers.
- `graph index --workspace` allows partial reindexing semantics that can leave cross-workspace graph relations less trustworthy than a global reindex.
- The CLI is currently responsible for reading `spec-lock` sidecars and projecting archived implementation links into the indexing payload, but that responsibility should belong to `@specd/code-graph`.
- Non-code textual files discovered under `codeRoot` are currently skipped when no language adapter recognizes them, which means potentially useful searchable project knowledge is lost.
- Some legitimate implementation targets for a change live outside a workspace `codeRoot` (for example root `package.json` or `docs/`), and this interacts awkwardly with both implementation-scope rules and graph identity.

## Approach / solution outline

The intended solution is:

- prioritize exact spec-id matches first in spec search results
- add spec identifiers into the spec FTS index so exact or near-exact identifier queries are searchable by the backend itself
- extend code-graph path handling so normal file paths can be passed to graph operations such as search and impact
- resolve those normal paths internally to canonical `workspace:relativeToWorkspacePath` using workspace `codeRoot` knowledge
- keep canonical graph storage unchanged
- change CLI presentation so any file path shown in graph-related outputs is rendered relative to `projectRoot`
- support progressively more specific symbol selectors, from bare names through canonical symbolic identities and full symbol ids
- remove partial-workspace indexing from the public CLI and from the indexing use case so indexing is always global in configured projects
- move `spec-lock` reading and spec coverage projection from the CLI layer into `@specd/code-graph`
- add a `Document`-style fallback model for textual files with no recognized code parser, instead of dropping them entirely
- separate change implementation scope controls from graph indexing scope controls, instead of stretching workspace `codeRoot` semantics to cover project-wide files

## User-confirmed requirements

- Exact spec search:
  - searching `core:change` should surface the exact spec first
  - this should be fixed both by ranking behavior and by adding the identifier into the FTS-backed searchable fields
- File input normalization:
  - preserve canonical graph storage format as `workspace:relativeToWorkspacePath`
  - allow normal file paths as inputs to graph commands
  - supported normal inputs should include absolute paths and relative paths
  - conversion should happen inside code-graph using workspace `codeRoot`
- CLI output normalization:
  - whenever graph CLI output shows a file, render it as a path relative to `projectRoot`
  - this applies broadly, including impact output and symbol search output
  - "project-relative" explicitly means relative to the configured `projectRoot`
- Symbol selector normalization:
  - endpoints that accept a symbol-name-like input should accept all of these forms:
    - `invalidate`
    - `core:src/domain/entities/change.ts:invalidate`
    - `core:src/domain/entities/change.ts:method:invalidate`
    - `core:src/domain/entities/change.ts:method:invalidate:697:2`
  - equivalent selectors using real project-relative or absolute file paths should also be accepted, without requiring the workspace-prefixed canonical file path in the user input
- Indexing semantics:
  - remove `--workspace` from the graph index CLI
  - remove single-workspace indexing mode from the indexing use case
  - indexing in configured projects should behave as a global index run across all configured workspaces
- spec-lock ownership:
  - the CLI should not be the layer that reads `spec-lock` to derive `spec -> file` / `spec -> symbol` links
  - `@specd/code-graph` should own that responsibility during spec indexing
- Document fallback:
  - if a file under `codeRoot` is not handled by any recognized parser/adapter
  - and the file is textual rather than binary
  - index it as a document node instead of dropping it
  - this is considered useful both for real document-like files (`html`, `md`, `txt`, etc.) and as a temporary fallback for unsupported programming languages
  - binary files should continue to be skipped
- Risk mitigation for document fallback:
  - rely on existing `.gitignore` behavior and workspace `excludePaths`
  - if needed, expand the default excludes so generated or low-value textual files are filtered before document fallback is applied
- Config separation idea:
  - outside `code-graph`, add something like `allowedCodePaths` per workspace for implementation-scope guards
  - inside `code-graph`, allow workspace-level `graph.allowedPaths`
  - add root-level graph config with something like `graph.paths` for project-global indexed content outside workspaces
  - project-global indexed content should use a reserved canonical namespace `root:`
  - `root` must therefore become an invalid workspace name in config validation

## Key codebase observations

### Exact spec-id search problem

Observed behavior:

- `node packages/cli/dist/index.js graph search "core:change" --specs --format text` did not return `core:change` in the top results.
- `graph search "core:change" --symbols` returned no symbol results, confirming the issue is in spec search semantics/ranking.

Root cause discovered in code:

- `searchSpecs()` in `packages/code-graph/src/infrastructure/sqlite/sqlite-graph-store.ts` uses FTS/BM25 ordering only.
- The SQLite spec FTS table currently indexes `title`, `description`, and `content`.
- `spec_id` is stored in the FTS table but marked `UNINDEXED`, so it is not searchable/rankable.
- Query sanitation wraps whitespace tokens into quoted OR terms, but there is no exact-spec-id fast path or explicit exact-match boost.
- Ladybug backend follows analogous FTS-driven behavior for spec search.

### Path input handling

Existing capabilities observed:

- `cli:graph-impact` already documents support for workspace-prefixed, config-relative, and absolute file selectors.
- Current file selector resolution is implemented in `packages/cli/src/commands/graph/resolve-impact-file-selectors.ts`.
- `code-graph` stores canonical file paths in workspace-prefixed form.
- `code-graph` also stores `configRelativePath`, and some lookup helpers already use config-relative matching.

Gap identified:

- The user wants broader normalization owned by code-graph, not just command-local selector logic.
- The desired normalization should rely on workspace `codeRoot` and `projectRoot`, and should work consistently across search, impact, and other graph operations.

Additional gap identified:

- Symbol-targeting interfaces currently do not expose one normalized resolution model that accepts:
  - bare names
  - canonical file-qualified symbol selectors
  - full symbol ids
  - file-qualified selectors built from real relative/absolute paths

### CLI output rendering

Observed behavior:

- Graph CLI outputs currently render canonical graph file paths in many places.
- The user wants all file rendering in graph CLI outputs to be relative to `projectRoot` instead.

### Non-code textual files

Observed behavior:

- Files discovered under `codeRoot` that have no registered language adapter are skipped by the current indexing flow.
- Specs are different: their artifacts are indexed as spec content through the spec discovery path, so markdown/text content inside specs already participates in spec search.
- Ordinary non-spec files like `html`, `md`, `txt`, templates, or source files in unsupported languages do not currently enter the graph unless a language adapter supports them.

User's design direction:

- introduce a graph-level document fallback rather than indexing such files as code files
- the fallback should apply when a file is textual and no parser recognizes it
- this is explicitly valued as:
  - a useful search surface for web/document assets such as HTML
  - a "better than nothing" temporary representation for unsupported languages

Guardrails identified by the user:

- do not overfit inclusion by extension alone
- existing exclusion mechanisms already help manage noise
- default excludes can be expanded if needed to keep low-value textual files out

### Workspace scope vs project-global targets

Problem identified by the user:

- Some changes legitimately need to modify files outside a workspace `codeRoot`, such as:
  - root `package.json`
  - `docs/`
  - other project-global config or documentation files
- A naive `implementationRoots` approach per workspace fails conceptually.
- If a workspace like `core` gets an implementation root of `/` or repo root just to allow `package.json`, then:
  - the workspace effectively absorbs the whole monorepo
  - the identity model `<workspace>:path` becomes misleading or broken

Refined direction proposed by the user:

- keep workspace identity narrow and honest
- do not model project-global files as if they belonged to `core` or another normal workspace
- separate implementation-scope policy from graph-indexing scope

Configuration split proposed by the user:

- Outside `code-graph`:
  - add per-workspace `allowedCodePaths` (pattern-based, analogous in spirit to exclusion patterns)
  - this is for implementation/change-scope guards, not graph identity
- Inside `code-graph`, per workspace:
  - add `graph.allowedPaths`
  - this constrains or broadens graph indexing within the workspace's own domain
- At project root config level:
  - add a root-level `graph.paths`
  - these are project-global indexed paths outside workspace roots
  - their canonical graph namespace should be `root:`

Validation rule proposed by the user:

- if `root:` is reserved for project-global graph identity
- then `root` must not be allowed as a workspace name in config
- config validation should fail when a workspace is named `root`

### Partial workspace indexing semantics

Observed behavior and code findings:

- `graph index --workspace <name>` is supported by the CLI and filters `WorkspaceIndexTarget[]` before indexing.
- Deletion behavior is scoped correctly to indexed workspaces only.
- Spec `DEPENDS_ON` relations can still be rebuilt reasonably because the indexer starts from persisted known specs and merges discovered spec ids from the current run.
- However, import and symbol dependency resolution during pass 2 depends on the in-memory `symbolIndex`.
- The `symbolIndex` is populated from processed files plus unchanged files only for the discovered paths of the workspaces in the current run.
- This means partial indexing can leave cross-workspace code relations from the reindexed workspace toward non-indexed workspaces less complete or less reliable until a full global reindex happens.

Decision from the user:

- remove `--workspace` from the CLI and from the indexing use case rather than preserving partial indexing semantics

### spec-lock responsibility boundary

Observed behavior:

- The CLI currently reads `spec-lock` through `SpecRepository.readSpecLock()` while building `DiscoveredSpec` payloads for indexing.
- `@specd/code-graph` then consumes that prepared `implementation` payload and emits `COVERS_FILE` / `COVERS_SYMBOL` relations.

Design issue identified:

- The CLI is doing graph-domain enrichment work that belongs to indexing logic.
- This makes `code-graph` rely on an external caller to provide archived implementation coverage that should be derived inside the graph indexing pipeline.

Decision from the user:

- move `spec-lock` reading and coverage derivation into `@specd/code-graph`
- the CLI should only provide project/config/workspace wiring, not spec coverage extraction semantics

## Design decisions and agreements

- Keep canonical graph file identity unchanged.
- Improve ergonomics at the query/input/output boundaries instead of changing graph storage identity.
- Group these related search/path/output improvements into one dedicated change rather than mixing them with the active archive-reporting work.
- Keep canonical symbol ids unchanged while broadening accepted user-facing selector forms.
- Prefer removing partial indexing semantics over trying to preserve them with caveats about cross-workspace relation accuracy.
- Make `@specd/code-graph`, not the CLI, responsible for reading archived implementation coverage from `spec-lock` during indexing.
- Prefer a separate document-style graph concept over abusing `FileNode` for files that do not produce symbols or code relations.
- Use "textual but unparsed" as the inclusion rule for fallback document indexing, not a hardcoded extension allowlist.
- Use exclusion policy, not only file extension checks, as the main control surface for document-indexing noise.
- Do not solve project-global implementation targets by inflating a normal workspace to repo-root scope.
- Keep graph identity and implementation permission policy as separate configuration concerns.
- Reserve `root:` as a distinct canonical namespace for project-global indexed content if that model is adopted.
