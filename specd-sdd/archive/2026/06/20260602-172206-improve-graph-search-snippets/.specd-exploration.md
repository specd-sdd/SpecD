# Exploration Context

## Problem statement

The user wants to improve `code-graph` search result excerpts because current search output does not provide useful match context.

Observed user complaints:

- For document-style results, the excerpt is effectively the beginning of the document rather than the relevant matching area.
- For spec results, text mode only shows the spec name plus a small part of the description, with no useful snippet from matched content.
- For symbol results, text mode only shows a very short JSDoc/comment preview that is often too small to be informative.
- The user explicitly said they were talking about text mode first, then confirmed the desired scope is broader: "lo suyo es todo ya", meaning the improvement should cover the full search surface, not only text mode.

The user also asked how symbol search currently works and whether it only considers the symbol name. We investigated and clarified the current behavior before creating the change.

## Approach / solution outline

Target outcome:

- Enrich search results across symbols, specs, and documents with useful match-context snippets.
- Improve both text output and structured output (`json` / `toon`), not just CLI presentation in text mode.
- Make the search contract clearer about:
  - what fields participate in ranking
  - what snippet / excerpt / preview is returned
  - how snippets are generated when the primary match is not at the start of the underlying content

Likely implementation direction inferred from investigation:

- The current CLI hardcodes text-mode previews using raw truncation, so part of the change will be to stop using simple `substring(...)` previews in the CLI.
- Because the user wants the improvement across all output modes, the result shape likely needs to carry richer snippet or match-context data from the backend/provider layer rather than rebuilding all logic in the CLI.
- Backends may need parity updates so `sqlite` and `ladybug` both expose richer search result context consistently.

## Key codebase observations

### Current CLI preview behavior

In `packages/cli/src/commands/graph/search.ts`:

- symbol text output uses `symbol.comment.substring(0, 50)`
- spec text output uses `spec.description.substring(0, 60)`
- document text output uses `document.content.substring(0, 60)`

This directly explains the user-visible problem:

- symbol previews are too short and depend entirely on comment presence/quality
- spec previews do not expose matched content from spec bodies
- document previews are not centered around the actual match

### Current symbol search behavior

Clarified to the user:

- Symbol search does not inspect function/method body text directly.
- In SQLite:
  - FTS query searches symbol search text plus symbol comments.
  - exact-match boosts are applied for `symbol.id` and `symbol.name`.
- In Ladybug, spec and implementation imply support for:
  - symbol name
  - expanded compound-name search text (`searchName`)
  - symbol comment

### Symbol comment source

The `SymbolNode` model includes:

- `comment?: string`

This represents the comment attached to that concrete symbol, not arbitrary file-level comments.

Examples explained to the user:

- a function's own `/** ... */`
- a class's own JSDoc
- a method's own JSDoc

Not included:

- generic file comments unless attached by extraction to the symbol
- function body text
- general nearby comments without symbol association

### Search contract observations

`cli:graph-search` currently specifies in text mode:

- symbols show comment preview (first 50 chars)
- specs show description preview (first 60 chars)
- documents show a content snippet centered around the best textual match when available

There is already a mismatch between current text-mode implementation and the stronger expectation implied by the document requirement text:

- spec says documents should show a snippet centered around the best textual match when available
- current CLI implementation simply truncates `document.content.substring(0, 60)`

This suggests the spec/implementation gap is already real and should be addressed during design.
