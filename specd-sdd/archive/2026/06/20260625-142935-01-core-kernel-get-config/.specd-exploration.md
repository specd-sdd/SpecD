# 01-core-kernel-get-config — exploration

## Source

Refactor plan: `dev/docs/studio/core-refactor-on-main.md` (feat-user-interface worktree), phase **P0c**.

## Problem statement

Hosts (CLI `CliContext`, future API/SDK) keep a parallel `config` alongside `kernel`, duplicating the `SpecdConfig` snapshot already used to build the kernel. Need a single readonly escape hatch: `kernel.project.getConfig.execute()`.

## Approach

- New use case `GetConfig` in `packages/core/src/application/use-cases/get-config.ts`
- Constructor receives `Readonly<SpecdConfig>` from `createKernel`
- `execute(): Readonly<SpecdConfig>` — defensive copy or freeze if needed
- Wire as `kernel.project.getConfig`
- **Does not** replace baking config slices inside domain use cases (P1a still bakes `CompileContextConfig` internally)

## Design decisions

- Return readonly snapshot from kernel construction time; does not re-read disk
- Host must not mutate returned object; edits go through `ConfigWriter` factories (P1e)
- `getConfig` is for hosts, not for domain use cases to re-request full yaml in `execute`

## Resolved: immutability (2026-06-25)

**Decision:** `structuredClone(config)` in `GetConfig` constructor; `execute()` returns that clone as `Readonly<SpecdConfig>`.

- External host API — not comparable to internal `ListWorkspaces` wiring.
- Clone prevents host mutation from affecting kernel and avoids "I edited this object so the kernel should change" confusion.
- Live `SpecdConfig` ref stays internal to composition; only the clone is exposed.

## Rejected alternatives

- Passing `config` in every SDK context — rejected; kernel already holds snapshot
