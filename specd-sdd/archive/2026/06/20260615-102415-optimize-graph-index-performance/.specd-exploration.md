# Exploration: Graph Indexing Performance Optimization

## Problem

Indexing large projects (e.g., `iccms` with >3,000 files) is extremely slow or hangs at 80% (Pass 2 - Resolving imports).

### Technical Causes

1. **O(N²) in `SymbolIndex`**: The `findByFilePrefix` method performs a linear scan of ALL symbols stored in memory for every file processed during Pass 2.
2. **Binary File "Indigestion"**: File discovery does not filter known binary files (.gif, .tiff, .sql, etc.). The engine attempts to read them as text, saturating memory and CPU with binary "junk."
3. **PHP Inefficiencies**: The `PhpLanguageAdapter` attempts to guess CakePHP/CodeIgniter paths by launching blind resolutions against a massive symbol list, instead of efficiently verifying physical file existence.

## Proposed Solution

1. **`SymbolIndex` Refactor**:
   - Replace linear scans with O(1) access using an internal `Map<string, SymbolNode[]>` indexed by file.
   - Implement a composite lookup index by qualified name.
2. **Global Binary Filter**:
   - Incorporate a forbidden extensions list (.gif, .png, .jpg, .pdf, .sql, .zip, etc.) into the initial discovery process.
3. **PHP Resolution Optimization**:
   - Inject a `Set<string>` containing all existing file paths into the language adapter.
   - Refactor `selectCandidatePath` to use this `Set`, avoiding constant creation of new `Set` objects.
4. **Improved Progress & Monitoring**:
   - Ensure progress messages update immediately when a phase finishes (preventing stale "Discovering specs" labels during subsequent internal processing).
   - Add debug-level logging to track the execution time of each indexing phase (Pass 1, Pass 2, Symbol resolution, etc.).

## Decisions and Agreements

- **Not PHP-exclusive**: The `SymbolIndex` changes will benefit all languages, though the impact is most critical for PHP.
- **Escape Hatch**: Configurable exclusion filters in `specd.yaml` will remain, but common binaries will be added to the engine's defaults.

## Affected Areas
