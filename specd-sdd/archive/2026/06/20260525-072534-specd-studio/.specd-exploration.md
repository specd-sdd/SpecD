# Exploration context — specd-studio

Generated: 2026-05-25

## Problem statement

Build **SpecD Studio**: a spec-work IDE (not a dashboard) with HTTP API on `@specd/core`, shared React UI, standalone web client, and Electron desktop. Users manage changes, workspace specs, artifacts, validation, and code graph without duplicating business logic in CLI/MCP subprocesses.

## Approach / solution outline

- **`@specd/api`**: Fastify HTTP adapter → `createKernel` + `createCodeGraphProvider`; no CLI/MCP in request path.
- **`@specd/client`**: `SpecdDataPort` + `RemoteSpecdDataAdapter` (+ memory for tests).
- **`@specd/ui`**: IDE shell (Monaco, markdown preview, inspector tabs); modes `embedded` | `standalone` | `desktop`.
- **`apps/specd-studio-web`**: remote-only host (API URL + token, Connect panel).
- **`apps/specd-studio-desktop`**: **local** (folder / `specd.yaml` → IPC + kernel) **or** **remote** (URL + token).
- **`packages/cli`**: `specd serve` (API only), `specd ui serve` (API + static UI dist).
- Scaffolding already in repo: `packages/{api,client,ui}`, `apps/{specd-studio-web,specd-studio-desktop}`, workspaces in `specd.yaml`, empty `specs/*` dirs.

Design draft (this change): `specd-studio-api-and-ui.md` (English). Spanish notes (repo root): `ui-ideas.md` (historical).

## Design decisions (confirmed in conversation)

### Architecture

- API talks **directly** to core (and code-graph package), never shells out to CLI/MCP.
- `packages/` = api, client, ui, cli extensions; `apps/` = studio-web, studio-desktop; `public-web` unchanged.
- CLI stays in `packages/cli`; may depend on `@specd/api` + ui dist for `ui serve`.

### API

- Remove bogus `GET .../validation`; use `GET .../status` + `POST .../validate`.
- `PATCH /changes/{name}` via `EditChange`: `description` (no invalidate), `addSpecIds`/`removeSpecIds` (invalidates), `invalidationPolicy` (no invalidate by itself).
- No dedicated “linked changes” spec tab or API; use change `specIds` + `GET /changes/overlaps`.
- Workspace canonical specs read-only in Studio v1; edit change artifacts via `PUT`.

### UI tabs

- Drop **Transitions** tab → lifecycle actions in Overview; history in Events.
- Drop **Linked changes** on spec view.
- **Delta artifacts**: tabs **Delta** (edit yaml), **Preview** (merged), **Full diff** (unified base→merged from `PreviewSpec`, not per-op hunks).
- **Monaco** for edit; markdown render for preview.

### Delivery modes

| Mode           | Launcher                                   |
| -------------- | ------------------------------------------ |
| Integrated     | `specd ui serve` / `specd serve`           |
| Standalone web | `pnpm --filter @specd/studio-web dev`      |
| Desktop local  | Open folder or `specd.yaml` → kernel + IPC |
| Desktop remote | URL + token → RemoteSpecdDataAdapter       |

### Desktop-specific

- File menu: Open Local Project, Connect to Remote API, Open Recent (local + remote entries, ~20 MRU, userData JSON).
- Welcome screen with recents; one active profile per session; switch profile = close session.
- v1 token for remote recent may require re-entry; keychain v2.

### Artifact editing

- Change artifacts editable (phased); canonical workspace specs not in v1.
- Save uses `originalHash`; refresh status after save.

### Auth

- API: bearer static/JWT; loopback optional off for `specd ui serve`.
- Desktop local: no HTTP auth; remote: bearer like web.

## Rejected / deferred

- `GET /changes/{id}/validation` as separate resource.
- Linked changes reverse lookup tab.
- Transitions as dedicated tab.
- Direct edit of canonical `specs/` in v1.
- `apps/web` naming (conflicts with `public-web`).
- Putting CLI or API in `apps/` (they stay in `packages/`).

## Open questions for design

- Exact spec capability paths and whether one umbrella spec per workspace vs split (e.g. api routes vs auth).
- `?diff=unified` on preview API vs client-only diff computation.
- Electron main HTTP proxy vs renderer fetch for remote desktop.
- Phase ordering: confirm roadmap phases 0–6 in design doc.
- Global architecture delta scope (dependency graph text only vs ADR).
- Whether `code-graph` needs its own spec slice for `/graph/*` routes or folded into `api:api/studio-http-api`.

## Dependencies to consider in design

**Granularity (2026-05-25):** Proposal defines **133 specs** — split by hexagonal slice (domain/port/adapter/middleware/composition, routes+handlers, presenters+DTOs, client transport, UI hooks, desktop IPC). No umbrella package specs. See `proposal.md`.

```text
default:_global/architecture  →  documents acyclic graph (delta)
api/* (infra + dto + routes)  →  core, global constraints
client/dto-*                  →  matching api/dto-*
client/port-* + adapters      →  client DTO specs
ui/* (per tab/sidebar/inspector) →  client port groups + shell specs
studio-web/*                  →  ui connect + shell
studio-desktop/*              →  ui shell + ipc/remote adapters
cli/serve-*                   →  api/http-server
```

Register `specd changes deps` per spec during **specs** artifact (each spec’s `## Spec Dependencies` section is source of truth).
