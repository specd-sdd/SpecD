# Exploration: fix-change-status-cache-on-switch

## Problem statement

In SpecD Studio, when the user opens a change for the first time, the **Workflow & validation** section on the Overview tab loads correctly. When they switch to another change and return to a previously loaded one, the section shows **Workflow status unavailable.** instead of the cached workflow status.

The user suspects other UI areas may break for the same reason.

## Root cause (confirmed in code)

File: `packages/ui/src/hooks/use-changes-read.ts`

Two interacting bugs in `useChangesRead`:

1. **`lastModified` is hook-global, not per-change.** Status polling passes `ifModifiedSince: lastModified` to `loadChangeStatus`. When navigating A → B → A, the request for A may use B's `updatedAt` as `ifModifiedSince`. In `get-status.ts`, if `clientRevision >= change.updatedAt`, the API returns `{ unchanged: true }` without full status payload — a false positive when B is newer than A.

2. **`statusData` is cleared on `changeName` change but not restored on `unchanged` responses.** An effect resets `statusData` to `undefined` when `changeName` changes. Another effect ignores `unchanged: true` responses (`return` early). After a false `unchanged`, `statusData` stays `undefined` → `ChangeStatusPanel` shows embedded message **Workflow status unavailable.**

Relevant code paths:

- `packages/ui/src/hooks/use-changes-read.ts` — `useChangesRead`, lines 41–100
- `packages/ui/src/change/ChangeStatusPanel.tsx` — shows "Workflow status unavailable." when `!status` and `embedded`
- `packages/ui/src/shell/ShellLayout.tsx` — passes `changeRead.status.data` to `ChangeMainView`
- `packages/core/src/application/use-cases/get-status.ts` — `ifModifiedSince` short-circuit logic
- `packages/ui/src/lib/change-read-routes.ts` — `changeReadCacheKey` for per-section cache buckets

## Affected areas beyond Overview

- **Tasks tab** (`ChangeMainView.tsx`) receives `status` from the same hook — likely shows stale/missing task metadata when switching back.
- Any consumer of `changeRead.status.data` in `ShellLayout.tsx` (validate flows, refetch callbacks).

Artifact list hooks (`useChangeArtifactList`) use separate cache keys and do not share the `lastModified` bug — less likely affected.

## Approach / solution outline

Fix in `useChangesRead` only (minimal scope):

1. **Per-cache-key state** for `lastModified` and last full `ChangeStatusDto` (e.g. `Map` keyed by `changeReadCacheKey(listSection, 'change-status:...')` or equivalent).
2. **On change key change:** restore `statusData` from per-key cache instead of clearing to `undefined`.
3. **On full status response:** update per-key cache and `lastModified` for that key only.
4. **On `unchanged: true`:** restore `statusData` from per-key cache for the current change (do not leave undefined).
5. **Reset `ifModifiedSince`** when the cache key changes — never carry another change's timestamp.
6. **Add unit tests** for A → B → A navigation (mock port with distinct `updatedAt` values where B > A).

## Spec IDs attached

- `ui:change-tab-overview` — consumer: Workflow & validation card must show status for active/draft changes

## Design decisions

- Fix at hook layer, not in `ChangeStatusPanel` or `ShellLayout` — single source of truth.
- Do not change API `get-status` short-circuit semantics; client must send correct per-change `ifModifiedSince`.
- Keep `useAsyncResource` as-is; fix state layering in `useChangesRead`.

## Rejected alternatives

- Showing API error instead of "unavailable" — wrong UX; spec requires that copy only for archived/discarded.
- Disabling `ifModifiedSince` polling entirely — would increase load; per-key tracking is the correct fix.
