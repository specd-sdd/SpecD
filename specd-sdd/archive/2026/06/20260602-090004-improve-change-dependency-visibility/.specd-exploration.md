# Exploration Context: improve-change-dependency-visibility

Generated: 2026-06-02

## Problem Statement

AI agents and users working on a `specd` change currently lack visibility into the dependencies between specs within that change.

- `specd changes deps <name> <specId>` only allows modifying dependencies for a specific spec; it has no mode to list all dependencies in the change.
- `specd changes status <name>` shows artifact lifecycle status (DAG) but does not expose spec-to-spec dependencies (`specDependsOn` in the manifest).
- This makes it difficult for agents to understand the full context and relationship graph of the specs they are working on.

## Intent

Improve the CLI to expose spec dependencies in a clear, machine-readable, and human-readable way.

## Proposed Solution

1. **Enhance `changes deps`**:
   - Allow running `specd changes deps <name>` without a `specId` to list all spec dependencies currently registered in the change manifest.
   - Support `--format text|json|toon`.

2. **Enhance `changes status`**:
   - Add a "spec dependencies" section to the status output.
   - In text mode, this could be a simple list or a small graph.
   - In JSON/toon mode, ensure the dependencies are clearly represented in a dedicated field (like `specDependsOn`).

## Affected Areas

- **`cli:change-deps`**: Update to support zero-argument (specId-wise) call for listing.
- **`cli:change-status`**: Update to include spec dependency information in the output.
- **`core:get-status`**: Ensure `GetStatus` use case includes `specDependsOn` in its result (it likely already does, but need to verify).
- **`core:update-spec-deps`**: Verify if it needs to support a "list" query mode or if we just use the change entity data.

## Initial Specs

- `cli:change-status`
- `cli:change-deps`
- `core:get-status`
- `core:update-spec-deps`

## Design Decisions

- The dependency list should clearly distinguish between dependencies within the change and external dependencies (if relevant).
- JSON output should be optimized for agent consumption.

## Open Questions

- Should `changes deps <name>` without arguments show ALL specs (even those with no deps) or only those with declared dependencies?
- Should `changes status` show dependencies by default or only with a flag (e.g., `--deps`)? The user suggested it's missing from "status", so maybe it should be visible by default in a summarized way.
