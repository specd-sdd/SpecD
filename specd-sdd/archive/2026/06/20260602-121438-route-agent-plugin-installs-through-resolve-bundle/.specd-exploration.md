# Exploration Notes

## Problem statement

Agent plugins currently install skills by calling `SkillRepository.getBundle(...)`
directly instead of routing through `ResolveBundle.execute(...)`.

That split created a real bug in installed skills:

- templates now reference shared context via `@{{sharedFolder}}/shared.md`
- `ResolveBundle` injects the built-in default `sharedFolder`
- direct repository calls do not inject that built-in
- therefore installed `SKILL.md` files could keep the unresolved token
  `@{{sharedFolder}}/shared.md`

The user wants the canonical install path to go through `ResolveBundle`, so built-in
render defaults are applied centrally by `@specd/skills` instead of being partially
reimplemented in every plugin.

## Root cause discovered

The recent capability-aware templating change introduced a correct built-in default
for `sharedFolder` in:

- `skills:src/application/use-cases/resolve-bundle.ts`

But all five agent plugins still resolved bundles by calling:

- `SkillRepository.getBundle(name, context)`

directly from their local `install-skills.ts` use cases.

That means:

- `ResolveBundle.execute(...)` -> injects `configPath`, `schemaRef`, `sharedFolder`
- `SkillRepository.getBundle(...)` -> only renders whatever variables already exist

So the bug was not in `TemplateRenderer`; it was in bypassing the use case layer.

## What was observed in code

Relevant symbols and files inspected:

- `skills:src/application/use-cases/resolve-bundle.ts:ResolveBundle`
- `skills:src/infrastructure/repository/skill-repository.ts:getBundle`
- `skills:src/infrastructure/repository/template-renderer.ts:TemplateRenderer`
- `plugin-agent-claude:src/application/use-cases/install-skills.ts:InstallSkills`
- `plugin-agent-copilot:src/application/use-cases/install-skills.ts:InstallSkills`
- `plugin-agent-codex:src/application/use-cases/install-skills.ts:InstallSkills`
- `plugin-agent-opencode:src/application/use-cases/install-skills.ts:InstallSkills`
- `plugin-agent-standard:src/application/use-cases/install-skills.ts:InstallSkills`

Template example that exposed the bug:

- `packages/skills/templates/specd/SKILL.md.tpl`

It contains:

- `@{{sharedFolder}}/shared.md`

Observed behavior:

- plugins computed a default shared-folder path for physical install location
- but only passed `sharedFolder` into template variables when the caller explicitly
  provided `options.variables.sharedFolder`
- therefore default install path and default render path could diverge

## Temporary implementation fix applied outside this new change

A direct bug fix was applied to current code before this change was created:

- all five `install-skills.ts` files were patched to always pass
  `sharedFolder: resolvedSharedFolder.relativePath`
  into `repository.getBundle(...)`
- plugin tests were extended to assert that `getBundle()` receives
  `.specd/config/skills/shared`

This fixed the immediate runtime issue, but the user identified the deeper design
problem: plugins should not keep using the repository directly when `ResolveBundle`
already exists to centralize built-in render defaults.

## Intended direction for this change

Refactor agent-plugin installation flow so plugins resolve skill bundles through
`ResolveBundle` rather than calling `SkillRepository.getBundle(...)` directly.

Desired architectural outcome:

- `@specd/skills` owns built-in render context preparation
- plugins provide install intent, capabilities, and plugin-specific variables
- `ResolveBundle` remains the canonical boundary for default built-ins such as:
  - `configPath`
  - `schemaRef`
  - `sharedFolder`
- plugins stop duplicating or partially mirroring that logic

## Scope agreed

Initial specs to include in the change:

- `skills:resolve-bundle`
- `plugin-agent-claude:plugin-agent`
- `plugin-agent-copilot:plugin-agent`
- `plugin-agent-codex:plugin-agent`
- `plugin-agent-opencode:plugin-agent`
- `plugin-agent-standard:plugin-agent`

Not included initially:

- `skills:skill-repository`
- `plugin-manager:agent-plugin-type`

Reason:

- the user focused this change on routing plugin installs through the existing
  use case, not on redesigning the public plugin contract again
- current evidence points to a flow refactor, not a new template contract

Those specs can be added later during design if the artifact work proves they are
needed.

## User decisions and preferences

- The user explicitly wants plugins to use `ResolveBundle`.
- The user sees the current direct `getBundle(...)` usage as the wrong design.
- The user wants a new change created for this refactor instead of silently folding
  it into the already archived capability-aware templating change.

## Rejected / superseded alternative

Rejected as final design:

- keeping plugins on direct `SkillRepository.getBundle(...)` and just patching them
  to always pass the default `sharedFolder`

Reason:

- fixes the symptom but preserves duplicated boundary logic
- leaves room for future drift whenever `ResolveBundle` gains another built-in

## Risks and design concerns to address in `/specd-design`

- determine whether plugins should instantiate/use `ResolveBundle` directly or whether
  a different factory/helper in `@specd/skills` should expose that path more cleanly
- ensure no duplicate built-in injection remains in plugin `install-skills.ts`
- decide whether plugin-local `shared-folder.ts` helpers remain necessary once
  `ResolveBundle` becomes canonical
- confirm whether uninstall flows should keep their own `sharedFolder` resolution
  or reuse shared logic in another way
- preserve current runtime-specific frontmatter handling while moving bundle resolution
  behind `ResolveBundle`

## Conversation flow summary

1. After archiving the capability-aware template change, the user noticed installed
   skills still contained `@{{sharedFolder}}/shared.md`.
2. Investigation showed the renderer itself was fine; the bug came from plugins not
   passing the default `sharedFolder` into `getBundle(...)`.
3. A direct fix was applied to all five plugins so installed templates now resolve the
   variable correctly.
4. The user then asked why `ResolveBundle` had not been used, and confirmed that
   plugins should go through `ResolveBundle` instead of calling the repository directly.
5. Based on that, a new change was proposed and approved to formalize the refactor.
