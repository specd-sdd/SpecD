# Exploration Context: harden-implementation-tracking

## Problem Statement

The current implementation tracking system has issues managing files that no longer exist or are temporary. LLMs or build tools often create temporary files that are tracked, but there is no way to remove them from the manifest if they disappear, as `resolve` and `ignore` commands currently fail if the file is missing physically. Additionally, internal `specd` directories (like `changes/`, `drafts/`, etc.) are sometimes tracked, creating noise.

## Approach / Solution Outline

1.  **New State `removed`**: Introduce a `removed` state for `TrackedImplementationFileState`. If a tracked file is missing from the disk, it should be transitioned to this state.
2.  **Automatic Cleanup**: When a file is marked as `removed`, existing `ImplementationLink`s to that file should be automatically invalidated or removed.
3.  **Relaxed `ignore`**: Allow ignoring a file even if it doesn't exist on disk, as long as it is already in the tracking list (including `removed` state).
4.  **Directory Filtering**: Exclude internal `specd` management directories from implementation tracking using a mechanism similar to `discoverFiles` in `code-graph`.

## Affected Areas

- **Packages**: `@specd/core`, `@specd/cli`.
- **Domain**: `Change` entity (`packages/core/src/domain/entities/change.ts`), `TrackedImplementationFileState`.
- **Infrastructure**: `VcsImplementationDetector`, `FsChangeRepository`.
- **Use Cases**: `RefreshImplementationTracking`, `UpdateImplementationTracking`.
- **CLI**: `specd changes implementation` commands.

## Design Decisions

- Introduce `removed` as a first-class state.
- Lax existence checks for `ignore` when the file is already known to the manifest.
- Implement a blacklist for internal `specd` directories in the detector/repository layer.

## Agreements Reached

- The system should automatically detect missing files and mark them as `removed` during refresh.
- Links to non-existent files must not block lifecycle transitions or cause errors.

## Key Codebase Observations

- `TrackedImplementationFileState` currently only supports `open`, `resolved`, and `ignored`.
- `UpdateImplementationTracking` use case handles `add`, `remove`, `ignore`, `resolve`.
- `RefreshImplementationTracking` uses `ImplementationDetector` to find new files but doesn't check for deletions.
- `discoverFiles.ts` in `code-graph` has a robust exclusion pattern system that can be adapted.
