# Problem statement

The current `@specd/skills` package resolves templates using flat variable substitution only. This is sufficient for simple `{{key}}` replacement, but it is too limited for agent-install-time customization where the rendered skill content should vary depending on agent capabilities such as MCP support or subagents.

The user wants the skill templates to support capability-aware conditional content so that agent plugins can install different final markdown depending on the target runtime capabilities. The immediate use case is agent installation; this change is not intended to introduce a general-purpose runtime templating system beyond that scope.

# Scope

- Limit scope to `@specd/skills` and the `plugin-agent-*` packages involved in installing skills.
- Do not broaden this change into unrelated consumers outside the agent install path.
- Adapt existing skill templates as needed so they can use the new capability-aware context.

# Approach / solution outline

- Move `@specd/skills` from flat variable substitution toward a template renderer that supports conditional sections for install-time rendering.
- Preferred direction: use `Handlebars`, not `Liquid`.
- Introduce an explicit install-time contract for capabilities passed from each agent plugin into `@specd/skills`.
- Keep capabilities as structured data resolved by the plugin side and consumed declaratively by templates, rather than embedding runtime methods like `Set.has('mcp')` in templates.
- Preserve the current separation where base skill markdown lives in `packages/skills/templates/` without runtime frontmatter and agent plugins still inject runtime-specific frontmatter.

# Confirmed decisions

- Use a dedicated change for this work rather than reusing the active code-graph change.
- Preferred template direction is `Handlebars`.
- The new contract should include explicit capabilities provided by the installing agent plugin.
- Existing templates should be adapted where necessary, not just the underlying renderer/API.
- The practical execution point is agent installation; this is not being designed as a generic always-on templating runtime.

# Rejected alternatives

- `Liquid` was discussed and not selected as the preferred direction.
  Reason: the need is install-time conditional rendering in a TypeScript/Node codebase already using `{{...}}` placeholders; `Handlebars` better matches the current mental model and migration path.
- Leaving the behavior in plugin-specific ad hoc string rewriting was not preferred.
  Reason: it would spread template behavior across `plugin-agent-*` packages and weaken the single source of truth in `@specd/skills`.
- Keeping only `variables?: Record<string, string>` was rejected.
  Reason: capability-aware rendering needs structured context, not only flat string substitution.

# User preferences

- Keep the change focused and pragmatic.
- Capabilities should be part of an explicit contract, not inferred implicitly.
- Template adaptation is part of the work when needed.

# Current codebase observations

- `packages/skills/src/infrastructure/repository/skill-repository.ts` currently resolves bundles with:
  - `getBundle(name, variables?: Readonly<Record<string, string>>, config?: SpecdConfig): SkillBundle`
  - built-in flat variables `projectRoot`, `configPath`, and `schemaRef`
  - regex-based flat replacement through `applyVariables(...)`
- `applyVariables(...)` currently matches `{{key}}` style tokens and preserves unknown placeholders.
- `packages/skills/src/infrastructure/repository/template-reader.ts` only exposes lazy file-backed templates and has no rendering semantics.
- `plugin-agent-*` install use cases currently call `repository.getBundle(skillName, options?.variables, config)` and then inject frontmatter before writing files.
- The current `plugin-manager:agent-plugin-type` spec defines:
  - `skills?: string[]`
  - `variables?: Record<string, string>`
    as the only install options relevant to rendering.
- Existing specs for `skills:skill-repository` and `skills:resolve-bundle` are written around flat `{{key}}` substitution.
- Existing spec `skills:skill-templates-source` currently governs source layout and frontmatter responsibilities, but not capability-aware conditional rendering.

# Likely design implications

- `AgentInstallOptions` will likely need to expand beyond only `variables?: Record<string, string>`.
- `SkillRepository.getBundle(...)` and/or the `ResolveBundle` use case will likely need a richer render context shape.
- The design should clarify whether legacy flat variables remain supported for backward compatibility, are wrapped into a richer context, or are replaced outright.
- The design should define what capability shape plugins provide, how deterministic it is, and whether runtime identity (for example `codex`, `claude`, `copilot`) is also included alongside capability booleans.
- The design should define what subset of templating features is allowed in skill markdown so that templates remain auditable and not overpowered.

# Open questions for design

- Exact shape of the new install/render context contract.
- Whether capability data should be booleans, arrays, or both.
- Whether `Handlebars` helpers are allowed or the contract should rely only on plain `if`/`each` with precomputed data.
- Backward-compatibility rules for existing flat `{{projectRoot}}`-style placeholders.
- Which current templates need conditional branches immediately and which only need the new capability context available.
