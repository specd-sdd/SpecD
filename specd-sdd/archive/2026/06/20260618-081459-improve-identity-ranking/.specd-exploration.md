# Exploration Context

## Problem Statement

The user wants to improve code-graph search ranking, especially when searching by a spec's primary identity or a symbol's primary name.

Current pain points called out by the user:

- Searching for a spec whose ID contains common words like `default` does not sufficiently prefer the spec's own identity.
- Example query shape: `default:_global/architecture`.
- Searching for `architecture` can rank other specs above `default:_global/architecture` because the generic term appears more times elsewhere.
- The same problem exists for symbol names.
- The user also wants equivalent treatment for files and documents if that ranking preference is not already implemented.

Why this matters:

- Generic BM25-style textual relevance can be dominated by term frequency in body content.
- For identity-oriented search intents, the system should favor the primary identity fields before generic content density.

## Approach / Solution Outline

Discovery outcome suggests this is a ranking-semantics change for graph search, not a narrow CLI formatting fix.

Likely direction:

- Strengthen ranking behavior for primary identities:
  - spec ID
  - symbol name and possibly symbol ID
  - document path and related file identity if applicable
- Existing exact-match prioritization already exists in specs.
- The gap is likely partial / prefix / segment-based identity matching, where exact equality is not enough.
- The change should clarify or extend ranking semantics so strong identity-oriented matches outrank generic content-heavy matches even when the query is not an exact full-ID string.
