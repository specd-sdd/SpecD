# Exploration: fix-get-change-artifact-read-only

## Problem statement

Studio users observed `updatedAt` on a change advancing without editing anything. Direct
`GET /v1/changes/{name}/status` does not bump the revision clock, but opening artifacts in
the UI (or calling `GET /v1/changes/{name}/artifacts/{filename}`) does.

Confirmed reproduction endpoint:

`http://127.0.0.1:4450/v1/changes/test-change/artifacts/proposal.md`

This branch (`feat/user-interface`) introduced `@specd/api`, `GetChangeArtifact`, and Studio
UI hooks. **`main` does not have this endpoint or use case** — fix belongs on this branch,
not a separate main hotfix.

## Root cause (code investigation)

`GetChangeArtifact.execute()` wraps a read in `ChangeRepository.mutate()`:

- File: `packages/core/src/application/use-cases/get-change-artifact.ts`
- `mutate()` always calls `save()` after the callback (`change-repository.ts`)
- `save()` always calls `change.touchUpdatedAt()`

So every artifact body read persists the manifest and bumps `updatedAt`, even when the
callback only reads disk via `artifact(change, filename)`.

Current spec **requires** this broken behaviour:

- `specs/core/get-change-artifact/spec.md` — "Within `ChangeRepository.mutate`, the use case
  MUST load the change..."

`GetReadOnlyChangeArtifact` already shows the correct pattern for draft/discarded reads
(no `mutate`).

## Approach / solution outline

Make `GetChangeArtifact` a true read use case:

1. `changes.get(name)` — load change (existing read-side effects in repo are separate concern)
2. `findTrackedArtifactFile(change, filename)` — same confinement guard as today
3. `changes.artifact(change, filename)` — read bytes + hash
4. Return `{ content, originalHash }` **without** `mutate()` / `save()`

Update `core:get-change-artifact` spec to require read-only repository access (no `mutate`).

Add/adjust tests in `packages/core/test/application/use-cases/save-change-artifact.spec.ts`
(or dedicated get-change-artifact spec) asserting `updatedAt` unchanged after execute.

No API handler or UI code changes expected — behaviour fix is in core.

## Design decisions

- **Fix on `feat/user-interface`**, not main (code does not exist on main).
- **Narrow fix**: change `GetChangeArtifact` only; do not refactor `mutate()` to skip save
  when unchanged (broader blast radius).
- **Tracked-file guard** must remain identical to `SaveChangeArtifact`.
- **Do not** change `GetStatus` or polling hooks in this change unless verify finds gaps.

## Rejected alternatives

- Fix only in UI (skip `getChangeArtifact` polling): wrong layer; API still corrupts
  `updatedAt` for any client.
- Make `mutate()` skip `save()` when entity unchanged: affects all use cases; out of scope.
- Bump `updatedAt` on read-path `_writeManifestAtomic` in `get()`: separate issue; not this
  change.

## mutate() audit (2026-06-23)

Full scan of `packages/core/src/application/use-cases/` and API GET handlers:

**Only incorrect `mutate` usage:** `GetChangeArtifact` — wraps a pure read in `mutate()`.

All other `mutate()` callers are legitimate writes (`save-change-artifact`,
`validate-artifacts` conditional, `transition-change`, `edit-change`, etc.).

`RefreshImplementationTracking` only mutates when new tracked files are detected; guard-fail
path uses `get()` only.

Read use cases already correct (no `mutate`): `GetStatus`, `GetReadOnlyChangeArtifact`,
`OutlineChangeArtifact` (`get()` + `artifact()`), `GetImplementationReview`,
`GetArtifactInstruction`, `CompileContext`, `PreviewSpec`, etc.

API GET paths:

- `GET /changes/:name/artifacts/:filename` → `getArtifact` (**bug**)
- Draft/discarded/archived artifact GET → `getReadOnlyChangeArtifact` (OK)
- `GET /changes/:name/artifacts/:filename/outline` → `outlineArtifact` (OK)
- `GET /changes/:name/status?refreshImplementation=true` → intentional opt-in write (OK)

**Scope decision:** fix stays limited to `GetChangeArtifact` + `core:get-change-artifact` spec.
No need to expand change scope to other use cases.
