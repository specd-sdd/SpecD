# Exploration Context: llm-optimized-metadata

Generated: 2026-06-04

## Problem Statement

Agents currently consume raw metadata sections (rules, constraints, scenarios) when compiling context. This can be token-expensive. We want to introduce LLM-optimized fields (`optimizedDescription` and `optimizedContext`) to provide higher signal-to-token ratio representations of specs.

## Approach / Solution Outline

1. Update `SpecMetadata` domain interface to include `optimizedDescription` and `optimizedContext`.
2. Update `specd-metadata` skill instructions to:
   - Include these new fields in the quality gate.
   - Update the subagent optimizer prompt to generate these fields.
   - Use "smart caveman" style for `optimizedContext`.
3. The core use case `GetProjectContext` and `CompileContext` already have some infrastructure to use these fields if present, but the generation part (the skill) needs to be updated.

## Affected Areas

- `.agents/skills/specd-metadata/SKILL.md`
- `metadata.json` files for all specs.

## Key Codebase Observations

- `SpecMetadata` in `packages/core/src/domain/services/parse-metadata.ts` already has the fields.
- `GetProjectContext` in `packages/core/src/application/use-cases/get-project-context.ts` already prefers `optimizedDescription` and `optimizedContext` when `llmOptimizedContext` flag is on.
- The `specd-metadata` skill uses subagents for LLM optimization.

## Agreements Reached

- `optimizedDescription`: one punchy sentence (< 150 chars).
- `optimizedContext`: ultra-terse Markdown with `## Rules` and `## Constraints`.
- Skill should check if these fields are already present and high-quality before regenerating.
