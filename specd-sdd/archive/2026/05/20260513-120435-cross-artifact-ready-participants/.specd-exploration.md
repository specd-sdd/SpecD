# Problem statement

The user wants to fix a gap in cross-artifact validation for change artifacts. Today, when validating one artifact at a time with `ValidateArtifacts`, cross-artifact rules such as `specs-verify-requirement-parity` can be deferred repeatedly even after the counterpart artifact already exists and is marked `complete`.

The concrete failure mode discussed is:

- validating `spec.md` first when `verify.md` does not exist yet may defer cross-validation, which is acceptable;
- later validating `verify.md` should execute cross-validation against `spec.md` if `spec.md` already exists and is complete;
- today that second validation still defers because ready participants are only collected from the current invocation.

The user wants `readyParticipants` to be rehydrated/pre-populated from already-complete artifacts so the rule runs when the second artifact is validated.

# Approach / solution outline

The intended direction discussed with the user is:

- `readyParticipants` should not be limited to artifacts parsed in the current invocation;
- it should be pre-populated with artifacts already in `complete` when they are relevant participants for the cross-artifact rule being evaluated;
- for cross-validation to run correctly, the evaluator needs parsed/merged participant outputs available for those rehydrated participants, not just status flags;
- therefore the implementation likely needs to materialize or reload parsed outputs for already-complete participants when the second artifact is validated.

User hypothesis, confirmed as the intended direction for design:

- yes, ready participants should be pre-populated from `complete` artifacts;
- yes, cross-validation should have the merged/parsed artifacts available when running the rule.
