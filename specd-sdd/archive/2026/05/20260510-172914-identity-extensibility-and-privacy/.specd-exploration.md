# Exploration Context: Identity Extensibility and Privacy

Generated: 2026-05-10

## Problem Statement

The current `ActorIdentity` is too simple, only supporting `name` and `email`. This is insufficient for:

1. Supporting richer identity providers like LDAP or SSO that require additional metadata or internal IDs.
2. Privacy concerns in public repositories where exposing real emails is undesirable.

## Approach / Solution Outline

1. **Extend `ActorIdentity`**: Add optional fields `provider`, `providerId`, and `metadata: Record<string, string>` to the domain entity.
2. **Update Manifest**: Update the Zod schema and persistence logic in `manifest.json` to store these new fields.
3. **Privacy Configuration**: Add a `privacy` section to `specd.yaml` to allow users to configure identity obfuscation (e.g., `mode: 'hash' | 'mask' | 'anonymous'`).
4. **Privacy Decorator**: Implement a `PrivacyActorResolver` as a decorator that wraps any `ActorResolver` and applies the configured privacy rules.

## Design Decisions

- **Extensibility**: Use a `metadata` record for "unknown" future fields.
- **Privacy Decorator**: Use a decorator pattern for privacy to keep it independent of the identity provider.
- **Backwards Compatibility**: All new fields in `ActorIdentity` and `manifest.json` must be optional.

## Agreed Steps / Plan

1. **Phase 1: Research & Design**: Update specs to reflect the new requirements.
2. **Phase 2: Implementation**:
   - Update domain and infrastructure interfaces.
   - Implement the privacy configuration in `specd.yaml`.
   - Implement the `PrivacyActorResolver` decorator.
3. **Phase 3: Verification**: Ensure that identities are correctly stored and obfuscated according to config.
