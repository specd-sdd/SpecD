# Exploration: implementation-file-tracking

Generated: 2026-05-16

## Problem Statement

The user wants a way to record modified implementation files directly in the change manifest and then link these files (and specific symbols) with specs in the code graph. Currently, specd only tracks specs and their artifacts, but there is no formal link between the requirements (specs) and the code that implements them.

## Approach / Solution Outline

1. **Change Manifest Evolution**: Add `implementationCandidates` (automatic via VCS) and `implementationLinks` (explicit confirmation) to `manifest.json`.
2. **Automated Detection**: Update specd to automatically detect modified files using a VCS adapter (Git/Hg/Svn) and populate `implementationCandidates` in the manifest.
3. **Explicit Linking**: Introduce a new CLI command `specd change track-link` to link a specific symbol in a file to a spec.
4. **Permanent Materialization**: Formalize the `spec-lock.json` sidecar as the durable record for spec identity and implementation links. During `archive`, links from the manifest are moved to the target spec's `spec-lock.json`.
5. **Metadata Projection**: The `metadata.json` will act as a projection (cache) of the implementation links for fast consumption by tools like `code-graph`.
6. **Code Graph Integration**: `@specd/code-graph` will index these links to enable bi-directional traceability between requirements and code.

## Affected Areas

- `**core:change`\*\*: Entity logic for tracking links and candidates.
- `**core:change-manifest**`: Schema update for `manifest.json`.
- `**core:spec-lock` (New)\*\*: Formalize the sidecar for durable metadata.
- `**core:spec-metadata`\*\*: Project implementation links into `metadata.json`.
- `**core:archive-change**`: Materialize links during the archive process.
- `**core:vcs-adapter**`: Detect modified files.
- `**cli:change-status**`: Display candidates and links.
- `**cli:change-track-link` (New)\*\*: Command to register links.

## Design Decisions

- **Spec -> Symbol Linking**: The primary link level will be Spec to Symbol (function/class) within a file.
- **No Requirement IDs (yet)**: We will not force requirement IDs in specs for now to avoid breaking existing specs. Spec -> Symbol is a solid first step.
- **Sidecar Materialization**: Links will be stored in `spec-lock.json` (durable) and projected to `metadata.json` (performance).

## Agreements Reached

- The level of Spec -> Symbol is the point of balance between utility and effort.
- Automated detection via Git is a "safety net", while explicit linking is the "source of truth".
- `spec-lock.json` needs its own spec to formalize its role.
