# Problem statement

The current behavior in specd invalidates the whole artifact DAG when a previously validated artifact is validated again and drift is detected. This is coherent from a conservative review and approval perspective, because downstream artifacts may need to be redone if the changed artifact alters prior assumptions.

The user described a practical problem with this model during iterative design work: while an artifact is still being refined and changes are requested repeatedly, each modification invalidates all previously validated artifacts, even when the change is still local and the design is not yet closed. This makes normal iteration too expensive and noisy.

# Approach / solution outline

The conversation converged on separating two concerns that are currently too tightly coupled:

1. Global approval invalidation.
2. Artifact review invalidation scope.

The intended new model is:

- Automatic drift invalidation remains.
- Automatic invalidation should affect only the drifted artifact and downstream DAG impact by default.
- Upstream artifacts should not be reopened automatically just because a downstream or middle artifact changed.
- If review of the drifted artifact reveals semantic impact on earlier artifacts, those earlier artifacts should be reopened explicitly and traceably.

This leads to a design with two paths:

- Automatic invalidation by drift:
  - triggered by detected content drift
  - marks the drifted node as drifted
  - propagates review/blocking only downstream
  - invalidates any active approval snapshot globally if one exists

- Manual semantic review reopening:
  - explicit operator/agent action
  - used when a changed artifact is found to require reopening earlier or otherwise non-downstream artifacts
  - records a cause and free-text reason in history

# Key behavior agreed

## Automatic drift behavior

- Drift remains an internal automatic cause.
- The user does not want a manual `drift` flag in the CLI because drift should continue to be system-detected.
- The current global DAG downgrade is considered too aggressive.
- Proposed replacement:
  - automatic invalidation affects only the drifted node plus downstream impacted nodes
  - non-impacted siblings and ancestors should remain complete
  - global approvals/signoff snapshots should still be invalidated if they exist

## Manual invalidation / reopening behavior

The user proposed an explicit CLI command shape for semantic invalidation / review reopening:

`specd changes invalidate <change> --artifact <artifact-id> --spec <spec-id> --type <invalidation-type> --reason <reason-description>`

This was then refined semantically:

- No flags:
  - invalidate everything
- `--artifact <artifact-id>` only:
  - invalidate that artifact across all applicable specs
- `--artifact <artifact-id> --spec <spec-id>`:
  - invalidate only that specific artifact instance for that specific spec
- `drift` should not be exposed as a manual type because it is automatic
- A reason string is required so the event remains explainable and auditable

There was discussion about whether `--type` should really represent a state or a cause. A cleaner domain framing suggested during the conversation was:

- `type` / `cause` should represent why the artifact is reopened
- the resulting file state is a consequence of the command

The likely semantics discussed:

- manual targeted invalidation should move the targeted artifact/file(s) to `pending-review`
- `drifted-pending-review` should remain reserved for automatic drift detection
- downstream nodes should not necessarily be rewritten directly; they can be derived as blocked through lifecycle/DAG evaluation

# Example command shape discussed

The raw user-proposed command shape:

```bash
specd changes invalidate <change> \
  --artifact <artifact-id> \
  --spec <spec-id> \
  --type <invalidation-type> \
  --reason "<reason-description>"
```

Important semantics attached to it:

- no flags => all invalidated
- `--artifact` => that artifact for all specs
- `--artifact` + `--spec` => that concrete artifact file only
- type should correspond to manual review reopening semantics, not drift
- reason should be persisted in the event

# Design decisions reached

1. Automatic drift invalidation should become downstream-only by default.
2. Upstream reopening should not be inferred purely from DAG structure.
3. Upstream reopening should be explicit when semantic review determines it is needed.
4. Manual reopening should be represented by a dedicated CLI command on `changes`.
5. Manual reopening should persist a cause/type plus a human-readable reason.
6. `drifted-pending-review` remains an automatic state, not a user-specified one.
7. `pending-review` is the likely target state for manual reopening.
8. Approval invalidation and artifact review reopening should be treated as related but distinct concerns.

# Rejected alternatives

## Reject: always invalidate the entire DAG on any drift

Reason:

- too conservative for iterative artifact editing
- causes repeated reopening of unrelated validated work
- makes design refinement expensive and noisy

## Reject: automatic upward propagation through the DAG

Reason:

- a DAG `requires` relation models workflow/dependency order, not complete semantic impact
- if upward reopening is automatic, the system trends back toward global invalidation behavior
- many edits to a downstream or middle artifact do not truly require reopening upstream artifacts

## Reject: manual `drift` as a CLI invalidation type

Reason:

- drift is already automatically detected by the system
- exposing it manually weakens the meaning of drift as an observed state rather than a requested one

# Open questions for /specd-design

1. Should the CLI option be named `--type`, `--cause`, or `--kind`?
2. Should the new command be `change invalidate` or `changes invalidate` canonically, and what alias behavior should exist?
3. Should targeted manual invalidation append the same `invalidated` event type with a new cause, or should there be a distinct event for manual review reopening?
4. How should full invalidation with no flags behave at file-state level:
   - everything to `pending-review`
   - or maintain special cases for already drifted files?
5. Should descendants be directly downgraded on manual invalidation, or should only the explicit targets be downgraded and the rest appear blocked via `LifecycleEngine` effective status?
6. How should change-scoped artifacts behave with `--spec`:
   - most likely reject `--spec` for `scope: change`
7. Should manual invalidation invalidate active approvals/signoff globally every time, even for a narrow target?
8. Do `GetStatus` review summaries need richer reporting to distinguish:
   - automatic drift
   - explicit semantic reopening
9. Does the current `invalidated.cause` enum need extension, or should manual reopening reuse `artifact-review-required`?

# Conversation flow

1. The user raised a workflow problem: validating an already validated artifact currently invalidates the whole DAG, which is theoretically coherent under drift but too disruptive during iterative design changes.
2. Initial analysis separated approval invalidation from artifact review invalidation and suggested that invalidating the whole artifact DAG is too aggressive.
3. A first proposal was made: automatic invalidation should affect only the drifted artifact and downstream DAG impact.
4. The user challenged this with a key edge case: a change in `design.md` may imply `spec.md` must change even though `spec` is earlier in the graph.
5. The resulting refinement was:
   - downstream invalidation should be automatic
   - upstream reopening should exist, but only when semantic review explicitly determines it is needed
6. This led to discussion of an explicit CLI command to reopen/invalidate targeted artifacts and specs.
7. A first shape with `--artifact`, `--spec`, `--type`, and `--reason` was proposed.
8. The semantics were refined further:
   - no flags invalidates all
   - `--artifact` means all specs for that artifact
   - `--artifact` + `--spec` means one specific target
   - drift should remain automatic and not exposed as manual type
9. The user then requested using `specd-new` to formalize the work as a change.
