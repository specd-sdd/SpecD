# Exploration Context

## Problem Statement

Issue `#55` reports that archiving a change which introduces a new spec can partially materialize that spec into the permanent `specs/` tree, then fail on a later artifact and leave the repository plus change manifest in an inconsistent state.

Observed case from the issue:

- change: `unify-lifecycle-engine`
- new spec: `core:lifecycle-engine`

Observed failure sequence:

1. During archive, `spec.md` for `core:lifecycle-engine` was copied into the project spec tree.
2. Later in the same archive pass, `verify` failed because the change contained `deltas/core/lifecycle-engine/verify.md.delta.yaml`, but there was no base `specs/core/lifecycle-engine/verify.md` in the project.
3. Archive aborted.
4. Reload logic then treated `core:lifecycle-engine` as an existing spec because `specs/core/lifecycle-engine/spec.md` now existed in the repo.
5. Manifest filename normalization reinterpreted change artifact filenames as delta paths, causing drift and bouncing the change back to `designing`.

The user wants archive and reload behavior hardened so failed archive attempts do not corrupt repo state or reinterpret validated change artifacts incorrectly.

## Approach / Solution Outline

The requested change should cover four related concerns:

1. Archive application should be transactional or staged enough to avoid partial writes in the permanent spec tree when a later artifact fails.
2. Invalid new-spec artifact shapes should be rejected before archive.
3. Manifest reload / filename normalization should preserve validated intent and avoid silently flipping direct files into delta paths based on coarse repo existence checks.
4. Lifecycle handling around `archiving` should avoid misleading intermediate state when archive is blocked before meaningful archive work begins.

Recommended execution order from the issue:

1. Fix archive atomicity/staging first.
2. Add early validation for invalid new-spec direct/delta mixes.
3. Fix manifest reload/normalization semantics.
4. Revisit lifecycle timing / rollback around `archiving`.
5. Add regression tests for all reported scenarios.
