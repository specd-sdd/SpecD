# Exploration: archived-change-read-only

## Problem statement

After `drafted-change-read-only`, archived changes still expose very little information through specd APIs and CLI. An archived change directory on disk retains the full change tree (`manifest.json`, `proposal.md`, `design.md`, `tasks.md`, `deltas/`, `specs/`, `reports/`), but `ArchiveRepository.get()` returns a slim `ArchivedChange` index projection (name, archivedAt, archivedBy, artifact type IDs, specIds, schema). `archive show` only prints name, hardcoded `archivable` state, specs, and schema — not description, history, artifact states, or paths to archived files.

Users and agents cannot inspect archived work through the same read models used for drafted/discarded changes without manually reading the filesystem.

## Approach / solution outline

Reuse the read-only change view pattern from `drafted-change-read-only`:

1. **`ArchivedChangeIndexEntry`** — lightweight index row for fast listing (from `index.jsonl`, no manifest I/O).
2. **`ArchivedChange`** — full read-only detail loaded from `manifest.json` in the archive directory (manifest-backed), analogous to `DraftedChangeView` / `DiscardedChangeView` but for archived storage.

Split responsibilities:

- **`list()`** → `ArchivedChangeIndexEntry[]` (fast, index-only).
- **`get(name)`** → `ArchivedChange | null` (full detail, reads manifest).

Extend or align with `ReadOnlyChangeView` where fields overlap (`name`, `createdAt`, `description`, `state`, `specIds`, `workspaces`, `schemaName`, `schemaVersion`, `artifacts` map, `history`), plus archive-specific fields: `archivedAt`, `archivedBy`, `archivedName`.

Do **not** expose mutable `Change` from archive paths. Archive remains append-only — no `mutateArchived`.

## Design decisions (agreed)

| Decision                                  | Rationale                                                                                                                                                           |
| ----------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `ArchivedChange` = full read-only detail  | User explicitly chose this naming: detail type is `ArchivedChange`, not a separate `ArchivedChangeView` name (though it may implement shared read-only interfaces). |
| `ArchivedChangeIndexEntry` for list/index | Fast listing must not read every `manifest.json`; index.jsonl already has enough for list UIs.                                                                      |
| `get(name)` loads manifest                | Single-name lookup should return full inspectable data — user: "si alguien hace un get específico necesita el full".                                                |
| `list` stays fast                         | Only list use case needs index-only performance.                                                                                                                    |
| Reuse read-only pattern from drafts       | Same defence-in-depth: repo split + read-only projection, no mutable `Change` on read paths.                                                                        |

## Rejected alternatives

| Alternative                                                                      | Why rejected                                                                             |
| -------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------- |
| Keep `ArchivedChange` as index summary only; add `ArchivedChangeView` for detail | User preferred `ArchivedChange` as the full detail type and a separate index entry type. |
| Make `ArchivedChange` implement only index fields (Option C)                     | Does not solve "lost info" problem.                                                      |
| Single type for both list and get with lazy loading                              | User agreed list vs get should differ; explicit index entry type is clearer.             |
| `get` active-only + flag for archive                                             | Not applicable; archive is separate repository, not `ChangeRepository`.                  |

## Relationship to drafted-change-read-only

Completed and archived change `drafted-change-read-only` (20260526-121447). Introduced:

- `ReadOnlyChangeView`, `DraftedChangeView`, `DiscardedChangeView`
- `getDraft` / `getDiscarded`, `mutateDraft`
- Active-only `get`/`mutate`

This change extends the same philosophy to **archived** storage under `ArchiveRepository`, not `ChangeRepository`.

## Open questions (for /specd-design)

1. Should `ArchivedChange` **extend** `ReadOnlyChangeView` interface or compose via facade like drafts?
2. Fixed `state` for archived detail: always report `archivable` (current CLI) or last persisted lifecycle state from manifest?
3. `archivePath()`: accept `ArchivedChange` detail only, or also `ArchivedChangeIndexEntry` (by name lookup)?
4. Breaking change: `ListArchived` return type changes from `ArchivedChange[]` to `ArchivedChangeIndexEntry[]` — document migration for any external consumers.
5. New spec `core:archived-change-index-entry` vs fold into `archive-repository-port`?
6. Should `archive list` JSON include more fields from index entry only, or stay as today?
7. Future: `archive context` / read deltas from archive dir (out of scope unless user expands)?
