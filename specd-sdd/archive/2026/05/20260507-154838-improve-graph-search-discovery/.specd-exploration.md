# Exploration Context: improve-graph-search-discovery

## Problem Statement

The `graph search` command in the SpecD CLI currently uses strict `AND` logic when a user provides multiple search terms. While this acutally filters results, it prevents discovery of related concepts that aren't contained within a single symbol's name or JSDoc. For example, searching for "effectiveStatus findBlockingParent" returns 0 results because no single symbol contains both terms, even though they are closely related in the lifecycle domain.

## Approach / Solution Outline

Modify the query sanitization logic to use `OR` instead of `AND`.

- **SQLite implementation:** Update `sanitizeFtsQuery` in `packages/code-graph/src/infrastructure/sqlite/sqlite-graph-store.ts` to join tokens with `OR`.
- **Ranking:** Rely on the existing BM25 scoring. Symbols or specs containing _more_ of the search terms will naturally rank higher, providing a "Google-like" discovery experience while still surfacing individual matches.

- `packages/code-graph/src/infrastructure/ladybug/ladybug-graph-store.ts` (to ensure parity)

## Design Decisions

- **Transition from AND to OR:** Confirmed that `OR` is the more "natural" expectation for a discovery tool.
- **Keep BM25:** The scoring mechanism already handles relevance, so `OR` won't degrade the quality of top results.

## Key Codebase Observations

- `sanitizeFtsQuery` currently uses `.join(' AND ')`.
- `expandSymbolName` is used to index symbols with parts (e.g., `handleError` -> `handleError handle error`), which already aids in multi-term matching but doesn't solve the cross-symbol discovery issue.
- Discovery tests confirmed that individual searches for the terms return results, but the combined search fails due to the current `AND` logic.
