# Exploration Context

## Problem statement

The user reported a non-atomic archive failure mode in `specd`.

Current behavior as described:

- Archive starts merging some canonical spec artifacts.
- A later check, currently observed with structural metadata extraction validation, runs after canonical archive publication has already started.
- When that later check fails, some specs may already be partially merged or fully merged.
- This leaves the project in an inconsistent state where later archive attempts can fail or behave incorrectly because the prior archive already mutated canonical specs.

Required outcome:

- Archive must not begin canonical publication until all checks that can fail have already completed.
- This includes structural checks and any other archive-time checks that may reject the archive attempt.
- If any such check fails, no canonical spec artifact or sidecar publication may have begun.

The user wants a new change that fixes archive atomicity so archive only starts once everything has been checked.

Interpreted design intent:

- The preflight phase must be complete before archive starts any canonical write.
- The guarantee is broader than `metadataExtraction`; it applies to all validations and consistency checks that can still fail during archive.

## Relevant spec findings

### `core:archive-change`

The current spec already contains archive atomicity language and several archive-phase guarantees, including:

- archive must happen atomically and in the right order
- prepare archive plan before permanent writes
- later artifact failure prevents permanent writes
- delta base is checked during prepare phase
- pre-publication failure leaves no canonical spec writes
- staged archive output is preserved for manual recovery on publication failure
- archive currently guarantees atomic publication per spec, not necessarily one filesystem transaction for the entire multi-spec batch

Important current requirement shape:

- It already distinguishes preparation from canonical publication.
- It already mentions metadata consistency checks before canonical publication for some cases.
- The user reports the real implementation still allows a late failure after canonical writes begin.

### `core:spec-metadata`

The current spec says archive-time flow for a modified spec is:

1. Prepare merged canonical artifact content in memory.
2. Determine final persisted `dependsOn`.
3. Run `extractMetadata()` over prepared merged content to validate archive-time consistency.
4. Determine final `spec-lock.json` content.
5. Publish canonical spec artifacts plus `spec-lock.json`.
6. After publication succeeds, run `GenerateSpecMetadata` against canonical persisted spec.
7. Persist `metadata.json`.

This suggests the intended design already places metadata consistency checks before publication, but the user has observed a live failure mode where at least one failing check still occurs after archive publication starts.

### `core:save-spec-metadata`

The current spec explicitly says:

- `SaveSpecMetadata` persists `metadata.json` only.
- Sidecar consistency is an archive concern enforced by `ArchiveChange`, not by `SaveSpecMetadata`.

This indicates the fix is likely centered on archive orchestration rather than moving logic into `SaveSpecMetadata`.

## Design direction implied by the conversation

Likely design direction:

- Introduce or clarify a full archive preflight phase.
- Require that every archive-time operation that can still fail before a successful archive result must run during preflight.
- Canonical publication may start only after preflight completes successfully.
- Separate:
  - preflight failure: zero canonical writes
  - publication failure after successful preflight: preserve staged output and avoid partial canonical publication
- Make the spec language stronger than the current metadata-specific wording so it covers all fail-able checks, not only structural metadata extraction.
