# Exploration: move-impl-tracking-to-cli

## Problem statement

Today, implementation file autodetection runs inside core use cases (`GetStatus`, `TransitionChange`) whenever a change has historically entered `implementing`. That is correct behavior for keeping tracked files fresh, but it creates redundancy when another entry point (e.g. `specd changes implementation` or a future dedicated refresh) also performs tracking: the same VCS-backed detection can run multiple times in one user flow.

The user wants autodetection to happen at **CLI orchestration** for `changes status` and `changes transition`, not inside those core use cases.

## Approach / solution outline

1. **Remove** implementation autodetection side effects from `GetStatus` and `TransitionChange` in `@specd/core`.
2. **Add** CLI-layer orchestration in `cli:change-status` and `cli:change-transition` that runs targeted implementation refresh **before** calling the respective core use case, under the same guard (`hasEnteredImplementing` / historical implementing signal from `core:change`).
3. Prefer a **single reusable core operation** for “refresh implementation tracking from detector” (existing `UpdateImplementationTracking` or a thin sync use case) so CLI, and optionally `cli:change-implementation`, share one code path instead of duplicating detector + persist logic.
4. `GetStatus` / `TransitionChange` continue to **project** `implementationTracking` from the loaded `Change` entity; they just stop invoking `ImplementationDetector`.

## Design decisions (confirmed)

- Tracking/autodetection belongs at **CLI entry points** for `status` and `transition`, not in core use cases that are also used as pure “read state” / “apply transition” operations.
- Original intent was “always track when CLI calls these use cases”; moving orchestration to CLI preserves that intent without baking side effects into core.
- User confirmed change name `move-impl-tracking-to-cli` and the four initial specs on 2026-05-23.

## Rejected alternatives

- **Keep autodetection in core and dedupe inside detector** — user prefers architectural separation (CLI orchestrates side effects) over internal idempotency hacks.
- **Remove autodetection entirely from status/transition** — rejected; still needed at those CLI moments, just not inside core use cases.

## Open questions (for `/specd-design`)

1. **MCP (future)**: Long-running MCP will use **filesystem watcher** to update tracked files—not `RefreshImplementationTracking`, not detection inside GetStatus/CompileContext/TransitionChange. Main reason for removing tracking from those use cases. Out of scope for this change.
2. **Tests / direct kernel callers**: May call `RefreshImplementationTracking` explicitly when VCS refresh is needed, or use persisted state only.
3. **`cli:change-implementation list`**: Only show persisted state (no autodetect on list).
4. **New use case**: `core:refresh-implementation-tracking` (`RefreshImplementationTracking`).
5. **Kernel wiring**: Drop detector from GetStatus, TransitionChange, CompileContext; wire refresh use case; CLI orchestrates before status/transition/context.
6. **CompileContext** — **IN SCOPE** (user 2026-05-23): `cli:change-context` orchestrates refresh before `CompileContext`, same as status/transition.
