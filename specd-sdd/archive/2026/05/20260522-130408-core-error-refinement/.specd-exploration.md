# Exploration Context: core-error-refinement

Generated: 2026-05-22

## Problem Statement

The `@specd/core` package contains several instances of generic `throw new Error()` calls in critical use cases (like `ArchiveChange` and `UpdateSpecDeps`). These generic errors lack machine-readable codes and proper typed identity, causing the CLI to display full stack traces instead of user-friendly, actionable feedback. This violates the "AI blindness" mitigation principles and project conventions for robust error handling.

## Approach / Solution Outline

1. **Typed Errors:** Create specific `SpecdError` subclasses for all identified generic error scenarios in `core:domain/errors`.
2. **Refactor Use Cases:** Replace `throw new Error()` with the new typed errors in `ArchiveChange.ts`, `UpdateSpecDeps.ts`, and potentially others.
3. **New Specification:** Create a new spec `core:error-handling-conventions` (or similar) to formalize when to use `SpecdError`, how to structure them (code, message, metadata), and where they should live.
4. **Update Existing Specs:** Ensure `core:archive-change` and others explicitly document these errors in their `verify.md` and JSDoc `@throws` tags.

## Key Codebase Observations

- `ArchiveChange.ts` throws generic errors for:
  - Dependency mismatches (`extracted vs persisted`).
  - Missing tracked/base artifacts.
  - Open implementation files.
  - Out-of-scope implementation sidecars.
- `UpdateSpecDeps.ts` throws generic errors for:
  - Mutually exclusive flags.
  - Missing mandatory flags.
  - Dependency not found.
- The `SpecdError` base class exists and requires a machine-readable `code`.

## Agreements Reached

- The change must include a new specification for error handling conventions in the core.
- The primary goal is to eliminate stack traces for user-error scenarios in the CLI.

## Open Questions
