# Problem statement

The user wants to explore how to support richer validation in `schema.yaml` using the current validation system where possible, or extending it where necessary.

Primary goals discussed:

- Add **cross-artifact validations**
- Add **intra-artifact validations that go beyond existence checks**

Concrete examples given by the user:

- Every `Requirement` in `spec.md` should have a matching `Requirement` in `verify.md`
- A single file should not allow duplicate sections of the same kind / same identity
- Validations should not be limited to "section exists"; they should also express deeper structural and consistency constraints

# Approach / solution outline

The likely direction is to extend the schema-driven validation model so it can express:

- Cross-artifact consistency checks between `spec.md` and `verify.md`
- Richer intra-artifact constraints such as uniqueness/cardinality
- Requirement mirroring / coverage rules

The likely implementation center is the existing validation pipeline rather than a separate ad hoc validator, because:

- `ValidateArtifacts` is already the validation chokepoint during change workflow
- `ValidateSpecs` already validates archived/project specs
- `rule-evaluator` already owns rule execution semantics for AST-based validation

Based on discovery, the current model is strongly artifact-local. Existing `validations` and `deltaValidations` operate over a single AST root at a time, so true cross-artifact correlation is not a first-class concept yet.

# Design decisions and conclusions reached

- The change should be modeled as a schema/validation capability change, not as a one-off rule in one validator.
- `ValidateArtifacts` should remain the enforcement point for validations during active changes.
- `ValidateSpecs` should likely participate if the new validation model must also apply to already-existing specs outside a change.
- The rule model probably needs new semantics or new schema constructs for:
  - cross-artifact matching
  - uniqueness / duplicate detection
  - requirement coverage / mirroring
- This is not well served by a narrow patch only in docs or examples.
