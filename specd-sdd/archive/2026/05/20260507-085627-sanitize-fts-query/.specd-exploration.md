# Exploration Context: sanitize-fts-query

## Problem statement

Running `graph search "effectiveStatus findBlockingParent pending-parent-artifact-review overlapDetail" --symbols --format toon` throws `SqliteError: no such column: parent`. The FTS5 MATCH clause interprets hyphens as NOT operators and `parent` as a column filter, causing a runtime error.

The root cause: both `searchSymbols()` and `searchSpecs()` in **both** graph store implementations (SQLite and Ladybug) pass user query strings directly to FTS without sanitizing FTS-specific syntax.

## Approach / solution outline

Add a shared `sanitizeFtsQuery()` utility function that:

1. Splits the query into individual terms (by whitespace)
2. Wraps each term in double quotes (FTS5 phrase literal)
3. Joins them with `AND`
4. Handles edge cases: empty terms, terms that already contain quotes, special characters

The function should be used in:

- `sqlite-graph-store.ts`: `searchSymbols()` (line 447), `searchSpecs()` (line 505)
- `ladybug-graph-store.ts`: `searchSymbols()` (line 999), `searchSpecs()` (line 1048)

## Design decisions

- Sanitize at the store level, not at the CLI/provider level — stores are the boundary where FTS syntax matters
- Wrap each whitespace-separated token in double quotes to make FTS5 treat them as literal phrases
- Join with `AND` so multi-word queries still match all terms
- For Ladybug, the same approach works because its FTS uses similar query semantics

## Audit findings (comprehensive)

### SQLite (`sqlite-graph-store.ts`) — 59 methods analyzed

**SQL injection:** SAFE. All queries use `?` parameterized bindings. No user input interpolated into SQL.

**Dynamic SQL:** 4 methods (`findSymbols`, `deleteFileLocalState`, `getRelationsBySourceTypes`, `getRelationsByTargetTypes`) use dynamic `?` placeholder generation — safe pattern.

**FTS MATCH issue:**

- `searchSymbols()` (l447): `options.query.trim()` passed directly to `MATCH ?` — NO FTS syntax sanitization
- `searchSpecs()` (l505): identical issue
- FTS5 interprets: `-` as NOT, `AND`/`OR`/`NOT` as boolean ops, `column:term` as column filter, `*` as wildcard
- The error `no such column: parent` comes from `pending-parent-artifact-review` where FTS5 sees `pending NOT parent NOT artifact NOT review` and `parent` is treated as a column name

### Ladybug (`ladybug-graph-store.ts`) — 38 methods analyzed

**Cypher injection:** SAFE. All user input passes through `this.escape()` which escapes `\`, `'`, `\n`, `\r`.

**FTS issue:**

- `searchSymbols()` (l999): `options.query` passed through `this.escape()` (Cypher-safe) but NOT sanitized for FTS operators
- `searchSpecs()` (l1048): identical issue
- Ladybug uses `CALL QUERY_FTS_INDEX(...)` with Porter stemmer — FTS operators can alter behavior similarly

**Other observations:**

- `this.escape()` only handles Cypher string escaping, not FTS syntax
- `createFtsIndex()` uses raw string interpolation for table/index/column names — currently safe (hardcoded callers only)
- `COPY` commands use internally-generated CSV paths — safe
- `RelationType` enum values used as Cypher labels/table names — safe via TypeScript typing
