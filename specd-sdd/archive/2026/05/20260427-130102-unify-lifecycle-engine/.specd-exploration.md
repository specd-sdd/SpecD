# Exploration: Lifecycle Engine Unification

Generated: 2026-04-27

## Problem Statement

The `specd change context` output (via `CompileContext`) reports workflow steps as "available" based solely on artifact presence, ignoring the domain's state machine protocol (`VALID_TRANSITIONS`) and internal task completion status. This creates a "liar context" where AI agents see steps as available that will actually be rejected by the CLI upon transition attempt.

## Approach / Solution Outline

Extract the comprehensive lifecycle validation logic currently residing in `GetStatus` into a shared domain service: **`LifecycleEngine`** (or `ChangeLifecycleService`).

This service will unify the "triple validation":

1. **Protocol Validation**: Checking against the `VALID_TRANSITIONS` state graph.
2. **Artifact Validation**: Checking schema-defined `requires`.
3. **Content Validation**: Checking `requiresTaskCompletion` for `- [ ]` items.

The engine will be consumed by:

- `GetStatus`: To report state and blockers.
- `CompileContext`: To filter `availableSteps` accurately for agents.
- `TransitionChange`: To validate moves before execution.
- `MCP Server`: To expose valid tools/actions to external agents.
- `ArchiveChange`: To ensure closure criteria are met.

## Design Decisions & Agreements

- The engine must provide machine-readable **Repair Blockers** (codes and messages).
- The engine should derive the **Next Action** recommendation to ensure consistency between the status report and the agent's context.
- The schema's `workflow` section remains the definition of "what should happen" at each step, while the engine provides the "when and how" it is validated.

## Key Codebase Observations

- `GetStatus._computeLifecycleContext` already implements most of this logic but is not easily accessible from other use cases.
- `CompileContext` currently performs a redundant and shallow check of `workflowStep.requires`.
- `TransitionChange` performs manual checks before calling `change.transition()`.

## Open Questions

- Should the `LifecycleEngine` also handle `ReviewSummary` logic (drift detection, overlap conflicts), or keep that separate in `GetStatus`?
- Should we introduce a new `AvailableStep` field in the context output to distinguish between "Artifacts ready" and "Transition permitted"?
