# Exploration Context: shared-skill-bundle-targets

Generated: 2026-04-29

## Problem Statement

The current agent plugin skill installation flow duplicates shared skill files into every installed skill directory. The user wants a minimal design that lets shared files be copied once during installation and referenced from skills, for example through paths like `@../_specd-shared/shared.md`.

The immediate goal is to create a specd change, not to implement it yet.

## Conversation Flow

The discussion started with the question of how to make shared skill files copy only once during install while still being readable from skills. Several possible approaches were discussed, including a shared directory, a content-addressed store, a manifest, symlinks, and versioned cache directories.

The discussion narrowed to a shared directory under the agent skills root, such as `_specd-shared/`, with skills referencing shared content via a relative path. The user asked whether a syntax like `@../_specd-shared/shared.md` would make sense, because existing skills currently use `@shared.md`.

Web research found that the Agent Skills standard treats a skill as a directory with `SKILL.md` and optional bundled resources such as `scripts/`, `references/`, and `assets/`. It does not define a standard shared directory across multiple skills. However, clients generally discover only subdirectories containing `SKILL.md`, so a `_specd-shared/` directory without `SKILL.md` should not be discovered as a skill. This makes `_specd-shared/` viable as a specd convention, even if it is not part of the portable standard.

The user then asked what support currently exists in agent plugin installation for writing shared files to other folders. Local inspection showed that current support is not enough: shared templates exist in `@specd/skills`, but once resolved they are emitted as ordinary files inside each skill bundle and every agent plugin writes all bundle files into the skill-specific directory.

The user proposed the minimal design: mark which files in the bundle are shared, support two install directories, one normal and one shared, and make the shared directory optional so the normal directory is used when no shared directory is passed. This was accepted as the intended design direction.

## Current Behavior Observed

Shared templates currently live under:

- `packages/skills/templates/shared/shared.md`
- `packages/skills/templates/shared/shared.meta.json`

The metadata declares a shared `filename` and the skills that consume it.

`packages/skills/src/infrastructure/repository/skill-repository.ts` scans `templates/shared/` through `listSharedFiles()`. During `getBundle(name, ...)`, it appends matching shared files into `bundle.files` as normal resolved files.

`packages/skills/src/domain/skill-bundle.ts` currently defines `ResolvedFile` with only:

- `filename: string`
- `content: string`

`SkillBundle.install(targetDir: string)` currently accepts only one target directory.

Each agent plugin installer uses the same pattern:

- compute agent skills root, e.g. `.codex/skills`, `.claude/skills`, `.github/skills`, `.opencode/skills`
- compute `skillDir = path.join(targetDir, skillName)`
- write every `bundle.files` entry to `path.join(skillDir, file.filename)`
- prepend runtime-specific frontmatter for markdown files

Relevant installer files:

- `packages/plugin-agent-codex/src/application/use-cases/install-skills.ts`
- `packages/plugin-agent-claude/src/application/use-cases/install-skills.ts`
- `packages/plugin-agent-copilot/src/application/use-cases/install-skills.ts`
- `packages/plugin-agent-opencode/src/application/use-cases/install-skills.ts`

`packages/plugin-manager/src/domain/types/agent-plugin.ts` currently defines `AgentInstallOptions` with only:

- `skills?: readonly string[]`
- `variables?: Record<string, string>`

There is no current result field for shared installed assets.

## Intended Design Direction

The minimal change should mark shared files in the resolved bundle instead of relying on filenames or installer-specific inference.

A likely model:

```typescript
export interface ResolvedFile {
  readonly filename: string
  readonly content: string
  readonly shared?: boolean
}
```

Bundle install should support two target directories:

```typescript
install(options: {
  targetDir: string
  sharedTargetDir?: string
}): Promise<void>
```

The behavioral rule should be:

- non-shared files write to `targetDir/<filename>`
- files with `shared === true` write to `(sharedTargetDir ?? targetDir)/<filename>`
- when `sharedTargetDir` is omitted, current behavior is preserved

Agent plugins can then use:

- normal target: `<agentSkillsRoot>/<skillName>/`
- shared target: `<agentSkillsRoot>/_specd-shared/`

This supports skill references such as:

```markdown
Read @../\_specd-shared/shared.md before doing anything.
```

## Design Decisions

- The shared marker should live on resolved bundle files, because that is the point where installers can make routing decisions without knowing template-source details.
- The shared install directory should be optional, defaulting to the normal install directory to preserve current behavior for simple consumers.
- The design should remain minimal. Do not introduce content hashes, manifests, symlinks, or versioned stores in this change unless later design work finds a concrete need.
- Shared files should not be inferred by filename such as `shared.md`; the repository should provide explicit metadata through the bundle.
- `_specd-shared/` should not contain `SKILL.md`, so agent discovery does not treat it as a skill.
- The likely installed path for shared files is under the agent's skills root, for example `.codex/skills/_specd-shared/shared.md` or `.opencode/skills/_specd-shared/shared.md`.

## Rejected Alternatives

- Content-addressed store by hash was considered but rejected for this change because it adds complexity that is not needed for the current problem.
- Manifest-based resolution was considered but rejected for the minimal design because existing skills can use direct relative paths.
- Symlinks or hardlinks were considered but rejected as the primary mechanism because of portability and packaging concerns.
- Keeping a local full copy of `shared.md` in every skill was rejected because the goal is to avoid duplication.
- A local stub `shared.md` in each skill remains a possible fallback for clients that cannot resolve `..` paths, but it is not the preferred minimal design if direct `@../_specd-shared/shared.md` works.

## Open Questions

- Whether `SkillBundle.install()` should remain backward-compatible with `install(targetDir: string)` while adding an object overload, or whether changing the signature is acceptable.
- Whether `AgentInstallOptions` should expose a configurable shared target directory, or whether each plugin should hardcode `_specd-shared` under its own skills root.
- Whether `AgentInstallResult` should report shared files or shared install paths, or whether reporting installed skills is enough.
- Whether uninstall with a selected skill filter should remove shared files immediately, leave them in place, or only remove them when uninstalling all skills.
- Whether installers should deduplicate repeated shared file writes across multiple skills in one install run. Idempotent overwrites are simpler, but deduplication avoids unnecessary writes.
- Whether shared files with the same filename but different content should be detected as conflicts. This is not currently needed with the existing single `shared.md`, but could matter later.
- Whether frontmatter injection must skip shared markdown files. Since shared files are not skills, they probably should not receive agent frontmatter.

## Key Codebase Observations

- `packages/skills/src/infrastructure/repository/skill-repository.ts` already knows which files are shared because `listSharedFiles()` returns `SharedFile` entries and `getBundle()` appends them to bundles.
- `packages/skills/src/application/ports/skill-repository.ts` exposes `SharedFile` and `listSharedFiles()`.
- `packages/skills/src/domain/skill-bundle.ts` is the likely place to extend `ResolvedFile` and `SkillBundle.install()`.
- All four agent plugin installers currently duplicate similar installation loops. The same behavioral change will likely need to be applied consistently to all four.
- Current plugin specs define install locations as agent skills roots only. They do not mention shared subdirectories yet.
- Current skill templates reference `@shared.md`, for example in `packages/skills/templates/specd/SKILL.md` and other specd lifecycle skill templates. These references will need design attention if shared files move to `_specd-shared/`.
- `node packages/cli/dist/index.js project status --format toon` reported the code graph as stale, so graph impact results were not used for final scope decisions.

## User Preferences

- The user wants the minimum viable change.
- The user explicitly prefers marking bundle files as shared and supporting normal plus shared install directories.
- The shared install directory should be optional, with fallback to the normal directory.
- The user is exploring design through specd workflow and has not asked for implementation yet.
