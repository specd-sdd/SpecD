## Current Code Changes That This Change Likely Needs To Spec

### 1. Indexer staging to `.specd/tmp`

The indexer no longer behaves as if all intermediate `allFiles`, `allSymbols`, and `allRelations` must stay in memory until final persistence. Instead, current code stages analysis results to repo-local temporary files under `.specd/tmp/index-stage-*`.

This is a meaningful behavior/design change relative to the older conceptual model and likely belongs in the indexer spec.

### 2. Cleanup of staged index artifacts

The indexer now cleans up staging directories even when indexing exits unsuccessfully. This also likely belongs in the indexer spec if the spec currently implies only in-memory accumulation or does not mention cleanup expectations for staged intermediate artifacts.

### 3. Graph-store temp CSV location moved to repo-local `.specd/tmp`

The Ladybug graph store no longer relies solely on system temp directories for bulk-load CSVs. Current implementation uses repo-local `.specd/tmp` paths instead, and relation-loading temp files were aligned with that behavior as well.

This likely belongs in the graph-store spec if temporary artifact placement or cleanup behavior is described there, or if the spec should now state repo-local temporary storage expectations.

### 4. PHP language-adapter reduced duplicate parsing

The PHP adapter gained behavior that avoids a redundant parse when extracting namespace and symbols. A method/contract extension was introduced so namespace and symbols can be derived together in one pass.

This likely belongs in the language-adapter spec, at least as an allowed/expected capability or behavior. The change is internal but non-trivial and may need to be reflected if the spec currently describes the adapter contract too narrowly.

## Important Technical Findings From The Debugging Session

### Ladybug/lbug instability under indexing load

The debugging work strongly suggested that many observed failures are not simple business-logic bugs. Instead, they appear related to LadybugDB/lbug behavior under heavy indexing load, including:

- instability around `COPY`
- slow or pathological first queries after analysis
- large RSS growth that was not explained by JS heap alone
- sensitivity around FTS and connection/session handling

This matters for spec work because the specs should describe the intended behavior and architecture, but should avoid locking in debugging hacks that were only exploratory.

### Memory growth was primarily in `pass1`

Instrumentation showed that the biggest RSS jump happened during `pass1`, not in hashing or spec discovery. The PHP parser path was a major contributor.

### Duplicate PHP parsing mattered a lot

A specific optimization reducing duplicate PHP parses significantly reduced RSS, roughly by GiB-scale amounts in prior experiments. This is strong evidence that the PHP adapter behavior changed materially and should be reflected if the language-adapter spec currently describes a simpler or older shape.

### The apparent spec-discovery slowdown was misleading

The "Discovering specs" phase was not the real culprit. Investigation showed the delay was tied to store queries and process state, not to scanning a specs directory. This is relevant mostly as design context, not necessarily as something to encode into specs.
