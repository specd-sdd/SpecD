# Design: public-web-site

## Non-goals

- Publishing ADR content on the public website
- Rewriting the repository documentation corpus into a second handwritten docs tree under `apps/public-web`
- Expanding the initial API reference scope beyond `@specd/core`
- Choosing or implementing production deployment infrastructure beyond keeping the app buildable as a static site

## Affected areas

- `package.json`
  Change: add root convenience scripts for the new public web workspace and include the new workspace in the root `test` flow.
  Callers: file-level config only · Risk: LOW
  Note: `graph impact --file package.json` reports `riskLevel: LOW` with no dependents.

- `pnpm-workspace.yaml`
  Change: add `apps/*` so `pnpm` recognises `apps/public-web` as a workspace package.
  Callers: file-level config only · Risk: LOW
  Note: `graph impact --file pnpm-workspace.yaml` reports `riskLevel: LOW` with no dependents.

- `turbo.json`
  Change: ensure the root build pipeline accounts for Docusaurus output under `build/**` instead of only `dist/**`.
  Callers: file-level config only · Risk: LOW

- `.gitignore`
  Change: ignore app-specific generated artifacts such as `build/`, `.docusaurus/`, and generated API docs output.
  Callers: file-level config only · Risk: LOW

- `eslint.config.js`
  Change: extend lint coverage to `apps/*/src/**/*.{ts,tsx}` and add a narrow exception for framework-mandated `default export` entrypoints in `apps/public-web`.
  Callers: file-level config only · Risk: MEDIUM
  Note: this is the main repo-level safeguard that keeps the Docusaurus exception narrow rather than silently weakening the global convention.

## Approach

The implementation should proceed in four layers.

First, wire the monorepo so `apps/public-web` behaves like a normal workspace. That means adding `apps/*` to `pnpm-workspace.yaml`, adding root convenience scripts in `package.json`, updating `turbo.json` for Docusaurus `build/**` outputs, and ignoring generated app artifacts in `.gitignore`.

Second, create the app shell itself in `apps/public-web` with Docusaurus and a small named-export component structure. The public root route should be a thin Docusaurus entrypoint that delegates to named components so the framework-required `default export` exception stays minimal and localised.

Third, mount public docs from the repository `docs/` tree directly instead of copying them into a second handwritten docs directory. The docs plugin should point at `../../docs`, exclude `adr/**`, and curate navigation through `sidebars.ts`. This satisfies `public-web:public-site` while preserving the repository `docs/` tree as the authored source of truth.

Fourth, generate the API reference into an app-local derived directory such as `apps/public-web/.generated/api` and mount that directory as a separate Docusaurus docs instance at `/api`. The generation step must start with `../../packages/core/src/index.ts`, satisfying `public-web:api-reference` without introducing handwritten duplicate API docs.

No existing spec is modified, so there is no spec ripple effect to resolve. The design is additive and concentrated around new app code plus repo-level wiring.

If Docusaurus ingestion later requires doc metadata changes, those edits must happen in the root `docs/` files, not in duplicated copies under the app workspace.

## Key decisions

**Use Docusaurus as the public site framework** → it matches the requested tool choice and supports a homepage, docs, and API reference in one static site. **Alternatives rejected** → VitePress or a custom React app would reduce friction around `default export`, but they would move away from the requested framework and require more bespoke docs wiring.

**Read the repository `docs/` tree directly** → this keeps `docs/` as the only handwritten documentation source of truth and aligns with `default:_global/docs`. **Alternatives rejected** → syncing docs into `apps/public-web/docs` would create a second documentation tree that can drift, even if the drift starts as “generated”.

**Generate API docs into `.generated/api` inside the app** → this keeps generated reference output derived and disposable while allowing the public site to mount it as `/api`. **Alternatives rejected** → committing generated API docs into the root `docs/` tree would mix authored and generated material and create churn in normal docs review.

**Split the change into two specs** → `public-web:public-site` and `public-web:api-reference` separate site behaviour from generated API reference concerns, which keeps verification and future evolution cleaner. **Alternatives rejected** → a single broad spec would be shorter initially but would blur the boundary between docs UX and API generation behaviour.

**Allow only framework-mandated `default export` exceptions** → this contains the Docusaurus conflict with the global convention instead of silently weakening the rule for the whole app. **Alternatives rejected** → a blanket “apps can use default exports” rule would be broader than the actual need.

## Trade-offs

- `[Framework convention mismatch]` Docusaurus requires some `default export` entrypoints → Mitigation: restrict the exception to route/config entrypoints and keep all reusable modules on named exports.
- `[Docs structure mismatch]` the root `docs/` tree was not authored specifically for Docusaurus navigation → Mitigation: curate sidebars explicitly and exclude `adr/**` at the docs-plugin boundary.
- `[Graph/package identity drift]` the graph index already reported `Package identity collision: "specd" already mapped to workspace "default"` for `public-web:<manifest>` → Mitigation: create `apps/public-web/package.json` with a unique workspace name (`@specd/public-web`) as one of the first implementation steps.
- `[Root script drift]` the current root `test` script filters only `@specd/core`, `@specd/cli`, and `@specd/code-graph` → Mitigation: add `@specd/public-web` to the root test flow once the app has tests.

## Testing

### Automated tests

- `apps/public-web/test/lib/public-docs-config.spec.ts`
  Assert that the public docs source path points to `../../docs`, ADRs are excluded, and the initial API entrypoints include `../../packages/core/src/index.ts` and `../../packages/cli/src/index.ts`.

- `apps/public-web/test/components/landing-page.spec.tsx`
  Assert that the landing page renders presentation content plus primary links to docs and GitHub, covering the homepage behaviour from `public-web:public-site`.

- `apps/public-web/test/lib/api-generation.spec.ts`
  Assert that the API generation config writes into `.generated/api` and keeps the generated docs mounted as a site-integrated section rather than a disconnected output.

### Manual / E2E verification

- Run `pnpm --filter @specd/public-web build`
  Expected: the app generates API docs first, then produces a static Docusaurus build successfully.

- Run `pnpm --filter @specd/public-web dev`
  Expected: the site serves locally with:
  - `/` showing a presentation homepage
  - `/docs` exposing public docs
  - no ADR entries in public navigation
  - `/api` serving generated API reference

- Run `pnpm lint`
  Expected: the new app participates in linting, and any `default export` usage is limited to the framework-required entrypoints defined by the design.

- Run `pnpm test`
  Expected: the new `public-web` tests execute alongside existing workspace tests after the root test script is updated.

### Documentation notes

If any Docusaurus-specific metadata changes are needed for public docs rendering, those edits must be applied to files under the repository `docs/` tree. Do not introduce a second handwritten docs corpus under `apps/public-web`.
