# Exploration Context

## Problem Statement

The user wants specd to distinguish artifact drift explicitly instead of collapsing it into `in-progress`.

Current pain points discussed:

- When an artifact file drifts after validation or approval, it currently appears as `in-progress`, which hides the reason.
- When a change is invalidated because of drift, it is not clear which artifact drifted.
- For `scope: spec` artifacts, it is also not clear which spec drifted.

The requested direction is to add a `drifted` artifact state and improve invalidation reporting so the affected artifact and spec are visible.

## User Intent

- Add a new artifact state named `drifted`.
- Make invalidation caused by artifact drift explicit, not ambiguous.
- Preserve enough detail to identify:
  - which artifact drifted
  - which spec drifted when the artifact scope is `spec`

## Key Codebase Observations

### Existing drift awareness already exists in domain and repository

- Graph search for `drift` found `Change.invalidate` in `packages/core/src/domain/entities/change.ts`.
- The method already accepts optional `driftedArtifactIds`.
- The method comment states that when `driftedArtifactIds` is provided, only those artifacts and downstream dependents have validation reset.

Concrete anchor:

- `packages/core/src/domain/entities/change.ts` — `invalidate` method around line 410

### Drift currently resets validation but surfaces as `in-progress`

From the loaded `core:core/change` spec:

- Artifact file status is currently derived as one of:
  - `missing`
  - `in-progress`
  - `complete`
  - `skipped`
- A hash mismatch currently maps to `in-progress`.
- Verification scenarios explicitly say:
  - changed artifact content after validation becomes `in-progress`
  - invalidation without selective drift reset clears hashes and turns `complete` into `in-progress`

This matches the user's complaint: the visible state does not distinguish "work started" from "validated before, then drifted".

### ValidateArtifacts already computes drifted artifact IDs

From the loaded `core:core/validate-artifacts` spec:

- Approval invalidation on content change already collects drifted artifact type IDs.
- It calls `change.invalidate('artifact-change', actor, driftedArtifactIds)`.
- It supports a single invalidation call even when multiple artifacts drift.

This suggests the change may not be about inventing drift tracking from scratch, but about exposing it consistently in status/result surfaces and possibly preserving more per-file/per-spec detail.

### Repository auto-invalidation on get already detects drift

From the search result content for `core:core/change-repository-port`:

- `FsChangeRepository.get()` already detects artifact drift.
- It may auto-invalidate when a previously validated artifact becomes `missing` or `in-progress`.
- It already collects drifted artifact type IDs and calls `change.invalidate('artifact-change', SYSTEM_ACTOR, driftedArtifactIds)`.

This reinforces that the backend can already identify drifted artifact types during eager invalidation.

### CLI surfaces are currently too coarse

From the loaded CLI specs:

- `cli:cli/change-status` shows only artifact `effectiveStatus`.
- `cli:cli/change-artifacts` lists per-file entries, but still only exposes `effectiveStatus`.
- Neither spec currently requires explicit reporting of drift cause, drifted artifact IDs, or drifted spec IDs.

This likely explains why the user sees ambiguity in the current CLI UX.

## Impact / Risk Assessment

Graph impact on `Change.invalidate` returned:

- `riskLevel: CRITICAL`
- many direct dependents across domain, use cases, composition, and status/reporting flows

Implication:

- introducing a new state like `drifted` is not a local serializer tweak
- the design must carefully define:
  - whether `drifted` is a file-level status, artifact-level effective status, or both
  - how dependency cascading interacts with `drifted`
  - whether `drifted` blocks transitions the same way as `in-progress`
  - how old manifests and backward compatibility are handled

## Design Questions To Resolve In /specd-design

- Is `drifted` added to the core `ArtifactStatus` union, or exposed as a secondary diagnostic field while keeping lifecycle gating based on existing statuses?
- For `scope: spec`, should drift be tracked per file key (`specId`) rather than only per artifact type ID?
- Should `GetStatusResult` include explicit drift metadata in addition to `effectiveStatus`?
- Should `change status` text output show something like `drifted` directly, or include a separate diagnostic section?
- Should `change artifacts` JSON/text include the spec/file key that drifted even when the artifact type is shared across multiple specs?
