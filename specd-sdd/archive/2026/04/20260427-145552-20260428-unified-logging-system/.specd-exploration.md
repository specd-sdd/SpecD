# Exploration: Unified Logging System

## Problem Statement

The current codebase lacks a unified logging strategy, relying on scattered `console.log/warn/error` calls. This makes observability difficult, lacks structured output (JSON) for processing, and doesn't provide a clean way to toggle debug levels or capture domain-specific metadata without manual boilerplate.

## Approach / Solution Outline

- **Library:** Use **Pino** as the base logging engine due to its high performance and native support for structured (JSON) logging.
- **Pattern:** **Ambient Context (Global Proxy)**.
  - Define a `LoggerPort` interface in `@specd/core`.
  - Provide a static `Logger` proxy in `@specd/core` that redirects to a configurable implementation.
  - The `@specd/cli` (Infrastructure layer) initializes the real Pino instance and "plugs" it into the core's proxy during bootstrapping.
- **Architectural Rationale:**
  - Avoids "Logging Pollution": No need to inject the logger into every entity, service, or use case constructor.
  - Maintains Hexagonal Integrity: `core` only depends on its own interface (`LoggerPort`). The implementation is external.
  - Pragmatic DX: Developers can use `Logger.debug()` anywhere in the core for tracing without refactoring signatures.

## Affected Areas

- `packages/core`: Define the Port and the Proxy.
- `packages/cli`: Implement the Adapter, configure Pino, and handle bootstrapping.
- Future: All other packages will migrate from `console` to `Logger`.

## Spec IDs

- `core:core/logger-port` (Proposed)
- `cli:cli/logging-integration` (Proposed)

## Design Decisions

- **No official PSR equivalent:** Acknowledged that Node.js lacks a PSR-3 equivalent; our `LoggerPort` will serve as our internal standard.
- **Proxy over Pure Injection:** Chosen for maintainability and to match industry standards (Java/SLF4J, Go/Zap, NestJS).
- **Default behavior:** The proxy will default to a "No-op" (silent) or basic `console` fallback to keep unit tests clean without extra setup.

## Agreements Reached

- The `core` package MUST NOT know about Pino.
- The `core` package defines the log levels and metadata structure.
- Entities and Value Objects in the domain are allowed to use the static `Logger` for critical lifecycle events (e.g., invalidations).
- The CLI is responsible for the final output format (e.g., using `pino-pretty` for interactive terminals).

## Open Questions

- Should we support "Child Loggers" (loggers with pre-bound context)? (Likely yes, via a `.child()` method in the port).
- How to handle log level configuration via `specd.yaml`? (To be decided during design).
