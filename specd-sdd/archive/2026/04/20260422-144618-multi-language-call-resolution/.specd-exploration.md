## GitHub Issues

### Issue 52

- URL: https://github.com/specd-sdd/SpecD/issues/52
- Title: `code-graph: complete dependency detection in TypeScript, Python, and Go`
- State at discovery time: `OPEN`
- Key points:
  - TypeScript handles static `import ... from ...` reasonably but lacks dynamic `import()`, CommonJS `require()`, side-effect imports, better member-call resolution, and `tsconfig` path alias support.
  - Python covers `import ...` and `from ... import ...`, but lacks dynamic imports via `importlib.import_module()` and `__import__()`, has limited package/submodule resolution, and weak heuristic `CALLS` resolution.
  - Go parses imports and symbols but currently does not build useful `IMPORTS` or `CALLS` relations; selector expressions (`pkg.Func()`, `obj.Method()`), alias imports, dot imports, blank imports, and module/package resolution need work.
  - Cross-cutting gap: inheritance and type hierarchy affects call analysis, navigation, and blast-radius accuracy.
  - Suggested direction includes completing Go `IMPORTS`/`CALLS`, extending TypeScript and Python import detection, designing inheritance/type relation support, and defining what counts as a detectable dependency.

### Issue 54

- URL: https://github.com/specd-sdd/SpecD/issues/54
- Title: `Add a shared scoped binding environment for multi-language call resolution`
- State at discovery time: `OPEN`
- Key points:
  - Introduce a shared, multi-language Scoped Binding Environment for deterministic call-resolution support.
  - Separate raw syntax extraction by adapters, shared scope/binding lookup in common code-graph infrastructure, and language-specific binding rules owned by adapters.
  - Model deterministic bindings visible at a source location: lexical ownership, receiver identity, local bindings, class-managed bindings, imported/referenced type names, and framework-managed deterministic bindings.
  - Keep first version conservative and deterministic.
  - Initial binding sources suggested by the issue: typed parameters, typed properties/fields, explicit construction (`x = new X()`), `this`/`self`/`parent`/`super` equivalents, existing framework-managed bindings, and simple local alias assignments when the source binding is already known.
  - Explicitly avoid fuzzy/global inference, runtime container IDs, reflection-based resolution, interprocedural propagation, and whole-program data flow in the first iteration.
  - Questions to settle in design include where the environment is built, minimal cross-language binding fact model, first-class scope levels, unresolved binding diagnostics, and framework-managed binding integration.

## Design Direction Discussed

The likely direction is to combine issue 52 and issue 54 into a single design because dependency and call-detection gaps are closely related to shared binding and receiver resolution.

The design should likely define:

- A shared binding fact model contributed by language adapters.
- A scoped binding environment builder in common code-graph infrastructure or domain/application code, depending on whether it remains pure and side-effect-free.
- Shared call-form classification for free calls, member calls, static calls, and constructor calls.
- Adapter-owned extraction of syntax facts and language-specific semantics.
- Conservative deterministic resolution only for the first iteration.
- Regression tests across at least PHP and one non-PHP language, with user-requested scope implying coverage for all current built-in adapters where behavior is relevant.
