# Exploration: serialize-change-mutations

Generated: 2026-04-02

## Problem statement

The current `ChangeRepository` mutation pattern allows concurrent lost updates when multiple CLI processes or skills operate on the same change at the same time.

Today, use cases commonly do:

1. `ChangeRepository.get(name)`
2. mutate the in-memory `Change`
3. `ChangeRepository.save(change)`

This is unsafe under concurrency because two processes can load the same snapshot, mutate independently, and the last `save()` wins.

The motivating symptom discussed with the user is artifact validation in parallel. When several CLI processes validate artifacts on the same change concurrently, only the last manifest write survives because all processes loaded the manifest before any lock or coordination existed.

## Approach / solution outline

The agreed direction is to introduce a serialized mutation API on `ChangeRepository` instead of trying to solve this at the CLI layer.

Preferred shape discussed:

- Add a repository operation such as `mutate(name, fn)` rather than exposing locking mechanics directly.
- The repository implementation is responsible for:
  - acquiring a per-change lock
  - reloading the freshest persisted change state after the lock is acquired
  - executing the mutation callback against that fresh state
  - persisting the updated change
  - releasing the lock

The goal is for use cases to express "mutate this persisted change safely" without knowing anything about `manifest.json` or lock files.

## Why this layer

The user and agent aligned on these design points:

- The CLI should remain a thin adapter and should not know about locking.
- A lock only around manifest file writing is insufficient because the stale snapshot problem happens before the write.
- The coordination belongs in `ChangeRepository` because the shared resource is the persisted `Change` state, not the CLI process.
- The use case knows it is mutating a persisted `Change`, but it does not know or care that persistence is implemented with a manifest file.

## Scope agreed

This change is intentionally broader than `ValidateArtifacts`.

The user explicitly clarified that the new mutation path should be used in all use cases that modify an existing `Change`, not just in `ValidateArtifacts`.

The core rule captured from the conversation:

- Any use case that currently does `get() -> mutate Change -> save()` for an existing change should move to the new serialized mutation API.

This excludes pure reads and likely excludes `CreateChange`, which creates a new change rather than mutating an existing persisted one.

## API shape discussed

Two names were compared:

- `mutate(name, fn)`
- `withLock(name, fn)`

The preferred option from the discussion was `mutate(name, fn)` because it expresses repository/domain intent rather than exposing the infrastructure mechanism.

Example shape discussed:

```ts
const result = await this._changes.mutate(input.name, async (change) => {
  // validate invariants
  // mutate the Change entity
  return result
})
```

Repository sketch discussed:

```ts
async mutate<T>(name: string, fn: (change: Change) => Promise<T>): Promise<T> {
  return await this._withChangeLock(name, async () => {
    const change = await this.get(name)
    if (change === null) throw new ChangeNotFoundError(name)
    const result = await fn(change)
    await this.save(change)
    return result
  })
}
```

Important semantic difference captured during the discussion:

- `get()` returns a snapshot only.
- `mutate()` provides a serialized read-modify-write section for one change.

## Rejected alternatives

The following alternatives were discussed and rejected:

### Lock in the CLI

Rejected because:

- wrong architectural layer
- the CLI is just one adapter
- other entry points could still race
- it leaks persistence/concurrency concerns into delivery code

### Lock only the manifest write

Rejected because:

- it does not solve stale snapshots loaded before the write
- atomic rename prevents partial file corruption, but not lost updates

### Treat the problem as only `ValidateArtifacts`

Rejected because:

- the pattern is systemic across use cases that mutate an existing change
- fixing only one use case would leave other races in place

## User preferences and agreements

- The user wanted a concrete proposal first, then discussion.
- The user agreed with the direction of putting coordination around repository mutation rather than solving it in the CLI.
- The user explicitly wants the fix applied consistently across all use cases that modify a change, not as a narrow patch in `ValidateArtifacts`.

## Open questions for design

- Exact `ChangeRepository` port signature for serialized mutation.
- Whether `mutate()` should be the only new API or whether the repository also needs lower-level locking helpers internally.
- Which use cases should migrate immediately in the first pass and which, if any, need bespoke handling.
- Whether any additional verification scenarios are needed around lock release, errors inside the callback, and sequential visibility between concurrent writers.
- Whether the implementation should use a filesystem lockfile, a lock directory, or another Ladybug-safe mechanism for per-change exclusion.
- Whether defensive optimistic checks should also be added around manifest revision/versioning, or whether serialized mutation alone is sufficient for the first iteration.

## Tangential observations

- While exploring, an initial `graph stats --format json` attempt failed with a Ladybug lock error on `.specd/config/graph/code-graph.lbug`.
- The failure was caused by running two graph commands in parallel (`graph stats` and `graph search`) against the same Ladybug-backed graph store.
- Retrying both commands sequentially succeeded. This is important operational guidance for future work: graph queries should be treated as sequential, not parallel, even when they are read-only from the CLI's perspective.
- This is not the same storage path as the change manifest problem, but it is relevant background: the repository already has another visible concurrency surface in the toolchain.
