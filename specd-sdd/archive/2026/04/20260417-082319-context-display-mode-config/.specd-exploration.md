# Exploration Context

## Problem statement

The user wants to change compiled spec-context presentation so it is configurable instead of being hardwired to the current tiered behavior.

Current mental model described by the user:

- today the compiled context has tiered behavior (`tier 1`, `tier 2`, etc.) depending on how context is loaded
- the user wants to decide via config how spec context is shown
- desired display modes are:
  - `list`: only a list of specs to consult
  - `summary`: list plus spec summary/description
  - `full`: full spec content
  - `hybrid`: current mixed/tiered behavior

The user also wants a new CLI flag:

- `--include-change-specs`
- default: `false`
- effect: include or omit the change's own `specIds` in compiled context independently of `dependsOn`
- this must still respect the new config mode

The user explicitly wants existing section filters to continue working:

- `--rules`
- `--constraints`
- `--verify` was mentioned by the user as part of "respect what the user asks for"; likely maps to scenario/verification-style filtering and should be preserved semantically in design

The user also wants text CLI output to make it obvious when entries are not full content:

- indicate whether entries are lists or summaries
- instruct users to call `specd change spec-preview <change-name> <specId>` to obtain the full version of specs they need

The default config value for this context should become `summary`.

The user believes there is already an option for context mode and wants it reused instead of inventing a parallel setting.

## Approach / solution outline

- Reuse the existing project-level context-mode configuration rather than adding a second unrelated config field if possible.
- Evolve the current context-mode model from today's `lazy`-style behavior into explicit display modes:
  - `list`
  - `summary`
  - `full`
  - `hybrid`
- Preserve current dependency traversal behavior for `dependsOn` unless the new `--include-change-specs` flag changes the explicit inclusion of change-scoped `specIds`.
- Preserve current section filtering semantics for whatever specs are rendered in full mode.
- Update text-mode CLI rendering so non-full entries are clearly labeled and point users at `spec-preview`.
