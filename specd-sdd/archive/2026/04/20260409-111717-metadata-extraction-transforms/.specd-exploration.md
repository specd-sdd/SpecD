# Exploration Context

## Problem Statement

The user wants to make schema-declared `metadataExtraction.extractor.transform`
actually work during metadata extraction from artifact content, instead of
depending on special-case application logic.

The immediate trigger was that `change validate` fails when validating deltas
whose extracted `dependsOn` values are still filesystem-style paths, and the
same limitation affects other schemas that define other transforms in other
metadata fields.

The user explicitly clarified that this is about extraction from artifacts, not
about reading already-written `metadata.json`.

## User Intent

- Make generic metadata transforms possible in production runtime.
- Avoid hardcoded special handling for `dependsOn`.
- Move the behavior to the generic transform mechanism because "for that they
  exist".
- A new change was preferred over reusing an existing active change.

## Key Findings

### Generic transform support exists only in the extraction engine

`extractContent(...)` supports an optional transform map and applies
`extractor.transform` only if a transform callback map is provided by the
caller.

Concrete anchor:

- `packages/core/src/domain/services/content-extraction.ts`

### Production callers are not wiring transforms consistently

The extraction engine is called in production without a runtime transform map in
the paths inspected during exploration.

Concrete anchors:

- `packages/core/src/application/use-cases/validate-artifacts.ts`
- `packages/core/src/application/use-cases/generate-spec-metadata.ts`
- `packages/core/src/application/use-cases/compile-context.ts`
- `packages/core/src/application/use-cases/get-project-context.ts`
- `packages/core/src/application/use-cases/_shared/depends-on-traversal.ts`

### `dependsOn` currently has special-case behavior outside generic transforms

Observed during investigation:

- `GenerateSpecMetadata` extracts values, then resolves `dependsOn` manually via
  `SpecRepository.resolveFromPath(...)`.
- `ValidateArtifacts` has a dedicated `_extractDependsOn(...)` path.
- This means `dependsOn` can behave differently from other transform-driven
  fields and confirms special-case logic exists outside the generic transform
  system.

Concrete anchors:

- `packages/core/src/application/use-cases/generate-spec-metadata.ts`
- `packages/core/src/application/use-cases/validate-artifacts.ts`

### `change validate` validates extracted metadata before special-case `dependsOn` repair

The user reported that `change validate` fails when delta-derived `dependsOn`
values are FS-style paths. That matches the explored control flow:

- extraction from artifact content happens without runtime transforms
- extracted metadata is validated
- `dependsOn` resolution logic happens later and only in a dedicated code path

This ordering means generic transform-dependent extraction is not reliably
available at validation time.

Concrete anchors:

- `packages/core/src/application/use-cases/validate-artifacts.ts`
- `packages/core/src/domain/services/parse-metadata.ts`

### `schema-std` already declares `transform: resolveSpecPath`

The standard schema is already expressing intent through the schema mechanism.
The runtime is the missing piece.

Concrete anchor:

- `packages/schema-std/schema.yaml`

## Design Direction Agreed In Conversation

The agreed direction is:

- generic runtime transform support should be made possible
- hardcoded `dependsOn` logic should not remain the long-term mechanism
- extraction from artifacts should use schema-declared transforms rather than
  special-case branches

Important nuance raised during discussion:

- the current transform contract only accepts `string[]`
- fully replacing `dependsOn` special handling likely requires passing richer
  runtime context to transforms, not only raw strings

This was surfaced as an architectural gap, not as a reason to keep the
hardcoded design.

## Rejected / Criticized Approach

Rejected in principle:

- keeping `dependsOn` as a hardcoded special case while generic transforms stay
  non-functional in production

Reason:

- it breaks the schema abstraction and makes `transform` declarations misleading
- it does not solve other schemas and other transformed metadata fields

## Open Questions For Design

- What should the runtime transform API look like?
- Should transforms receive only `values`, or a richer context object including:
  - current spec ID / spec path
  - artifact ID
  - workspace
  - accessors for `SpecRepository` / repository resolution
  - maybe a stable application-layer transform context abstraction
- Where should transform resolution / registration live?
  - use-case-local assembly
  - shared helper / factory
  - composition layer service
- Should `dependsOn` resolution become a normal transform implementation,
  or should there still be a second pass for normalization/validation?
- Which use cases must be updated immediately to keep runtime behavior
  consistent?
  - `GenerateSpecMetadata`
  - `ValidateArtifacts`
  - `CompileContext`
  - `GetProjectContext`
  - shared depends-on traversal fallback path
- What tests/spec scenarios need to be added to prove transforms are really
  wired in production, not only unit-tested at engine level?
