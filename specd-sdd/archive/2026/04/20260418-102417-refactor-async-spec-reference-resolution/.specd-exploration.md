## Current behavior observed

### Direct bug in current transform logic

The current transform implementation is [packages/core/src/composition/extractor-transforms/resolve-spec-path.ts](/Users/monki/Documents/Proyectos/specd/packages/core/src/composition/extractor-transforms/resolve-spec-path.ts:62).

Key observation from the current algorithm:

- `originWorkspace` and `originSpecPath` are read from transform context.
- Relative paths are resolved by splitting `originSpecPath` and consuming `..` segments.
- Workspace escape is only marked when another `..` is encountered after `baseParts` is already empty.
- For `originSpecPath = core/actor-resolver-port` and `../../_global/architecture/spec.md`:
  - first `..` removes `actor-resolver-port`
  - second `..` removes `core`
  - `baseParts` becomes empty
  - `escapedWorkspace` is still `false`
  - the code falls through to the same-workspace return path and produces `core:_global/architecture`

This explains the symptom the user reported.

### Why prefixes alone do not fix the current code

The project config in [specd.yaml](/Users/monki/Documents/Proyectos/specd/specd.yaml:23) defines:

- workspace `default` with `prefix: _global` and `specsPath: specs/_global`
- workspace `core` with `prefix: core` and `specsPath: specs/core`

The current `resolveSpecPath` transform does not consult workspace configuration at all. It only receives:

- `originWorkspace`
- `originSpecPath`
- artifact metadata keys in the generic transform context map

There is a hardcoded special case for `_global -> default`, but no generic prefix-based resolution and no access to the real workspace map.

## Design decisions discussed

### Rejected: small patch in current transform only

An initial idea was to patch `resolveSpecPath` so that emptying `baseParts` on the second `..` already counts as leaving the origin workspace.

Why this was rejected as insufficient:

- it would fix the immediate bug but keep the transform dependent on weak logical origin context
- it would preserve the `_global -> default` hardcode
- it would still model resolution around current fs-style path assumptions
- it would not address the backend-agnostic concern raised by the user

### Rejected: filesystem-absolute resolution inside transform

Another idea was to pass real workspace config and absolute artifact paths into the transform, resolve the relative path to a physical path, and detect which workspace contains it.

Why this was rejected as the intended long-term design:

- it would effectively assume `SpecRepository` is filesystem-backed
- it would bake `specsPath` and local path semantics into transform behavior
- it would not generalize cleanly to non-fs adapters

### Rejected: synchronous wrapper that internally uses `await`

The user asked whether a synchronous function could just call something like:

- `const res = await repository.resolveFromPath(...)`

This was ruled out because:

- a synchronous function cannot use `await`
- making the function `async` changes its return type from `string` to `Promise<string>`
- that would break the current `ExtractorTransform` contract and force propagation through extraction callers

### Preferred direction: do it properly through repository-backed resolution

The conversation converged on the idea that the correct architectural direction is to make the resolution use repository-backed behavior rather than fs-only path math.

That strongly suggests:

- redesigning extractor transforms and extraction runtime to support async transforms
- or introducing another abstraction equivalent in capability

At the end of the discussion, the conclusion was that "doing it properly" likely means changing the extraction pipeline so repository-backed async resolution can participate directly.

## Agreements reached

- This should not be solved as a narrow one-off fix for `_global`.
- The solution should avoid hardcoded `_global -> default` logic.
- Prefix handling should come from workspace/repository semantics, not from ad hoc transform conditionals.
- If repository-backed resolution is the right abstraction, the team should accept the cost of changing the extraction pipeline rather than forcing a partial sync-only workaround.
- The likely design center is asynchronous extraction/transforms for this path-resolution problem.
