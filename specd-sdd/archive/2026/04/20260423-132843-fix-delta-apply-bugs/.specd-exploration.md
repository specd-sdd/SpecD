# Exploration Context: fix-delta-apply-bugs

**Generated:** 2026-04-23
**Updated:** 2026-04-23

## Problem Statement

When applying deltas with `position.last: true` to markdown content that contains multiple sections (e.g., multiple `### Requirement:` headers), the last section is lost. Two bugs were identified:

1. **Bug 1 — `last: true` not implemented:** In `applyDelta` (apply-delta.ts lines 603-623), the code treats `last: true` the same as "append" (no position hint).

2. **Bug 2 — Only first child added:** When the delta `content` has multiple top-level sections, only the first child is added (lines 580-587 take `parsed.root.children?.[0]` only).

## Additional Investigation Results

After deeper investigation, found 5 issues total:

### Issue 1: CONTENT MULTIPLE SECTIONS — CONFIRMADO como Bug Principal

- **Ubicación:** líneas 580-587
- **Cómo falla:** `added` solo toma `parsed.root.children?.[0]`, ignorando el resto
- **Comparación:** `modified` (línea 475) usa TODOS los children: `parsed.root.children ?? []`
- **Inconsistencia:** Diferente comportamiento para misma operación (parse content)

### Issue 2: POSITION.PARENT AMBIGUO — Confirmado

- **Ubicación:** líneas 567-574
- **Cómo falla:** No valida unicidad — silenciosamente toma `paths[0]` si hay múltiples matches
- **Comparación:** Para `modified`/`removed` hay validación explícita (líneas 350-353): `if (paths.length > 1) throw`
- **Es contrario al spec:** `delta-format` verify.md "apply rejects on ambiguous selector" establece el patrón para rejectar selectores ambiguos

### Issue 3: METADATOS PRESERVADOS — NO ES BUG

- **Verificado:** `setChildren()` (líneas 207-215) preserva todos los campos excepto `children` y `value`
- Tests confirman: `markdown-parser.spec.ts` líneas 253-270 pasa correctamente

### Issue 4: NODETYPE UNKNOWN — Potencial

- **Ubicación:** línea 591
- **Cómo falla:** Pasa `nodeType: 'unknown'` a `valueToNode` para operaciones `added` con `value`
- Algunos adaptadores podrían necesitar el tipo real para construir nodos correctamente

### Issue 5: FALLBACK SILENCIOSO — Intencional (documentado)

- **Ubicación:** líneas 606, 614
- **Cómo funciona:** Si `after`/`before` no encuentran el nodo, hace fallback a append
- **Verificado:** `delta-format` verify.md "position.after resolves to no node — fallback to end of parent"
- **Comportamiento esperado** — NO es bug

## Approach / Solution Outline

Fix both confirmed bugs in `packages/core/src/infrastructure/artifact-parser/_shared/apply-delta.ts`:

### Para Issue #1 (Content Multiple Sections):

Cambiar líneas 580-587 para usar TODOS los children, no solo el primero:

```typescript
const parsed = parseContent(entry.content)
const contentChildren = parsed.root.children ?? []
// Usar todos los children, no solo el primero
newNode = contentChildren.length > 0 ? { type: 'fragment', children: contentChildren } : parsed.root
```

### Para Issue #2 (position.parent ambiguo):

Añadir validación explícita después de línea 573:

```typescript
if (paths.length > 1) {
  throw new DeltaApplicationError(
    `position.parent selector is ambiguous — matched ${paths.length} nodes: ${JSON.stringify(entry.position.parent)}`,
  )
}
```
