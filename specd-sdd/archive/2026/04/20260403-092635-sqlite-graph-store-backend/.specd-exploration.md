# SQLite Graph Store Backend Exploration

## Problem statement

The user wants a new change to introduce a new `SQLite`-backed storage backend for the
`@specd/code-graph` package because `Ladybug`/`LadybugDB` has been causing operational
problems in the repository.

The motivation comes from prior debugging work around the current code-graph backend:

- `Ladybug` showed instability and poor behavior under indexing load
- there were memory issues and odd query behavior after indexing
- there were locking and concurrency problems while using graph commands
- the current backend is no longer considered a safe long-term default

The user explicitly wants this as a **new change**, separate from the currently active
`public-web-site` change.

## Approach / solution outline

The desired solution is:

- add a new `SQLiteGraphStore` backend behind the existing abstract `GraphStore` port
- keep backend selection as an **internal composition decision**, not a public config
  option in `specd.yaml`
- support internal backend selection through kernel options and the kernel builder
- likely use a registry/factory pattern, similar to other internal registries already
  present in the kernel/composition layer
- keep persistence rooted at the same logical graph path currently used by the
  Ladybug-backed graph store
- switch the internal default backend to `SQLite` **once it reaches parity**
- keep `Ladybug` as a coexisting backend for now rather than removing it immediately

This is not intended to be a user-facing backend toggle via project configuration.

## Core design decisions agreed

The user explicitly confirmed the following:

1. This should be a **new change**
2. The backend choice **should not be selected in `specd.yaml`**
3. Backend choice is an **internal decision**
4. The kernel should support passing the selected graph store:
   - through options
   - through the builder
5. A registry/factory model is a good fit for this selection
6. `SQLite` should become the **default** once it achieves parity
7. The `SQLite` database should be stored in the **same path where Ladybug stores the
   graph now** (same logical graph persistence root)

## Rejected alternatives

### Rejected: backend selection in `specd.yaml`

This was discussed and rejected.

Reason:

- the user does not want backend choice to become public configuration
- backend selection is considered an internal composition concern
- exposing it in config would prematurely solidify an infrastructure choice as public API

### Rejected: immediate permanent replacement with no coexistence

Not chosen at this stage.

Reason:

- the user wants `SQLite` to become the default **once it works / reaches parity**
- this implies an intermediate phase where both backends coexist

## Spec and graph observations

### `graph-store` spec

The current `graph-store` spec is already abstract and suitable for multiple backends.

Important parts:

- it explicitly does **not** prescribe a storage engine or physical schema
- it defines the minimum graph semantics a backend must support
- it includes `recreate()`
- it defines search behavior abstractly via `searchSymbols`, `searchSpecs`, and
  `rebuildFtsIndexes()`

This means the backend expansion should not require making `graph-store` concrete again.

### `ladybug-graph-store` spec

The current `ladybug-graph-store` spec already owns:

- backend-specific layout under `configPath`
- node tables / relationship tables
- FTS details
- schema versioning
- bulk loading details

This creates a clear precedent for introducing a sibling spec:

- `sqlite-graph-store`

### `composition` spec

The current composition spec still describes `LadybugGraphStore` as the default concrete
backend inside the factory wiring.

This will likely need to change so composition:

- introduces backend selection internally
- does not hardwire Ladybug as the only implementation
- can switch the default to SQLite once parity is achieved

### Kernel / builder context

The repo has clear kernel registry and builder concepts already:

- `core:core/kernel-builder`
- `core:src/composition/kernel-registries.ts`
- `core:src/composition/kernel-internals.ts`

This supports the user's suggestion that graph-store backend selection can fit the
existing additive registry pattern rather than introducing ad hoc branching logic.
