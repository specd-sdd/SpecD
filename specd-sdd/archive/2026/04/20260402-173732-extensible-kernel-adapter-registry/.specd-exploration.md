# Exploration Context

## Problem statement

Implement GitHub issue `#49`:

- URL: `https://github.com/lsmonki/SpecD/issues/49`
- Title: `feat(core): extensible adapter registry for kernel`

The issue states that `createKernel` in `@specd/core` hardcodes built-in adapter implementations and does not allow external registration of:

- storage backends
- VCS providers
- actor providers
- artifact parsers
- external hook runners

This blocks custom backends such as git/http/S3/database storage, custom VCS providers, custom actor identity sources, additional artifact formats, and hook execution outside the shell.

The requested direction is additive, not replacement-based:

- built-ins must remain available
- external registrations extend the kernel
- the merged registries should be exposed from the kernel for discovery and reuse

## Approach / solution outline

The issue proposes extending `KernelOptions` so callers can register additional adapter capabilities before kernel construction, and exposing the merged registry on the resulting `Kernel`.

The issue body explicitly suggests:

- `KernelOptions.storages`
- `KernelOptions.vcsProviders`
- `KernelOptions.actorProviders`
- `KernelOptions.parsers`
- `KernelOptions.hookRunners`

It also proposes:

- registry exposure via `kernel.registry`
- additive merging with built-ins
- immutable registration at build/create time only
- optional builder API as an ergonomic layer over `createKernel(config, options)`

The problem is centered on `@specd/core`, especially kernel composition and schema/config support for adapter names and external hooks.

## Key codebase observations

### Kernel spec constraints

From `core:core/kernel`:

- `createKernel(config, options?)` already exists and already has `KernelOptions`
- shared adapters must be constructed once
- project-level VCS and actor adapters already use autodetect factories rather than hardcoded git
- kernel is a plain immutable object, not a mutable class

This means the new design should preserve:

- manual DI
- one-time construction
- immutable post-construction state
- no runtime mutation after kernel creation

### VCS and actor factory behavior today

From `core:core/vcs-adapter` and `core:core/actor-resolver`:

- current autodetect order is `git -> hg -> svn`
- both factories support optional `cwd`
- both fall back to null implementations rather than throwing

The issue's provider-chain design should extend these detection factories rather than bypassing them.

### Hook model tension

From `core:core/hook-execution-model`:

- hooks currently come in exactly two kinds: `instruction:` and `run:`
- `instruction:` hooks are passive text
- `run:` hooks are executed by the `HookRunner` port

The issue proposes a third effective hook form, `external`, which likely requires a spec change in:

- `core:core/hook-execution-model`
- `core:core/schema-format`

This is likely one of the highest-friction parts of the design.

### Parser registry

From `core:core/artifact-parser-port`:

- `ArtifactParserRegistry` is already specified as `ReadonlyMap<string, ArtifactParser>`
- standard keys are currently `markdown`, `json`, `yaml`, and `plaintext`

This aligns naturally with the issue's additive parser registration proposal.

### Config implications

From `core:core/config`:

- config startup validation is strict
- project context and workspace semantics are already formalized

If adapter names become extensible, config validation and/or kernel construction must be explicit about when unknown adapter names are rejected and what registry source is authoritative.
