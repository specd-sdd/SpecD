# Exploration: Add --artifact flag to change spec-preview

Generated: 2026-04-28

## Problem Statement

Users want to be able to filter the output of `specd change spec-preview` to a specific artifact (e.g., just the `spec.md` or just the `verify.md`), similar to how `specd spec show --artifact <name>` works. This is useful for focused reviews of changes in large specs with multiple artifacts.

## Approach / Solution Outline

The implementation will follow the pattern established in `specd spec show`:

1.  Update the CLI command `change spec-preview` to accept an optional `--artifact <name>` flag.
2.  In the command action, if `--artifact` is provided:
    - Resolve the current active schema.
    - Resolve the artifact ID (e.g., `specs`, `verify`) to its target filename using the schema.
    - Validate that the artifact ID exists and its scope is `spec`.
    - Filter the results from the `PreviewSpec` use case to only include the file matching that filename.
3.  Ensure the header line `--- <filename> ---` is still printed in text mode when filtered, and the JSON/TOON output contains only the single entry.
4.  If the requested artifact is missing on disk or has no changes (and thus was skipped by the use case), provide an appropriate error/info message when filtered.

## Affected Areas

- **Spec:** `cli:cli/change-spec-preview` (needs to be updated with the new flag and requirements).
- **Code:** `packages/cli/src/commands/change/spec-preview.ts` (needs implementation of the flag and filtering logic).

## Design Decisions

- **Consistency:** The behavior (headers, error handling for unknown artifacts) must match `specd spec show` exactly.
- **Layering:** Filtering will be done in the CLI layer for now, keeping the `PreviewSpec` use case unchanged, consistent with how `spec show` handles it.
