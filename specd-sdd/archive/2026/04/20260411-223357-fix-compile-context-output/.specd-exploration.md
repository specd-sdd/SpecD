# Exploration Context

## Problem statement

The user reported a bug in compiled change context:

- `CompileContext` does not include the specs from `change.specIds` in the compiled context reliably.
- The CLI text output does not show the context fingerprint.
- There is no clear way to know whether the compiled context is `lazy` or `full`.
- The user clarified that the `lazy`/`full` visibility problem also affects JSON output, not only text output.

The goal of this change is to fix the completeness of compiled context and make the resulting mode and fingerprint visible in CLI output.

## Key codebase observations

### 1. `CompileContext` drops `specIds` unless they are collected again by another source

File:

- `packages/core/src/application/use-cases/compile-context.ts`

Observed behavior in the implementation:

- `change.specIds` are added to `sourceMap` early.
- `change.specDependsOn` targets are also added to `sourceMap`.
- The actual collected set starts from `includedSpecs`, which is populated only by:
  - project include patterns
  - workspace include patterns
  - optional `dependsOn` traversal
- There is no step that seeds `includedSpecs` with `change.specIds` or direct `change.specDependsOn` entries.

Practical consequence:

- A spec that is in `change.specIds` but not matched by an include pattern can disappear from `result.specs`.
- This contradicts the intended behavior of `CompileContext`, where tier-1 specs include `specIds`.

Relevant nearby logic:

- Tier classification later assumes `specIds` are present and classifies by `source`.
- Preview behavior also assumes specs from `specIds` are in the result set.

### 2. CLI text output omits `contextFingerprint`

File:

- `packages/cli/src/commands/change/context.ts`

Observed behavior:

- JSON/toon output includes `contextFingerprint` when `status === 'changed'`.
- Text output renders project context, full specs, summary catalogue, and available steps.
- Text output does not print the fingerprint anywhere.
- When `status === 'unchanged'`, text mode prints only `Context unchanged since last call.`

Practical consequence:

- A human or agent using text mode cannot see the current fingerprint value to reuse it on a later call.

### 3. CLI does not expose context mode clearly enough

Files:

- `packages/core/src/application/use-cases/compile-context.ts`
- `packages/cli/src/commands/change/context.ts`
- `packages/cli/src/commands/project/context.ts`

Observed behavior:

- Per-spec JSON entries include `mode: 'full' | 'summary'`.
- Text mode indirectly suggests laziness via the `## Available context specs` summary section.
- There is no explicit top-level field or clear summary that tells the caller whether the compiled context was produced in `lazy` or `full` mode.
- The user specifically noted that this ambiguity also exists in JSON output.

Likely implication:

- The result shape may need an explicit top-level context-mode field rather than forcing consumers to infer mode from per-spec entries or section presence.

## Investigation notes

### `core:core/compile-context`

The spec explicitly says:

- collected specs are classified after steps 1-5
- in `lazy` mode, tier 1 includes specs from `change.specIds` and `change.specDependsOn`
- the result includes a fingerprint

This reinforces that the current implementation is missing an inclusion step for `specIds`.

### `cli:cli/change-context`

The spec currently says:

- text mode prints project context, then full specs, then summary specs, then available steps
- JSON mode returns the structured result directly
- examples show `contextFingerprint` and per-spec `mode`

Current gap relative to the user report:

- fingerprint is not visible in text mode
- the top-level compiled mode is not obvious in JSON or text

## Proposed design direction for `/specd-design`

- Update the `core:core/compile-context` spec so the structured result explicitly carries the compiled context mode in a top-level field.
- Clarify that `change.specIds` are always included in the compiled context set before pattern-based additions.
- Update the `cli:cli/change-context` spec so text output includes both:
  - `contextFingerprint`
  - an explicit mode indicator (`lazy` or `full`)
- Update JSON output so callers can read the compiled mode directly without inference.
- Add tests in core and CLI covering:
  - `specIds` present even without matching include patterns
  - text output includes fingerprint
  - JSON output exposes compiled mode explicitly
  - text output exposes compiled mode explicitly

## Open questions for design

- Should the top-level result field be named `contextMode`, `compiledMode`, or something else?
- Should text output show the mode in a dedicated header line near the fingerprint, or inside a metadata block?
- Should `project context` output gain the same top-level explicit mode field for consistency, or should this change remain scoped to `change context` only?
- Should unchanged text output include the fingerprint and mode too, instead of only `Context unchanged since last call.`?
