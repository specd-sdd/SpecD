# Exploration: context-fingerprint

Generated: 2026-04-05

## Problem Statement

Every time an AI agent invokes a specd skill for an existing change, the skill has to re-read `change context`. This re-reads the same specs and project context every time, even when nothing has changed. Since the context output is deterministic based on:

- The change's specIds
- The project config (projectContext, contextIncludeSpecs, contextExcludeSpecs)
- The current step

...the output is identical unless specs or config changed.

## Solution

Add a `--fingerprint` flag to `change context` command. The flow:

1. **First call**: Agent calls `change context <name> <step>`. CLI returns full context + a `contextFingerprint` field (hash of all relevant inputs).

2. **Agent remembers fingerprint**: The agent stores the fingerprint in its conversation context.

3. **Subsequent calls**: Agent calls `change context <name> <step> --fingerprint <value>`. CLI:
   - Calculates current fingerprint
   - If `current === provided`: returns `{"status": "unchanged", "contextFingerprint": "sha256:..."}`
   - If `current !== provided` (or no fingerprint provided): returns full context + new fingerprint

## Design Decisions

1. **Fingerprint calculation happens in the CLI** — it has access to all inputs (specs, config, step) so it can accurately detect changes.

2. **Agent stores fingerprint in conversation window** — no files needed, no extra complexity. The agent remembers it during the session.

3. **Session-based persistence** — if user closes and reopens chat, agent starts fresh (no fingerprint) and gets full context again. This is acceptable since it's the first call anyway.

4. **Status field in response** — JSON mode returns `{"status": "unchanged" | "changed", "contextFingerprint": "..."}`. Text mode could show a brief "Context unchanged" message or just be empty/minimal.

## Technical Details

### Fingerprint composition

The fingerprint should hash:

- Change specIds (sorted for consistency)
- Project context entries (instructions + file contents hashes)
- Context include/exclude patterns from config
- The step being queried
- Schema version

### Implementation locations

1. **core:core/compile-context** — add fingerprint calculation logic
2. **cli:cli/change-context** — add `--fingerprint` flag, implement comparison logic

### API change

```bash
# First call
$ change context my-change designing --format json
{
  "contextFingerprint": "sha256:abc123...",
  "status": "changed",
  ...full context...
}

# Subsequent calls with matching fingerprint
$ change context my-change designing --fingerprint sha256:abc123... --format json
{
  "contextFingerprint": "sha256:abc123...",
  "status": "unchanged"
}

# When something changed
$ change context my-change designing --fingerprint sha256:abc123... --format json
{
  "contextFingerprint": "sha256:xyz789...",
  "status": "changed",
  ...full context (with new fingerprint)...
}
```

## Rejected Alternatives

1. **Context cache file in change directory** — requires agent to know if cache is stale. Would need fingerprint anyway, so it's circular.

2. **Fingerprint stored in manifest.json** — would persist across sessions but adds complexity. The session-based approach is simpler and sufficient.

3. **Agent comparing context output itself** — too much work for the agent, and it doesn't have access to spec file hashes to know if they changed.
