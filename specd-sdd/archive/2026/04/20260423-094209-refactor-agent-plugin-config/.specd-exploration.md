# Exploration: refactor-agent-plugin-config

## Problem statement

The `AgentPlugin` interface and related types have three issues:

1. **`AgentPlugin.install()` / `.uninstall()` receive `projectRoot: string`** — a single primitive that forces every consumer to reconstruct project context from scratch. The fully-resolved `SpecdConfig` is already available at call sites and contains everything plugins need (`projectRoot`, `configPath`, workspaces, storage paths, etc.).

2. **`InstallOptions` and `InstallResult` are generic names** — they live in `@specd/plugin-manager` domain types but are specific to agent plugins. The names suggest a general install mechanism that doesn't exist, confusing consumers.

3. **The skills package has no built-in variable injection** — when resolving skill bundles via `getBundle(name, variables)`, only user-provided variables are expanded. Useful built-in variables like `{{projectRoot}}`, `{{configPath}}` are not automatically available in skill templates, forcing every consumer to remember to pass them.

## Approach / solution outline

### 1. Replace `projectRoot` with `SpecdConfig` in `PluginContext`

`SpecdPlugin.init(context: PluginContext)` currently receives:

```typescript
interface PluginContext {
  projectRoot: string
  config: Record<string, unknown> // plugin-specific config
  typeContext: unknown
}
```

Change to receive `SpecdConfig` (or a wrapper that includes it). `SpecdConfig` already has `projectRoot`, `configPath`, workspaces, storage config, plugins config (including per-agent `config` section). The `config: Record<string, unknown>` field becomes redundant since that data lives in `SpecdConfig.plugins.agents[].config`.

### 2. Replace `projectRoot: string` with `SpecdConfig` in `AgentPlugin.install()` / `.uninstall()`

```typescript
// Before:
install(projectRoot: string, options?: InstallOptions): Promise<InstallResult>
// After:
install(config: SpecdConfig, options?: AgentInstallOptions): Promise<AgentInstallResult>
```

Each concrete plugin extracts `config.projectRoot` when needed.

### 3. Rename `InstallOptions` → `AgentInstallOptions`, `InstallResult` → `AgentInstallResult`

Mechanical rename across all consumers. The names should make it clear these are agent-plugin-specific types.

### 4. Inject built-in variables in skills package

When resolving bundles, the skills repository (or `ResolveBundle` use case) merges built-in variables derived from `SpecdConfig` with user-provided variables:

Built-in variables to inject:

- `{{projectRoot}}` — `config.projectRoot`
- `{{configPath}}` — `config.configPath`
- Possibly more (workspace paths? schema ref?) — to be decided during design

This requires passing `SpecdConfig` to `createSkillRepository()` or to the `ResolveBundle` use case.

## Design decisions

### Confirmed by user

1. **`SpecdConfig` replaces `projectRoot` everywhere in plugin interfaces** — `AgentPlugin.install()`, `AgentPlugin.uninstall()`, `PluginContext`
2. **`SpecdConfig` already contains `projectRoot`** — confirmed it's a field on the resolved config interface
3. **`PluginContext` loses its `projectRoot` and `config` fields** — replaced by `SpecdConfig` which contains both the project root and per-plugin config (under `plugins.agents[].config`)
4. **`InstallOptions` → `AgentInstallOptions`**, **`InstallResult` → `AgentInstallResult`** — rename to make them agent-specific
5. **Skills package injects built-in variables** — `{{projectRoot}}`, `{{configPath}}` at minimum, possibly more during design

### Still to decide during design

1. **Exact shape of new `PluginContext`** — does it just wrap `SpecdConfig`? Does it keep `typeContext`? Does the `config` field (per-plugin config) get derived from `SpecdConfig.plugins.agents[].config` automatically, or does the plugin extract it itself?
2. **Built-in variables list** — `projectRoot` and `configPath` confirmed. Others? workspace paths? schemaRef?
3. **Where built-in variables are injected** — in `createSkillRepository()` (factory receives `SpecdConfig`), in `ResolveBundle` use case, or in a new composition layer?
4. **`PluginLoader` currently takes `PluginLoaderOptions: { projectRoot }`** — should it take `SpecdConfig` too?

## Agreements reached

- All plugin types should receive the full resolved config, not individual primitives
- The `config: Record<string, unknown>` on `PluginContext` is redundant when `SpecdConfig` is available (the plugin can read its own config section from there)
- The rename is mechanical and not controversial
- Built-in variables should be merged with user-provided ones (built-ins as base, user vars can override)

## Rejected alternatives

- None discussed — the approach was clear from the start

## Key codebase observations

### Current flow: plugins install

1. CLI `installPluginsWithKernel()` has `config: SpecdConfig` from `resolveCliContext()`
2. It passes `config.projectRoot` to `createPluginLoader({ projectRoot })`
3. `InstallPlugin.execute()` receives `{ pluginName, projectRoot }` and calls `plugin.install(projectRoot, options)`
4. The plugin's `InstallSkills.execute(projectRoot, options)` uses `projectRoot` to build target dir like `.claude/skills/`

So `SpecdConfig` is already available at the CLI layer — it's just stripped down to `projectRoot` before being passed into the plugin system.

### Current flow: skills variable substitution

1. `FsSkillRepository.getBundle(name, variables)` calls `applyVariables(content, variables)` for each file
2. `applyVariables` is a simple `{{key}}` regex replacement — no built-in variables
3. `ResolveBundle.execute({ name, variables })` passes variables straight through to `repository.getBundle()`
4. Neither the use case nor the repository has access to `SpecdConfig`
