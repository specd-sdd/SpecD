# Exploration: improve-specd-diagnostics-and-ux

## Problem statement

The current SpecD CLI output and skill instructions create significant friction for agents (and humans) during the change lifecycle.

1. **Ambiguous "Warnings"**: Non-blocking validation messages are labeled as "warnings", which agents interpret as errors requiring immediate fixing before proceeding. This leads to redundant "cleanup" cycles.
2. **Generic Transition Errors**: When a state transition fails (e.g., from `designing` to `ready`), the error message is often generic (e.g., "artifact specs is not complete"), hiding the specific root cause like a single missing file in a change with many specs.
3. **Information Overload in JSON**: Skills currently mandate JSON output for status and validation, which is hard to diagnose during failures compared to human-optimized text summaries.

## Approach / solution outline

- **Semantic Renaming**: Update the validation engine to use "note" instead of "warning" for non-blocking AST/Delta suggestions.
- **Root Cause Diagnostics**: Modify `GetStatus` (core) and `change status` (cli) to explicitly list **Blockers** (missing files, stale metadata, etc.) in a dedicated, high-visibility section.
- **Detailed Transition Failures**: Enhance `TransitionChange` to return structured, descriptive reasons for gating failures, and update the CLI to render these as actionable instructions.
- **Skill UX Optimization**: Update the project's agent skills to use `--format text` for diagnostic confirmation and `--format json` only when data extraction is required.

## Design decisions

- **Decision** → Rename "warning" to "note" in validation outputs.
  - _Rationale_: "Warning" implies a block or a bug; "Note" implies a non-blocking suggestion.
- **Decision** → List specific missing files in transition errors.
  - _Rationale_: Avoids "JSON blindness" where one `missing` state is lost in hundreds of `validated` lines.
- **Decision** → Add a "Blockers" section to `change status` text output.
  - _Rationale_: Humans and agents need a "too long; didn't read" summary of why they are stuck.

## Agreements reached

- Transition errors should be instructions for repair, not just statements of fact.
- JSON is for data; Text is for diagnostics.
