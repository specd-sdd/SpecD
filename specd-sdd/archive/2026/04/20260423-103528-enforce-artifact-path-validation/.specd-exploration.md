# Exploration: enforce-artifact-path-validation

## Problem Statement

The user identified that newly created change manifests can initially persist
spec-scoped artifact file paths under `specs/...` even when the target spec already
exists and the schema artifact is delta-capable. In that situation, the correct
working file is a delta under `deltas/<workspace>/<capability-path>/<filename>.delta.yaml`.

This matters because some agents read `manifest.json` directly even though they
should use the specd workflow. If the manifest starts with `specs/...`, those
agents may create a full replacement artifact in the wrong place instead of the
expected delta file.

The user also noted a related validation issue: when validating a `scope: spec`
artifact for an existing spec, the validator should know the exact expected path
and validate that file only. It should not accept a normal `specs/...` artifact
when the correct artifact is the delta. If the expected delta is missing, validation
should fail and report that the expected delta path could not be validated.

## Desired Behavior

- A new change manifest should be correct from creation time.
- Existing specs with delta-capable spec-scoped artifacts should be tracked as
  `deltas/<workspace>/<capability-path>/<artifact-filename>.delta.yaml`.
- New specs should continue to be tracked under
  `specs/<workspace>/<capability-path>/<artifact-filename>`.
- `ValidateArtifacts` should validate the expected artifact path for the current
  spec/artifact combination, not whichever candidate file happens to exist.
- For an existing spec and delta-capable artifact, missing delta file should be a
  validation failure for the expected delta path.
- `specd change validate` text output should clearly state which file path was
  validated on success or which expected file path could not be validated on
  failure.
- Validation output should include a note telling the user to inspect the final
  merged result with `specd change spec-preview`.

## Current Codebase Observations

Observed example: `.specd/changes/20260422-151314-curate-core-public-api/manifest.json`
tracks existing spec `core:core/composition` with:

- `specs/core/core/composition/spec.md`
- `specs/core/core/composition/verify.md`

The spec `core:core/composition` exists, so the expected working files for
delta-capable artifacts are:

- `deltas/core/core/composition/spec.md.delta.yaml`
- `deltas/core/core/composition/verify.md.delta.yaml`

The same change directory already has a scaffolded delta directory:

- `.specd/changes/20260422-151314-curate-core-public-api/deltas/core/core/composition/`

This proves the directory scaffold and manifest filename can disagree at initial
creation time.

Observed contrasting example:

- `.specd/changes/20260422-144618-multi-language-call-resolution/manifest.json`

After delta files were created and validated, its manifest tracks completed files

This indicates the repository can later resolve delta paths, but the manifest is
not correct from the start.

## Relevant Implementation Findings

`Change.syncArtifacts()` currently creates spec-scoped file entries using
`specs/<workspace>/<capability-path>/<basename>` unconditionally.

Relevant file:

- `packages/core/src/domain/entities/change.ts`

Relevant behavior:

- For `artifactType.scope === 'spec'`, it parses each spec ID and builds a
  `specs/...` relative path.
- The method is intentionally pure and does not perform I/O, so it cannot know
  whether the spec already exists.

`CreateChange.execute()` saves the change before scaffolding directories.

Relevant file:

- `packages/core/src/application/use-cases/create-change.ts`

Relevant behavior:

- `this._changes.save(change)` persists the manifest.
- `this._changes.scaffold(change, specExists)` runs afterward.

`FsChangeRepository.scaffold()` does know whether the spec exists.

Relevant file:

- `packages/core/src/infrastructure/fs/change-repository.ts`

Relevant behavior:

- If `artifactType.delta && exists`, it creates `deltas/<workspace>/<capPath>/`.
- If not delta-capable or the spec does not exist, it creates `specs/<workspace>/<capPath>/`.
- It does not currently update the manifest filenames during scaffold.

`FsChangeRepository._manifestToChange()` can later detect an existing delta and
replace a missing `specs/...` filename with `deltas/.../*.delta.yaml` in memory
and sometimes persist it when sync changed. That is too late for agents or tools
that inspect the initial manifest before a delta exists.

`ValidateArtifacts` currently has fallback behavior that can validate the primary
artifact directly when no delta is found. This is still correct for new specs, but
it is wrong for existing specs with delta-capable artifacts because the expected
file is the delta.

Relevant file:

- `packages/core/src/application/use-cases/validate-artifacts.ts`

Current spec wording in `core:core/validate-artifacts` says a missing delta file
for a delta-capable artifact is not itself a validation error because the artifact
may be new. The new behavior needs to refine this distinction:

- Missing delta is allowed only when the target spec does not exist yet and the
  expected file is a new `specs/...` artifact.
- Missing delta is a validation failure when the target spec already exists and
  the expected file is a delta.

`cli:cli/change-validate` currently defines simple pass/fail text output but does
not require explicit validated/missing file paths or a `spec-preview` guidance
note.

## Design Guidance For Next Step

The design should likely introduce one shared path resolution concept in the
core layer so create/scaffold/validate/status output do not each decide expected
artifact paths independently.

Candidate responsibilities for that resolver:

- Inputs: `ArtifactType`, spec ID, whether the base spec exists.
- Output: expected relative path for the change artifact.
- For change-scoped artifacts: basename from `artifactType.output`.
- For spec-scoped artifacts and existing base specs with `delta: true`:
  `deltas/<workspace>/<capability-path>/<basename>.delta.yaml`.
- For spec-scoped artifacts and new specs:
  `specs/<workspace>/<capability-path>/<basename>`.
- For spec-scoped artifacts that are not delta-capable:
  `specs/<workspace>/<capability-path>/<basename>`.

Important architectural constraint:

- Domain entities should remain pure and should not import filesystem or repository
  concerns. If expected path resolution needs spec existence, it should happen in
  application/infrastructure code with a `specExists` result supplied explicitly.

Potential implementation areas:

- `packages/core/src/domain/entities/change.ts`
- `packages/core/src/infrastructure/fs/change-repository.ts`
- `packages/core/src/application/use-cases/create-change.ts`
- `packages/core/src/application/use-cases/edit-change.ts`
- `packages/core/src/application/use-cases/validate-artifacts.ts`
- `packages/cli/src/commands/change/validate.ts`

Testing should include:

- Creating a change for an existing spec stores `deltas/.../*.delta.yaml` in the
  initial manifest for delta-capable spec-scoped artifacts.
- Creating a change for a new spec stores `specs/.../<filename>` in the initial
  manifest.
- Validating an existing spec with the expected delta missing fails even if a
  wrong `specs/.../<filename>` file exists.
- Validating a new spec uses the `specs/.../<filename>` file and does not require
  a delta.
- CLI text output on success names the validated path.
- CLI text output on failure names the expected missing/unvalidated path.
- CLI text output includes guidance to run `specd change spec-preview`.

## Open Questions

- Whether path metadata should be returned from `ValidateArtifacts` as structured
  result fields (`validatedFiles`, `expectedFiles`, or similar) or embedded in
  failure/warning descriptions. Structured fields are cleaner for JSON/toon output
  and for CLI rendering.
- Whether existing manifests with stale initial `specs/...` paths should be
  repaired opportunistically on load even before the delta file exists, using spec
  existence and artifact type information.
- Whether `change status` should also surface expected paths, or whether fixing
  manifest persistence is enough for status consumers.
