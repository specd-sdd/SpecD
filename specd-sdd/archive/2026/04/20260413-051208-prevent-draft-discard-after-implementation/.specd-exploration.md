# Exploration Context

## Problem statement

The user wants to prevent a change from being moved to `drafts/` or `discarded/` once implementation has already started.

Reason given:

- Today a change can be drafted or discarded at any time.
- If implementation already happened, code may already have changed while the permanent specs still remain in the repository.
- That creates a code/spec mismatch: specs continue to describe work that has effectively been abandoned from the workflow point of view, while code may already reflect it.

The user explicitly framed this as a pragmatic safety rule until the system can detect whether a change has actually modified code files.

## Requested approach

Use the change history as a temporary proxy for "implementation has happened":

- Detect whether the change has ever been in lifecycle state `implementing`.
- If yes, attempts to draft or discard the change should fail by default.
- The failure should explain why: allowing the operation would risk leaving specs and code out of sync because implementation may already exist.
- The CLI should expose a `--force` flag for both draft and discard flows.
- That force flag should be passed through to the corresponding use case in core.

The user also suggested that the relevant specs should explicitly document that this history-based heuristic is temporary and is being used until the project can detect whether a change has actually modified files.

## Key design decisions already made

- This should be a new change, not part of `plugin-system-phase-1`.
- The solution should be pragmatic, not perfect.
- The temporary implementation signal is:
  - "has this change ever been in `implementing`?"
- The signal must come from the change history, not from filesystem diff detection.
- The guard must exist in core use cases, not only in CLI.
- CLI must expose an explicit bypass mechanism via `--force`.
- The failure reason must explicitly mention the code/spec mismatch risk.
- Specs should record that this is a temporary heuristic until true file-modification detection exists.

## Agreements reached

- A separate change should be created.
- The change should cover both draft and discard behavior.
- The new protection should apply once a change has ever reached `implementing`, not only when it is currently in `implementing`.
- Force is an intentional escape hatch, not the default path.
- The initial scope should include both core specs and CLI specs because the requested behavior spans both layers.

## Suggested design direction

2. Update `DraftChange` and `DiscardChange` specs to:
   - add a `force` input flag
   - fail by default when historical implementation has occurred
   - allow override when `force === true`
3. Update `Change Draft` and `Change Discard` CLI specs to:
   - add `--force`
   - document the new failure case

   - previously reached `implementing` => draft/discard rejected
   - previously reached `implementing` + `force` => operation allowed
   - error messaging explains the code/spec mismatch risk
5. Make the temporary nature of the heuristic explicit in spec prose.
