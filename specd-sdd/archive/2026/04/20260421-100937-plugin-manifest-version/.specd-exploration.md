# Exploration: plugin-manifest-version

## Problem statement

Plugin versions are hardcoded as string literals in each plugin class (e.g. `get version(): string { return '0.0.1' }`). This means:

1. Every version bump requires editing 4+ identical strings across 4 plugin packages
2. The `specd-plugin.json` manifest has no version field — it can't be used for version checks without loading the plugin
3. No automation exists to keep manifests in sync with `package.json` versions during the release pipeline

The user wants:

- Plugin versions saved in manifests (`specd-plugin.json`)
- `version()` getter in plugin classes derived from the manifest (not hardcoded)
- A mechanism to regenerate manifests during the versioning/release pipeline

## Approach / solution outline

1. **Add `version` to `specd-plugin.json`** — sourced from `package.json`
2. **Update manifest schema** in `PluginLoader` to include `version: z.string()`
3. **Pass version to plugin `create()` factory** — the loader already reads the manifest and resolves the entry point. Instead of just calling `create()`, call `create(version)` or pass a context object with the version from the manifest
4. **Plugin classes receive version via constructor** instead of hardcoding it
5. **Add a manifest sync script** that reads each plugin package's `package.json#version` and writes it into the corresponding `specd-plugin.json`
6. **Wire the sync script into build/release pipeline** — either as a `pnpm build` step, a `prepublishOnly` script, or a dedicated release pipeline step

### Key design decision: where does the version flow?

The chain is: `package.json#version` → (build/release script) → `specd-plugin.json#version` → (PluginLoader reads manifest) → (passes to `create()`) → (plugin stores and exposes via `get version()`)

This means the manifest is the runtime source of truth, and the script keeps it synced with `package.json` at build time.

## Design decisions

1. **Version flows from manifest → plugin, not from plugin → manifest.** The manifest is the single source of truth at runtime. `package.json` is the source of truth at build/publish time. The sync script bridges them.
2. **Each plugin reads its own manifest** — the `create()` factory in each plugin's `index.ts` reads `specd-plugin.json` and passes version to the constructor. The `PluginLoader` does NOT extract or pass version — it only validates the manifest schema. This keeps the loader simple and makes each plugin self-contained.
3. **Factory signature stays the same:** `create(): AgentPlugin` — the factory reads the manifest internally, no parameter needed.
4. **The sync script should live in `dev/scripts/`** — consistent with the existing `dev/scripts/hooks/` pattern.
5. **Pipeline integration:** The user said "build or release, or both" — this needs to be decided during design. Build time ensures the manifest is always correct in dist; release time ensures it's synced right before publish.

## Impact analysis

- **Risk level: HIGH** on `plugin-loader.ts` — it's a core infrastructure component used by all plugin use cases (Load, Install, Uninstall, Update, List)
- The `SpecdPlugin` interface itself doesn't change — `version: string` stays. What changes is how it's populated.
- The `PluginLoader` port interface (`load(pluginName: string): Promise<SpecdPlugin>`) doesn't need to change — the loader internally reads the manifest and passes version to the factory.

## Open questions

1. **Pipeline placement:** Should the sync script run at build time (`pnpm build`), at release time (prepublish/prepack), or both? Running at build ensures dist always has correct version; running at release ensures it's fresh before publish. Need to decide.

## Rejected alternatives

- **PluginLoader passes version to factory** — user rejected this during design. The loader's job is validation and instantiation, not providing runtime data to plugins. Each plugin reads its own manifest instead.
- **Read `package.json` at runtime from the plugin class** — would require I/O in the domain layer, violating hexagonal architecture. Also wouldn't work for published packages where `package.json` might not be accessible relative to the dist output.
- **Use `import.meta.resolve` to find package.json at runtime** — same layer violation, and brittle across different module resolution strategies.
