# Exploration: validate-metadata-extraction

Generated: 2026-04-05

## Problem Statement

During the verification of the context-fingerprint change, we discovered a gap in the validation chain:

1. `change validate --artifact verify` only checks:
   - Delta syntax (`deltaValidations`)
   - Structural rules on merged preview (`artifactType.validations`)

2. `spec generate-metadata` validates:
   - Metadata extraction rules (`metadataExtraction`) - checks that scenarios have WHEN/THEN/AND, that requirements exist, etc.

The problem is that `metadataExtraction` rules are NOT checked during `change validate`. This means:

- A malformed scenario (missing WHEN/THEN) can pass validation
- Only `spec generate-metadata` catches it later

## Solution

Modify `ValidateArtifacts` to also validate `metadataExtraction` rules from the schema:

1. For each artifact being validated, check if the schema has `metadataExtraction` rules for that artifact ID
2. If so, run those rules against the **merged preview** (same merged content used for structural validations)
3. Any failures should be reported as validation failures

This ensures the full chain of validation is complete at validation time, not later at metadata generation time.

## Technical Details

### Current flow in ValidateArtifacts (lines 304-355):

1. For delta files: Apply delta to base spec → `mergedAST`
2. Serialize to get `validationContent`
3. Run `artifactType.validations` on `validationContent`

### Proposed change:

1. After step 2 (or parallel with step 3), check `schema.metadataExtraction()`
2. If there are rules for the current `artifactType.id`, run them on `validationContent`
3. Report any failures as validation failures
