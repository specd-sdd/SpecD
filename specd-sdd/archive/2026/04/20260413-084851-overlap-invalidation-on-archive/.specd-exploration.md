# Exploration: overlap-invalidation-on-archive

Generated: 2026-04-13

## Problem statement

When archiving a change with `--allow-overlap`, the archive succeeds but other active changes that share overlapping specs receive no notification. Their specs have been modified by the delta merge, but they continue as if nothing happened. This creates silent conflicts — the overlapping changes may be working against stale spec content or produce deltas that conflict with the already-archived changes.

## Approach / solution outline

When `allowOverlap=true` is passed to `ArchiveChange`, instead of just skipping the overlap check, the use case should:

1. Detect which other active changes overlap with the change being archived (using the existing `detectSpecOverlap` domain service)
2. Invalidate each overlapping change back to `designing` with a new invalidation cause: `spec-overlap-conflict`
3. Include in the invalidation message: which change was archived and which specs overlap
4. Return information about invalidated changes so the CLI can report it

For the `ReviewSummary.reason` field in `GetStatus`, currently it only derives `'artifact-drift' | 'artifact-review-required' | null` from file states. The new `'spec-overlap-conflict'` cause needs to be surfaced there so that `change status` and agents/skills can detect it.

## Design decisions

1. **Invalidation cause: `spec-overlap-conflict`** — New cause added to `InvalidatedEvent.cause` union. This is distinct from `spec-change` (which is for when the change's own specIds are edited) and `artifact-drift`/`artifact-review-required` (which are for content drift). `spec-overlap-conflict` specifically means "another change was archived with overlapping specs."

2. **`ArchiveChange` invalidates, not `detectSpecOverlap`** — The domain service remains pure. The application use case (`ArchiveChange`) is responsible for calling `ChangeRepository.list()`, running `detectSpecOverlap`, and invalidating overlapping changes. This keeps the domain service side-effect-free.

3. **`ReviewSummary.reason` derived from latest invalidation event** — Currently the reason is derived from file states (drift vs review-required). The new cause requires checking the latest `invalidated` event in `change.history` and, if its `cause` is `spec-overlap-conflict`, using that as the reason. This gives agents a clear signal.

4. **`ReviewSummary` fields for overlap detail** — In addition to `reason`, the review section should include the archived change name and overlapping spec IDs from the invalidation event's `message` or a new structured field. This allows agents to understand exactly why they were invalidated without parsing free-text messages.

5. **Invalidation message format** — The `invalidated.message` should follow a clear pattern: `"Invalidated because change '<archivedName>' was archived with overlapping specs: <specId1>, <specId2>"`. Machine-parseable and human-readable.

6. **`ArchiveChangeResult` extended** — The result object from `ArchiveChange.execute` should include an `invalidatedChanges` array with `{ name: string, specIds: readonly string[] }` entries so the CLI can report them.

## Rejected alternatives

- **Auto-discard overlapping changes instead of invalidating** — Rejected because discarding is a terminal action and the overlapping change may still be viable after re-evaluation. Rolling back to `designing` lets the agent decide.
- **Warn but don't invalidate** — Rejected because the overlap means the other change's spec content is now stale; continuing without invalidation would produce conflicting deltas.
- **Separate `--invalidate-overlap` flag** — Initially considered but rejected: if the user is passing `--allow-overlap`, they're already accepting the consequences. Invalidating overlapping changes is the natural consequence of archiving despite overlap, not a separate decision.

## Open questions

- Should `ArchiveChange` need `ChangeRepository.mutate()` for each invalidated change (serialized mutation per change), or can it invalidate multiple changes in a batch? Current domain model uses `mutate()` per change — likely needs to loop over each overlapping change and call `mutate()` individually.
- Should the `ReviewSummary` include structured overlap detail (archived change name, spec IDs), or just the reason string? Leaning toward structured for agent consumption.
