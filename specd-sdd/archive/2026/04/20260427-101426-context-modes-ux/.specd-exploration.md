# Exploration Context: context-modes-ux

## Problem Statement

The user wants more control over the context compilation mode in CLI commands (`project context`, `change context`, `spec context`). Currently, these commands have section flags (`--rules`, `--constraints`, `--scenarios`) but lack a direct `--mode` flag. Additionally, the interaction between these flags and the display mode needs to be more intuitive and automated.

## Approach / Solution Outline

1. Add `--mode <mode>` flag to `specd project context`, `specd change context`, and `specd spec context`.
2. Supported modes should include those from config: `list`, `summary`, `full`, `hybrid`, plus a possible `auto` (CLI convenience).
3. Implement logic:
   - If any section flag (`--rules`, `--constraints`, `--scenarios`) is provided, the mode automatically switches to `full`.
   - In `full` mode (explicit or implicit), if no section flags are provided, it should default to `rules` + `constraints`.
   - If section flags ARE provided, they override the default sections.
4. Update Core Use Cases (`CompileContext`, `GetSpecContext`, `GetProjectContext`) to support these defaults for `full` mode to ensure consistency.

## Design Decisions

- `full` mode will no longer mean "raw markdown" by default if it's meant to default to `rules` + `constraints`. It will use structured rendering.
- If the user wants the "raw markdown" (everything), they might need to pass all flags or we might need a way to distinguish "raw full" from "structured full". _Clarification might be needed if they want to keep raw markdown access._
