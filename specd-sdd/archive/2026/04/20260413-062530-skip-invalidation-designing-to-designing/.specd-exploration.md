# Exploration: skip-invalidation-designing-to-designing

Generated: 2026-04-13

## Problem statement

When a change is already in `designing` state and transitions to `designing` again (e.g., via `specd change transition <name> designing`), the `TransitionChange` use case invalidates all artifacts and clears approvals. This is destructive and unnecessary ŌĆö the change hasn't progressed past designing, so there's nothing to "re-review."

The invalidation is intended for cases where a change **returns to designing from a later state** (e.g., `implementing ŌåÆ designing`, `ready ŌåÆ designing`, `pending-spec-approval ŌåÆ designing`). It already correctly skips invalidation when the origin is `drafting` (the initial transition to designing). But it does NOT skip invalidation when the origin is already `designing`.

## Approach / solution outline

Exclude `designing` as an origin state, the same way `drafting` is already excluded. The condition in `TransitionChange.execute()` (line 229) currently reads:

```ts
if (effectiveTarget === 'designing' && freshChange.state !== 'drafting') {
```

Change it to:

```ts
if (effectiveTarget === 'designing' && freshChange.state !== 'drafting' && freshChange.state !== 'designing') {
```

This ensures invalidation only happens when genuinely returning to designing from a later step.

## Design decisions

- **Add `designing` to the exclusion condition** ŌĆö confirmed by user as the right approach
- **No drift check needed** ŌĆö drift is already detected at the repository layer; invalidation on `designing ŌåÆ designing` forces all artifacts to `pending-review` unnecessarily, removing LLM control over whether re-review is needed
- **Minimal fix** ŌĆö only the condition in `TransitionChange.execute()` changes; no domain entity changes required

## Rejected alternatives

- **Check drift before invalidating** ŌĆö rejected because drift detection is a separate concern handled at the persistence layer (`change-repository.ts`). Mixing drift logic into the transition use case would couple two concerns that are currently separate.
- **Leave as-is** ŌĆö rejected because `designing ŌåÆ designing` invalidation is destructive. It clears approvals and marks all artifacts as `pending-review` without cause, removing the LLM's ability to decide whether re-review is needed.
