# Exploration: cli-init-alias

## Problem statement

The user wants `specd init` to work as a top-level alias for `specd project init`, so new users can bootstrap a project with a shorter, more intuitive command. This is a CLI-only change — no domain or application layer modifications needed.

## Approach / solution outline

Register `init` as a top-level command on the Commander root program (`program`), reusing the existing `registerProjectInit()` function. This function takes a `parent: Command` parameter and calls `parent.command('init')`, so calling `registerProjectInit(program)` in addition to `registerProjectInit(projectCmd)` will make `init` available at both `specd init` and `specd project init`.

No new handler code is needed — the same action handler, interactive wizard, and plugin installation flow are reused.

## Design decisions

- **Reuse `registerProjectInit(program)`** — simplest approach, no duplication, no new files
- **Both forms (`specd init` and `specd project init`) must behave identically** — same flags, same interactive wizard, same exit codes, same output
- **Help text should indicate the alias** — users running `specd --help` should see `init` listed as a top-level command

## Rejected alternatives

- **Commander `.alias()` mechanism** — only works for alternative names on the same command at the same hierarchy level, not for cross-hierarchy aliases (top-level vs nested)
- **Separate handler file** — unnecessary duplication when `registerProjectInit()` already accepts a parent parameter
- **Redirect via `program.parseAsync`** — would add overhead and complexity when a simple double-registration achieves the same result
