# Exploration: plugin-type-validation

## Problem statement

The plugin-manager's install/uninstall/update use cases hardcode a cast to `AgentPlugin` without verifying the plugin type. Currently all plugins are agents, so it works, but the design is not future-proof. When new plugin types arrive (integrations, themes, etc.) that DON'T have install/uninstall methods, the cast would silently break or crash at runtime.

The user wants proper type validation:

1. A central registry of known plugin types (`PLUGIN_TYPES` const array)
2. `isSpecdPlugin` should validate that `type` is one of the known types
3. Use cases should check `isAgentPlugin` before calling install/uninstall
4. Plugins of other valid types (without install/uninstall) should NOT crash — they should get a clear error saying the operation isn't supported

## Approach / solution outline

**Design principle:** Think forward — there will be plugin types that don't have install/uninstall. The validation chain should be:

1. `isSpecdPlugin` → base contract + known type check
2. `isAgentPlugin` → sub-check for agent-specific contract (install/uninstall)
3. Use cases check `isAgentPlugin` before calling install/uninstall, throw clear error if not

**Key change:** Derive `PluginType` from a const array so the same source serves both compile-time types and runtime validation.

## Design decisions

1. **`PLUGIN_TYPES` const array** — derive `PluginType` from it so adding a new type means adding one string to the array. Currently `['agent']`.
2. **`isSpecdPlugin` checks `type` is in `PLUGIN_TYPES`** — ensures unknown types are rejected at the base level.
3. **`isAgentPlugin` moves to domain** — currently a private function in `plugin-loader.ts` (infrastructure). It belongs in domain because it's a type guard on a domain type.
4. **Use cases throw `PluginValidationError`** when plugin doesn't support the requested operation (not an AgentPlugin but install was called).
5. **`validateRuntimePlugin` stays extensible** — uses the type guards from domain, so adding new types just means adding new guard checks.

## Agreements reached

- The manifest schema (`z.enum(['agent'])`) is correct as-is — it limits what's accepted. When new types are added, the schema is extended.
- `PluginType` already exists — we invert it to derive from `PLUGIN_TYPES` const array.
- `isSpecdPlugin` should check that the runtime `type` field is one of the known `PLUGIN_TYPES`.
- Future plugins without install/uninstall are valid `SpecdPlugin`s but not `AgentPlugin`s — trying to install them gives a clear error.
- The user explicitly rejected an approach where use cases assume all plugins are agents.

## Rejected alternatives

1. **Option A (initially proposed):** Add type guards per use case, keep `isSpecdPlugin` as-is — rejected because the user wanted the validation chain to be bottom-up (base first, then subtype).
2. **Validators map pattern:** A `Record<string, validator>` in `validateRuntimePlugin` — rejected in favor of simpler type guard functions in domain.
3. **First proposal (before reset):** User asked to forget the first solution proposal and start fresh with a future-proof approach.

## Open questions

- Should `isSpecdPlugin` and `isAgentPlugin` be pure functions in the domain types files, or should they be in a separate `domain/services/` file? (Following architecture spec: pure functions for stateless domain services)
- Error message for "plugin type X does not support install" — should it include the actual type or just say "not an agent plugin"?

### Current `validateRuntimePlugin` logic (`plugin-loader.ts:123-142`)

```typescript
private validateRuntimePlugin(
  pluginName: string,
  plugin: unknown,
  pluginType: z.infer<typeof manifestSchema>['pluginType'],
): asserts plugin is SpecdPlugin {
  if (!isSpecdPlugin(plugin)) {
    throw new PluginValidationError(pluginName, ['name', 'type', 'version', 'configSchema', 'init', 'destroy'])
  }
  if (pluginType === 'agent' && !isAgentPlugin(plugin)) {
    throw new PluginValidationError(pluginName, ['install', 'uninstall'])
  }
}
```
