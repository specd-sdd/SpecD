# Proposal: separate-metadata

## Motivation

`.specd-metadata.yaml` is derived state — generated by core or refined by the LLM — but it lives inside spec directories alongside user-authored content and is accessed through the generic `SpecRepository.artifact()` method as if it were a regular spec artifact. This mixes concerns, pollutes spec directories with system files, and makes it impossible to relocate metadata storage without touching every call site. In multi-repo setups, metadata for external specs must be generated locally because there is no dedicated resolution path.

## Current behaviour

Metadata is read via `repo.artifact(spec, '.specd-metadata.yaml')` and written via `repo.save(spec, new SpecArtifact('.specd-metadata.yaml', content))`. The file lives inside the spec directory:

```
specs/core/config/
├── spec.md                  ← user content
├── verify.md                ← user content
└── .specd-metadata.yaml     ← derived state (mixed in)
```

Every consumer — `SaveSpecMetadata`, `InvalidateSpecMetadata`, `CompileContext`, `GetSpecContext`, `GetProjectContext`, `ListSpecs`, `depends-on-traversal` — uses the same generic artifact methods with the magic string `.specd-metadata.yaml`. This means:

- Spec directories contain a mix of authored and generated files
- The magic string is scattered across the codebase
- Metadata cannot be moved to a separate location without updating every call site at the code level
- In multi-repo, external workspace metadata cannot be resolved efficiently — it would need regeneration

## Proposed solution

1. **Add dedicated methods to SpecRepository** — `metadata(spec): Promise<SpecMetadata | null>` and `saveMetadata(spec, content, options?)` as abstract methods on the port. All call sites switch from `artifact(spec, '.specd-metadata.yaml')` to `metadata(spec)`.

2. **Move metadata to `.specd/metadata/`** — The `FsSpecRepository` adapter stores metadata under `.specd/metadata/<path>/metadata.yaml` instead of alongside spec content. Spec directories become clean (authored content only).

3. **Multi-repo VCS root resolution** — For external workspaces, the fs adapter resolves the `.specd/` directory by detecting the VCS root of the specs path via `createVcsAdapter()`. This works across heterogeneous VCS setups (git, hg, svn). `NullVcsAdapter` fallback: `.specd/` in the specs root directory itself.

4. **Update all call sites** — Replace `repo.artifact(spec, '.specd-metadata.yaml')` with `repo.metadata(spec)` across all use cases. The magic string disappears from the application layer.

## Specs affected

### New specs

None.

### Modified specs

- `core:core/spec-repository-port`: Add `metadata()` and `saveMetadata()` abstract methods to the port contract
- `core:core/spec-metadata`: Change file location from spec directory to `.specd/metadata/`, update naming from `.specd-metadata.yaml` to `metadata.yaml`, update version control requirement
- `core:core/save-spec-metadata`: Replace `SpecRepository.artifact()` / `SpecRepository.save()` calls with `metadata()` / `saveMetadata()` for metadata access
- `core:core/invalidate-spec-metadata`: Replace direct `SpecRepository.save()` with `saveMetadata()` for metadata writes
- `core:core/compile-context`: Replace `.specd-metadata.yaml` references with `repo.metadata(spec)` calls
- `core:core/list-specs`: Replace `.specd-metadata.yaml` references with `repo.metadata(spec)` calls
- `core:core/get-spec-context`: Replace `.specd-metadata.yaml` references with `repo.metadata(spec)` calls
- `default:_global/spec-layout`: Remove `.specd-metadata.yaml` from spec directory constraint — metadata no longer lives in spec directories
- `cli:cli/spec-metadata`: Update references from `.specd-metadata.yaml` to new metadata location
- `cli:cli/spec-generate-metadata`: Update references from `.specd-metadata.yaml` to new metadata location
- `cli:cli/spec-write-metadata`: Update references from `.specd-metadata.yaml` to new metadata location
- `cli:cli/spec-invalidate-metadata`: Update references from `.specd-metadata.yaml` to new metadata location
- `core:core/config`: Add optional `specs.fs.metadataPath` field to workspace config
- `core:core/config-loader`: Resolve `specs.fs.metadataPath` relative to config directory, auto-derive from VCS root when absent

## Impact

| Layer                                   | Change                                                                                                    |
| --------------------------------------- | --------------------------------------------------------------------------------------------------------- |
| **Port** (`SpecRepository`)             | Add `metadata()` and `saveMetadata()` abstract methods                                                    |
| **Infrastructure** (`FsSpecRepository`) | Implement methods resolving to `.specd/metadata/<specPath>/metadata.yaml` via VCS root detection          |
| **Use cases**                           | `SaveSpecMetadata`, `InvalidateSpecMetadata` — switch to new methods                                      |
| **Use cases**                           | `CompileContext`, `GetSpecContext`, `ListSpecs`, `depends-on-traversal` — switch to `repo.metadata(spec)` |
| **Spec** (`spec-metadata`)              | Updated file location, naming, version control scope                                                      |
| **CLI**                                 | No command-level changes — CLI delegates to use cases                                                     |
| **Migration**                           | One-time move of existing `.specd-metadata.yaml` files to `.specd/metadata/` (scriptable)                 |

## Open questions

1. **Migration strategy** — Deferred to a separate change. Manual migration is acceptable for now given the small number of existing metadata files.
