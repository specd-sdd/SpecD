# Exploration: schema-show-full-output

Generated: 2026-03-29

## Problem statement

The `schema show` command currently displays only a **summary** of the schema — artifact IDs,
scopes, requires, output patterns, and workflow steps. It omits the majority of the schema's
actual content:

- Per artifact: `instruction`, `deltaInstruction`, `rules` (pre/post), `validations`,
  `deltaValidations`, `preHashCleanup`, `taskCompletionCheck` details, `template` content
- Per workflow step: `hooks` (pre/post — both `run` and `instruction` types),
  `requiresTaskCompletion`
- Schema-level: `metadataExtraction`

The current spec (`cli:cli/schema-show`) explicitly constrains this:

> Artifact `instruction`, `deltaInstruction`, `validations`, and `template` are not
> included in the output — this command focuses on structure, not content

The user considers this a bug — neither text nor JSON format shows the "real" schema.
The command should show the complete merged schema by default.

## Solution: Three new capabilities

### 1. Full output by default

Remove the constraint that limits output to structural metadata. Show **all** schema
fields in both text and JSON formats:

- **Artifacts:** include `instruction`, `deltaInstruction`, `rules` (pre/post entries),
  `validations`, `deltaValidations`, `preHashCleanup`, `taskCompletionCheck` (full config,
  not just boolean), `format`
- **Workflow steps:** include `hooks` (pre/post arrays with id, type, command/text),
  `requiresTaskCompletion`
- **Schema-level:** include `metadataExtraction` (title, description, dependsOn, keywords,
  context, rules, constraints, scenarios — each with extractor config)

Templates are excluded by default (see #2).

### 2. `--templates` flag

Templates can be large multi-line strings. Include them only when `--templates` is passed.
When active, each artifact's `template` field (string content loaded from schema's
template files) is included in the output.

### 3. `--raw` flag

Show the schema YAML **as-is**, without:

- Resolving the `extends` chain
- Applying `schemaPlugins`
- Applying `schemaOverrides` from `specd.yaml`
- Merging template content

This is useful for debugging schema inheritance or understanding what a specific schema
file declares on its own. The output should be the raw parsed YAML data, not a constructed
Schema entity.

`--raw` is mutually exclusive with `--templates` (raw doesn't resolve templates).

## Affected areas

### CLI package (`packages/cli/`)

- `packages/cli/src/commands/schema/show.ts` — main command handler. Currently maps
  Schema entity to a shallow summary. Needs to:
  - Expand text formatter to show all fields
  - Expand JSON output to include all fields
  - Add `--templates` and `--raw` options
  - Handle `--raw` by getting raw schema data instead of resolved Schema entity

### Core package (`packages/core/`)

- **Use case `GetActiveSchema`** (`packages/core/src/application/use-cases/get-active-schema.ts`)
  — currently returns a `Schema` entity. For `--raw` mode, needs a way to return the raw
  `SchemaYamlData` before `buildSchema()` is called. Options:
  - Add a separate method/mode to the use case
  - Add a new use case for raw schema access
  - Expose raw data through the existing use case with a flag

- **Schema domain model** (`packages/core/src/domain/value-objects/schema.ts`) — the
  `Schema` class already exposes all the data needed for full output via its public API.
  No changes needed for the full output mode.

- **ArtifactType** (`packages/core/src/domain/value-objects/artifact-type.ts`) — already
  has public properties for `instruction`, `rules`, `validations`, `deltaValidations`,
  `deltaInstruction`, `preHashCleanup`, `taskCompletionCheck`, `template`. Just need to
  serialize them in the CLI.

- **WorkflowStep** (`packages/core/src/domain/value-objects/workflow-step.ts`) — already
  has `hooks` (pre/post) and `requiresTaskCompletion`. Just need to serialize.

### Specs

- `cli:cli/schema-show` — needs spec delta: remove the constraint, add new flags, update
  output format documentation, add new scenarios
- `core:core/get-active-schema` — needs spec delta: possibly add raw mode support

## Key codebase observations

### Current show.ts structure (packages/cli/src/commands/schema/show.ts)

- Lines 54-76: text formatter — builds artifact lines and workflow lines manually
- Lines 78-101: JSON formatter — maps to shallow object
- The Schema entity is obtained via `kernel.specs.getActiveSchema.execute(input)`

### Schema entity public API (already available, just not serialized)

- `schema.artifacts()` returns `ArtifactType[]` with full properties
- `schema.workflow()` returns `WorkflowStep[]` with hooks
- `schema.metadataExtraction()` returns `MetadataExtraction | undefined`
- Each `ArtifactType` exposes: `instruction`, `rules`, `validations`, `deltaValidations`,
  `deltaInstruction`, `preHashCleanup`, `taskCompletionCheck`, `template`, `format`, `output`

### GetActiveSchema use case

- For project mode: delegates to `ResolveSchema` which applies extends + plugins + overrides
- For ref/file modes: resolves extends chain but no plugins/overrides
- Returns a fully constructed `Schema` entity
- For `--raw`, we'd need to intercept before `buildSchema()` is called

### Schema YAML structure (packages/schema-std/schema.yaml — 574 lines)

- Full schema with 5 artifacts, each with instructions, templates, validations, rules
- metadataExtraction section with multiple extractors
- Workflow with hooks

## Design decisions

- **Full output by default** — the user explicitly wants the command to show "the real
  schema", not a summary. No `--full` flag needed.
- **Templates separate** — templates are long and clutter output. `--templates` is opt-in.
- **`--raw` as mutually exclusive with `--templates`** — raw mode shows pre-merge data,
  templates aren't resolved yet at that stage.
- **`--raw` compatible with `[ref]` and `--file`** — you should be able to see the raw
  content of any schema source.

## Agreements reached

1. Show all schema fields by default (not just structural summary)
2. Add `--templates` for template content
3. Add `--raw` for unresolved/unmerged schema YAML
4. Both `cli:cli/schema-show` and `core:core/get-active-schema` specs need updates

## Open questions

- For `--raw` mode: should it return the parsed YAML data structure (JSON-like), or
  literally dump the YAML file content? Parsed data (as JSON/text) is more consistent
  with the command's output format options. Literal YAML dump would be simpler but
  bypasses formatting.
- Should `--raw` in project mode show the schema YAML before or after reading it from
  the registry? (i.e., does it show the file content, or the parsed-but-unmerged data?)
- The `--summary` flag idea was mentioned but the user didn't confirm wanting it. Current
  behavior could become `--summary` for backwards compatibility, but this wasn't discussed.
