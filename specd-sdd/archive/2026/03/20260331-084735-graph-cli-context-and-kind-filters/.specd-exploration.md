# Exploration Context

## Problem statement

The user wants a dedicated change to define, evaluate, and then later design the CLI semantics around `--config` and `--path`, especially for graph-related commands and bootstrap scenarios. The immediate goal is not implementation. The goal is to create a clean change that can evaluate the behavior properly without mixing it into unrelated work.

## Why this change exists

A previous change about PHP dynamic dependencies temporarily included CLI behavior changes around `--config` for graph commands without prior discussion. The user explicitly objected to that. Those CLI changes were reverted so they would not be smuggled into the archived PHP/code-graph change. This new change exists to handle the CLI question explicitly and on its own merits.

## User intent

The user wants to examine whether `--path` should continue to exist, how it should interact with `--config`, and what should happen when there is no `specd.yaml`.

The user specifically raised this concern:

- `--path` may still be useful for quick bootstrap flows.
- In repositories without `specd.yaml`, the CLI may need a fallback strategy.
- A reasonable fallback candidate is resolving from VCS root when config is absent.
- This should be evaluated deliberately, not as an incidental side effect of another change.
- Explicit `--config <path>` behavior also needs to be defined clearly, not just the fallback when the flag is absent.

The user later clarified the intended semantics:

- `--config` and `--path` are mutually exclusive.
- `--config` means “use this alternate config file path”, consistent with other CLI commands.
- `--path` means “use this repository root path”.
- If `--config` is provided, use that config.
- Else if a config is autodetected in the repo, use that config.
- Else if no config exists, use the repo root fallback.
- If `--path` is provided, use repo-root mode for bootstrap and ignore any config.
- For `--kind`, the desired input is simply a comma-separated list like `class,method,symbol` in a single option value.

## Current observed behavior before this change

Based on prior investigation in the conversation:

- `graph index` currently uses `resolveCliContext()`.
- If a `specd.yaml` is found, the command resolves config/workspaces from it.
- Without `specd.yaml`, `graph index` does not appear to have a well-defined bootstrap fallback; the expectation discussed was that it would fail rather than infer a workspace automatically.
- Earlier accidental CLI edits introduced `--config` into multiple graph commands. Those edits were reverted so they would not silently become product behavior.
- One verify correction in `specs/cli/graph-index/verify.md` was intentionally left in place: the `--force` scenario now describes deleting `.lbug`, `.lbug.wal`, and `.lbug.lock` rather than calling `clear()`. The user explicitly allowed leaving that as-is.
- `graph search` and `graph hotspots` currently expose `--kind <kind>` as a single-value option.
- Repeating `--kind` currently results in last-write-wins behavior, which is not the desired UX.
- There is already a CLI helper `parseCommaSeparatedValues()` in `packages/cli/src/helpers/parse-comma-values.ts` that can parse and validate comma-separated values.
- `@specd/code-graph` currently models `kind` in query/filter options as a single `SymbolKind`, not a list:
  - `SearchOptions.kind?: SymbolKind`
  - `HotspotOptions.kind?: SymbolKind`
- The store/query implementation also assumes a single kind predicate.
- The semantics of an explicit config path are still underdefined here: precedence over discovery, relative-path anchoring, and the error contract for missing/invalid config all still need design decisions.
- Existing CLI commands already treat `--config <path>` uniformly as “path to specd.yaml”; graph commands are the outlier for not exposing it.
- No existing mutual-exclusion pattern was found in `packages/cli/src` during this exploration, so design should choose either Commander-native conflicts or the project's existing style of explicit CLI validation.

## Agreements reached

- The config/path question should be handled in a new dedicated change.
- It should not be mixed into the archived PHP dynamic dependency change.
- `--path` should not be assumed dead without evaluation.
- Bootstrap scenarios without config are important and should be considered explicitly.
- The change should start in discovery/design mode, not implementation mode.
- The change may begin with no attached specs to avoid prematurely locking scope.
- Additional graph CLI concern to include in this change: `--kind` should move to a single comma-separated list value rather than repeatable single-value flags whose last occurrence wins.
- Desired direction discussed: allow exactly one `--kind` occurrence, accept comma-separated values, parse with the existing helper, and pass the resulting kinds into the query layer.
- The change must also define what the CLI does when `--config <path>` is passed explicitly.
- `--config` and `--path` are mutually exclusive by design.
- `--config` keeps its established CLI meaning: explicit alternate config file.
- `--path` is repo-root bootstrap mode.
- Autodetected config wins when `--config` is absent and `--path` is absent.
- `--path` forces repo-root mode and ignores config, even if a config exists.
- `--path` and no-config fallback are for bootstrap/setup flows, not the intended steady-state mode for configured projects.
- `--kind` should accept a single comma-separated value such as `class,method,symbol`; the problem is not “CSV” as a product concept but supporting a list in one option occurrence.
- `--kind` values must still be validated against the allowed `SymbolKind` values, and the command must fail if any token is invalid.

## Rejected alternatives

1. Keep the accidental CLI changes inside the PHP/code-graph change.
   Reason rejected: wrong scope and no prior agreement.

2. Remove `--path` outright.
   Reason rejected: the user raised a valid bootstrap use case where `--path` may still provide value.

3. Treat `--config` as the sole path forward immediately.
   Reason rejected: this would decide product semantics before design and before evaluating no-config workflows.

- What exact error contract should apply when the config path does not exist, cannot be read, or fails schema validation?
- Should fallback behavior be global across CLI commands or only for some command families?
- In repo-root bootstrap mode, what inferred workspace/config shape should graph commands use?
- Which specs should own the behavior: CLI command specs only, or also core config/workspace specs?
- Should multi-kind filtering apply only to `graph search` and `graph hotspots`, or to any future graph command that filters by symbol kind?
- Should the query contract use `readonly SymbolKind[]`, `ReadonlySet<SymbolKind>`, or another normalized form?
- Should repeated `--kind` be treated as an explicit CLI error, or simply be unsupported because the option is defined once and parsed as CSV?
- What exact error message/format should be shown when one or more `--kind` tokens are invalid?
