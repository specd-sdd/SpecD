# Exploration: indexer-config-exclude-paths

Generated: 2026-03-30

---

## Problem statement

The code graph indexer has no way to specify custom exclusion rules in the project config (`specd.yaml`). The only exclusions available are:

- Hardcoded `EXCLUDED_DIRS` set in `discover-files.ts` (`node_modules`, `.git`, `.specd`, `dist`, `build`, `coverage`, `.next`, `.nuxt`)
- `.gitignore` files (loaded hierarchically)

Users who want to exclude custom paths (e.g. generated files, large non-relevant dirs) must add them to `.gitignore`, which may not always be appropriate.

The only existing exclude support (`--exclude-path`, `--exclude-workspace`) is at query time (`graph search`, `graph hotspots`), not at indexing time.

---

## Approach / solution outline

Add a `graph` block to the per-workspace config in `specd.yaml` with two fields:

```yaml
workspaces:
  default:
    codeRoot: ./
    graph:
      respectGitignore: true # default true; if false, gitignore files are ignored entirely
      excludePaths: # gitignore-syntax patterns, supports negation with !
        - .specd/*
        - '!.specd/metadata/'
  core:
    codeRoot: packages/core
    graph:
      excludePaths:
        - dist/
```

Key design decisions:

1. **Per-workspace, not project-level** — each workspace has its own `codeRoot`, so exclusions are naturally workspace-scoped. Patterns are applied relative to the workspace's `codeRoot`.

2. **Gitignore syntax with negation support** — uses the same `ignore` library already in `discoverFiles`. Supports `!pattern` negation.

3. **`respectGitignore` flag (default `true`)** — controls whether `.gitignore` files are loaded at all. When `false`, only `excludePaths` applies. When `true`, `.gitignore` has **absolute priority**: config `excludePaths` cannot re-include what gitignore already excluded.

4. **Hardcoded `EXCLUDED_DIRS` become defaults** — the current hardcoded set (`node_modules/`, `.git/`, `.specd/`, `dist/`, `build/`, `coverage/`, `.next/`, `.nuxt/`) is removed from code and becomes the default value of `excludePaths` at the config/composition layer. Users can override by specifying their own `excludePaths` (which replaces the defaults entirely, so users must re-include the ones they still want).

   **Open question:** Should the defaults be merged with user-provided `excludePaths` (user adds on top), or replaced entirely (user takes full control)? No decision reached — needs to be resolved during design. Merging is safer; replacing is more flexible.

5. **Evaluation order when `respectGitignore: true`:**
   - Step 1: Check `.gitignore` rules — if ignored, file is excluded (no override possible)
   - Step 2: Check `excludePaths` config rules — can exclude additional paths or negate hardcoded defaults

6. **Evaluation order when `respectGitignore: false`:**
   - Only `excludePaths` config rules apply

## Affected areas

### `core:core/config`

- Add `graph` block to workspace config schema
- Fields: `graph.respectGitignore` (boolean, default `true`), `graph.excludePaths` (string[], gitignore-syntax)
- Validation: patterns must be strings; `respectGitignore` must be boolean

### `code-graph:code-graph/indexer`

- Update `discoverFiles` signature to accept `excludePaths?: string[]` and `respectGitignore?: boolean`
- Remove hardcoded `EXCLUDED_DIRS` constant (or make it the fallback default)
- Apply `excludePaths` as a separate `ignore` instance after gitignore evaluation
- When `respectGitignore: false`, skip loading `.gitignore` files entirely

### `code-graph:code-graph/workspace-integration`

- Extend `WorkspaceIndexTarget` (or equivalent) to carry `graph.excludePaths` and `graph.respectGitignore` from the config
- Pass these values through the composition layer to `discoverFiles`

## Key codebase observations

### `discoverFiles` implementation (`packages/code-graph/src/application/use-cases/discover-files.ts`)

- Uses the `ignore` npm library
- `EXCLUDED_DIRS` hardcoded set at line 5–14: `node_modules`, `.git`, `.specd`, `dist`, `build`, `coverage`, `.next`, `.nuxt`
- `loadIgnoreFile()` loads `.gitignore` hierarchically (git root + codeRoot + subdirs during walk)
- `isIgnored()` correctly handles `result.unignored` for negation patterns (lines 88–99)
- **Critical:** `EXCLUDED_DIRS.has(entry)` check at line 126 runs BEFORE `isIgnored()` — so negation patterns CANNOT override hardcoded dirs with the current implementation. This is why hardcoded dirs must be moved to defaults, not kept as a separate guard.
- `ignore` library's `.test()` returns `{ ignored, unignored }` — negation is already supported by the library

### Config YAML schema

- Workspace config is in `specd.yaml` under `workspaces.<name>`
- Existing fields: `codeRoot`, `specs`, `context`, etc.
- `graph` block does not currently exist

### Query-time excludes (existing)

- `--exclude-path` and `--exclude-workspace` flags exist on `graph search` and `graph hotspots`
- These use `SearchOptions.excludePaths` and pass to `matchesExclude` service
- These are glob-style (not gitignore-syntax) and apply at query time only
- The new indexing `excludePaths` is a separate concern and uses different syntax (gitignore)

## Rejected alternatives

- **Global config instead of per-workspace** — rejected: each workspace has its own `codeRoot` and `.gitignore`, so per-workspace is the natural scope.
- **Gitignore negation can override `.gitignore`** — rejected: gitignore has absolute priority when `respectGitignore: true`. If a file is in `.gitignore`, it stays excluded no matter what `excludePaths` says.
- **Keeping EXCLUDED_DIRS hardcoded and adding excludePaths on top** — rejected: this would prevent users from indexing things like `.specd/metadata/` which require overriding a hardcoded dir. Moving to defaults enables full control.

---

## Open questions

1. **Default excludePaths: merge or replace?** If a user specifies `excludePaths`, do the hardcoded defaults still apply (merge), or does the user take full control (replace)? Safer to merge; more flexible to replace. Needs decision in design.

2. **Where do defaults live?** In the config schema as default values? In the composition layer? In `discoverFiles` itself as fallback params? Needs architectural decision aligned with hexagonal architecture rules.

3. **CLI `graph index` command** — `--exclude-path` flag added (merge semantics, additive over config). `--respect-gitignore` CLI flag rejected — config-only.
