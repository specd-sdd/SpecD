## Problem statement

The user wants to start a new specd change to introduce inheritance and hierarchy relations into the code graph in a language-agnostic way first, then implement support in each language adapter afterwards.

This is not intended to be a PHP-only improvement. ## Current state of inheritance in specd

Current code-graph model does not appear to persist inheritance as a first-class graph relation.

Existing relation types in the current symbol model were discussed as:

- `IMPORTS`
- `DEFINES`
- `CALLS`
- `EXPORTS`
- `DEPENDS_ON`
- `COVERS`

There is currently no first-class relation such as:

- `EXTENDS`
- `IMPLEMENTS`
- `OVERRIDES`
- `MEMBER_OF`

Implications of the current state:

- class inheritance is not stored as explicit graph edges
- interface implementation is not stored as explicit graph edges
- override relationships are not stored
- traversal cannot natively answer questions like:
  - what subclasses inherit from this class
  - what implements this interface
  - what method overrides this base method
- impact analysis does not currently propagate through hierarchy semantics
- hotspot scoring cannot currently treat heavily inherited base classes or widely implemented interfaces as structurally more important

## Affected graph consumers identified

The conversation established that inheritance relations would affect more than language adapters.

### Impact analysis

The user explicitly pointed out:

- if you modify a base class, impact should include classes that inherit from it

Further likely implications:

- if you change an interface, implementations may be impacted
- if you change an overridable method, subclasses/overrides may be impacted

### Hotspots

Inheritance should likely affect hotspot scoring:

- a heavily inherited base class should likely score higher risk/importance
- a widely implemented interface should likely score higher
- hierarchy centrality is structurally meaningful, not only call/import counts

### Calls / resolution

Inheritance could also improve call resolution:

- calls through base-class receivers
- `parent` / `super`
- inherited methods when no local override exists
- override-aware dispatch approximations where deterministic

### Navigation / graph queries

Potential new questions enabled by hierarchy relations:

- who extends this class
- what implements this interface
- what overrides this method
- what base method does this override come from

## Design questions explicitly surfaced

These questions should be addressed during `/specd-design`.

### What relations belong in the base graph model?

Candidates explicitly discussed:

- `EXTENDS`
- `IMPLEMENTS`
- maybe `OVERRIDES`

Open question:

- should `OVERRIDES` be first-class, or derived from symbol hierarchy plus method matching

### How universal should the model be?

Need to decide how to normalize different language constructs:

- class inheritance
- interface implementation
- protocols
- traits
- mixins
- embeds / composition-like inheritance-adjacent constructs
- other language-specific hierarchy constructs

Question:

- what belongs in the universal model vs adapter-specific semantics

### Should hierarchy exist only as internal analysis input, or as persisted graph edges?

This was discussed directly.

Key point:

- if impact and hotspots should consume hierarchy, persisting hierarchy as first-class graph relations is likely necessary or at least strongly favored

But the design must decide whether:

- hierarchy is a first-class persisted graph relation
- hierarchy is partially persisted and partially derived
- some relations are explicit while others are computed on read

### How should traversal use hierarchy?

Needs design answers for:

- impact propagation from base class to subclasses
- impact propagation from interface to implementations
- reverse traversal from implementation to contract/base
- whether inheritance edges affect risk levels directly

### How should hotspots use hierarchy?

Needs explicit scoring rules or principles for:

- base classes with many descendants
- interfaces with many implementations
- possibly methods with many overrides

### How should adapters participate?

The conversation established a likely order:

1. define global relation model and semantics first
2. then add support in each adapter

The design should clarify:

- what minimum extraction contract every adapter must support
- what happens when a language lacks a direct equivalent for `IMPLEMENTS` or `OVERRIDES`
- whether unsupported relations are omitted or normalized to a looser common relation
