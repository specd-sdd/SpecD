# Exploration: php-adapter-dynamic-deps

Generated: 2026-03-30

---

## Problem statement

The PHP language adapter in `@specd/code-graph` only extracts static `use` namespace declarations syntactically but:

1. Never resolves them to file paths (PSR-4 resolution is a TODO comment in the code)
2. Completely misses dynamic dependencies — when a model is loaded via `$this->loadModel('User')`, the dependency graph doesn't see it at all
3. Same gap for `require`/`include`/`require_once`/`include_once` — PHP pre-namespace code uses these to declare file dependencies

The user reported the issue specifically with CakePHP legacy systems, but the problem is broader across the PHP ecosystem.

## Approach / solution outline

Three new capabilities to add to the PHP adapter:

### 1. `require`/`include` resolution

- Walk AST for `require_expression`, `require_once_expression`, `include_expression`, `include_once_expression` nodes
- When the argument is a string literal, create an `IMPORTS` relation to the resolved path
- For relative paths: resolve relative to the file's directory
- For paths with PHP constants (`APPPATH`, `__DIR__`, etc.): best-effort — skip silently if not resolvable to a literal path
- Covers: CakePHP 1.x, CodeIgniter 1.x, Zend 1.x, WordPress, Drupal 7, pre-namespace PHP generally

### 2. Dynamic loader detection (framework-specific)

- Walk AST for known method/static call patterns with a string literal first argument
- Emit `DEPENDS_ON` relation with metadata `{ dynamic: true, via: '<method>', value: '<string-arg>' }`
- Do NOT attempt to resolve to file paths (too framework-specific without PSR-4)
- Patterns to detect:

| Call pattern                             | Framework           | AST node type                                               |
| ---------------------------------------- | ------------------- | ----------------------------------------------------------- |
| `$this->loadModel('User')`               | CakePHP 1.x–4.x     | member_call_expression, method=loadModel                    |
| `$this->load->model('User_model')`       | CodeIgniter 1.x–3.x | chained member_call_expression, method=model on load        |
| `$this->load->library('email')`          | CodeIgniter 1.x–3.x | same, method=library                                        |
| `$this->load->helper('url')`             | CodeIgniter 1.x–3.x | same, method=helper                                         |
| `App::uses('Controller', 'Controller')`  | CakePHP 2.x         | static_call_expression, class=App, method=uses              |
| `ClassRegistry::init('User')`            | CakePHP 2.x         | static_call_expression, class=ClassRegistry, method=init    |
| `Yii::import('application.models.User')` | Yii 1.x             | static_call_expression, class=Yii, method=import            |
| `Yii::createObject('ClassName')`         | Yii 2.x             | static_call_expression, class=Yii, method=createObject      |
| `Yii::$app->get('componentId')`          | Yii 2.x             | chained, harder — lower priority                            |
| `Zend_Loader::loadClass('Zend_Db')`      | Zend 1.x            | static_call_expression, class=Zend_Loader, method=loadClass |
| `\Drupal::service('entity.manager')`     | Drupal 8+           | static_call_expression, class=Drupal, method=service        |
| `app('ClassName')` / `resolve('X')`      | Laravel             | function_call_expression (free function)                    |
| `$container->get('service_name')`        | Symfony/generic     | member_call_expression, method=get — too generic, skip      |

Implementation strategy: define a lookup table of `{ class?, method, framework }` tuples and walk the AST once checking against it. Avoids N separate walks.

### 3. PSR-4 namespace resolution

- Add optional method to `LanguageAdapter`: `resolveQualifiedNameToPath?(qualifiedName: string, codeRoot: string, repoRoot?: string): string | undefined`
- PHP adapter implements it by reading `composer.json` `autoload.psr-4` + `autoload-dev.psr-4` sections
- Builds a prefix→path map (e.g. `"App\\" → "src/"`)
- Longest-prefix match on the qualified name
- Returns the resolved file path (e.g. `App\Models\User` → `src/Models/User.php`)
- The indexer calls this method (if present) during Pass 2 when resolving PHP `use` statements
- Result: `use App\Models\User` generates a real `IMPORTS` relation (not just a syntactic declaration)
- Caching: the PSR-4 map should be cached per codeRoot to avoid re-reading composer.json on every call

## Design decisions

- `vendor/` exclusion is OUT OF SCOPE — handled in parallel change
- PSR-4 is IN SCOPE — user explicitly requested it
- Dynamic loader relations use `DEPENDS_ON` (not `IMPORTS`) because we can't resolve to a file path without PSR-4. The metadata carries enough info for tooling to display these.
- `require`/`include` with string literals use `IMPORTS` (we can resolve these)
- `require`/`include` with dynamic expressions (constants, concatenation) are silently dropped — same policy as unresolvable CALLS
- Service container patterns like `$container->get('x')` are too generic (every ORM uses `->get()`) — NOT detected
- Laravel's `app()` / `resolve()` free functions are detectable but low priority since Laravel is modern (uses `use` statements extensively). Include if easy, otherwise defer.
- New `LanguageAdapter` optional method for PSR-4: `resolveQualifiedNameToPath?` — pure, synchronous, but does I/O (reads composer.json). Same contract as `getPackageIdentity`.
- Caching of PSR-4 map: internal to the PHP adapter instance (Map<codeRoot, prefix[]>)

---

## Rejected alternatives

- **Resolving dynamic loader strings to file paths without PSR-4**: rejected — too fragile, framework conventions differ
- **Implementing PSR-4 inside the indexer**: rejected — spec constraint says indexer must NOT contain language-specific logic
- **Detecting `$container->get()`**: rejected — too broad, would create false positives on any method named `get`
- **Implementing `resolveQualifiedNameToPath` inside `extractRelations`**: considered, but would require `codeRoot` to be passed to extraction methods, changing the interface. New optional method is cleaner.

---

## Open questions

- Where exactly does the indexer call `resolveQualifiedNameToPath`? During Pass 2, after `extractImportedNames` returns declarations. The indexer should check if the adapter implements the method and call it for each PHP `ImportDeclaration` where `isRelative: false` and the specifier looks like a qualified name (contains `\`).
- Should `resolveQualifiedNameToPath` be added to the `LanguageAdapter` interface spec, or documented as a PHP-specific extension? Decision: add to `LanguageAdapter` spec as an optional method — same pattern as `buildQualifiedName`.
- Does the indexer spec need updating? Likely yes — Pass 2 should mention calling `resolveQualifiedNameToPath` when available.

## Key codebase observations

- `extractRelations` receives `_content` (unused) and `_importMap` — to add PSR-4 resolution it either needs `codeRoot` passed in, or a new method is the right approach
- The `walk()` method currently uses a `switch` on kind — dynamic call detection will need to handle nested AST patterns (chained calls like `$this->load->model(...)` are harder than direct calls)
- `ensureLanguagesRegistered()` is called at the top of each public method — new methods must follow this pattern
- CakePHP 1.x uses `member_call_expression`; the tree-sitter PHP grammar node names need verification during implementation
- `findManifestField` utility already exists and handles the walk-up-to-repoRoot pattern — PSR-4 reading should use a similar approach or extend it

## Tangential topics

- PSR-4 + autoload-dev: dev dependencies (test fixtures etc.) may also have autoload entries — should include both `autoload` and `autoload-dev` sections
- CakePHP 3.x+ also uses `use` statements in addition to `loadModel` — PSR-4 will cover those; `loadModel` detection covers the rest
- In CakePHP 1.x the convention was that `loadModel('User')` maps to `app/models/user.php` — but without knowing the app root this can't be resolved. The `DEPENDS_ON` with metadata is the right approach.
- Indexer spec (`code-graph:code-graph/indexer`) may need a delta if Pass 2 logic changes to call `resolveQualifiedNameToPath` — check during design
