# Exploration Context

## Problem Statement

The change is driven by GitHub issue `#53`:

- URL: `https://github.com/lsmonki/SpecD/issues/53`
- Title: `code-graph: hotspots overweights file-level imports at symbol level`

The issue states that `graph hotspots` currently becomes semantically noisy because
file-level centrality is propagated too directly into symbol-level ranking.

Observed problem captured from the issue:

- Symbols inside heavily imported files can rank very highly even with little or no
  direct caller evidence.
- Multiple symbols from the same file can all inherit large hotspot scores mainly
  because the file itself is imported often.
- This makes hotspot output less useful for prioritization, review, and impact
  assessment.

Desired outcome captured from the issue:

- Symbols should rank primarily by symbol-level change risk.
- File-level information should remain a secondary signal, not the dominant one.

## Approach / Solution Outline

The conversation did not lock a final scoring formula. The intended design direction is:

- Revisit hotspot scoring in `@specd/code-graph`, especially how file-level import
  counts contribute to symbol-level score.
- Evaluate reducing or changing propagation of file importer counts so they do not
  dominate symbols that lack direct caller evidence.
- Consider whether symbol-level `CALLS` should be weighted more strongly than file-level
  `IMPORTS`.
- Consider whether low-signal symbol kinds should be downweighted, excluded by default,
  or exposed via explicit CLI filtering instead of appearing in the default ranking.

No implementation decision has been made yet about:

- the new formula,
- whether the change is purely scoring,
- whether default symbol kinds should change,
- whether the CLI should expose or document a new default view.

Those decisions are for `/specd-design`.

## Key Codebase Observations

### Current hotspot implementation

File: `packages/code-graph/src/domain/services/compute-hotspots.ts`

Observed current behavior:

- `computeHotspots(store, options?)` runs:
  - `store.getSymbolCallers()`
  - `store.getFileImporterCounts()`
- For symbols with callers, it currently computes:
  - `sameWs`
  - `crossWs`
  - `fileImporters`
  - `score = sameWs * 3 + crossWs * 5 + fileImporters`
  - `riskLevel = computeRiskLevel(totalCallers, totalCallers + fileImporters, 0)`
- For symbols with no callers but imported files, it can still emit entries with:
  - `score = fileImporters`
  - `riskLevel = computeRiskLevel(0, fileImporters, 0)`
