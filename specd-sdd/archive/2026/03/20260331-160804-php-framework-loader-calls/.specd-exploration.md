## Diagnosis reached in conversation

### Primary diagnosis

The adapter is indexing PHP symbols, but `CALLS` extraction is missing several dominant
legacy-framework patterns used in `iccms`.

### Specific gaps identified

1. `var/public/protected $uses = array(...)` is not currently recognized as a dynamic
   dependency declaration.

2. `loadController(...)` and `loadComponent(...)` currently only match with `$this->`
   and therefore miss bare forms like:
   - `loadController('X')`
   - `loadComponent('Y')`

3. The adapter does not seem to resolve the common sequence:
   - `loadModel('X')`
   - `new X()`
   - `$instance->method()`

   even though this appears frequently in `iccms`.

4. The existing loaded-instance logic is tightly tied to same-scope alias discovery and
   does not cover all realistic CakePHP loader-to-instance flows.

5. Hotspots are not the root problem. Hotspots are downstream consumers of graph data.
   The real defect is under-populated PHP `CALLS`.

## Heuristics and reasoning agreed in conversation

The user explicitly reasoned that:

- `var $uses = array(...)` should be recognized somehow
- it is a class property, typically appearing near the start of a CakePHP class
- in this context it is unlikely to mean something else
- in a CakePHP codebase, it should be safe to interpret it as framework dependency
  declaration rather than an arbitrary custom property

This led to an agreed heuristic direction:

- treat `uses` as a class-property loader signal when it is a class property
- require literal array-of-strings shape
- prefer CakePHP-ish locations or class contexts to avoid overfitting generic PHP
- derive aliases such as `$this->ModelName`

The user also pointed out that there are standalone `loadController` and
`loadComponent` usages without `$this->`, which should also be captured.

The user asked whether other frameworks had similar patterns or something like
`$this->uses`. The review of the adapter showed support for multiple framework loaders
already, but not `$this->uses` / class-property `uses`.

## Proposed implementation directions

These were either directly stated or strongly implied in conversation:

1. Add CakePHP class-property `uses` detection:
   - `var $uses = array(...)`
   - `public $uses = array(...)`
   - `protected $uses = array(...)`
   - consider short array syntax `[...]` if present in newer code

2. Extend CakePHP loader matching to accept bare forms without `$this->`:
   - `loadController('X')`
   - `loadComponent('X')`

3. Resolve instance creation after loader declarations:
   - `loadModel('X')`
   - `$foo = new X()`
   - `$foo->bar()`

4. Revisit alias derivation and binding propagation so that common CakePHP naming
   conventions map to resolvable callee methods more often.

5. Expand verification against a real-world target such as `../iccms`, even if only as
   manual validation or fixture-inspired tests.

## Open questions for design

1. Should class-property `uses` generate only loader bindings, or also immediate
   file-level `IMPORTS`?
2. Should `uses` support both `array(...)` and `[...]` syntaxes?
3. How strict should the heuristic be:
   - any class property named `uses`
   - only under Cake-style paths
   - only in classes extending Cake base classes when detectable
4. How should `new X()` instance binding interact with:
   - direct class names
   - namespaced classes
   - aliases created from loaders
5. Should Drupal and Yii patterns remain import-only, or should some service/object
   creation flows also produce `CALLS` when method calls are resolvable?
6. Do we need more framework-agnostic instance-tracking beyond CakePHP to avoid one-off
   heuristics?
