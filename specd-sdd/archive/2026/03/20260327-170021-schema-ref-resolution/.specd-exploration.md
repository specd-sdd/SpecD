# Exploration: schema-ref-resolution

Generated: 2026-03-27

## Problem statement

Currently `schema show` and `schema validate` can only operate on the project's active
(configured) schema. There is no way to show or validate an arbitrary schema by reference
(the same format used in `specd.yaml`'s `schema` field). Specifically:

- `schema show` has no `[ref]` argument and no `--file` flag — it can only show the active schema.
- `schema validate` has `--file <path>` for files on disk, but no way to pass a schema
  reference by name (e.g. `@specd/schema-std`, `#workspace:name`).

This means you cannot inspect or validate a schema before putting it in your config, nor
can you validate workspace schemas independently.

## Approach / solution outline

Add an optional positional argument `[ref]` to both commands, consistent with how `fork`
and `extend` already use `<ref>`. The `[ref]` uses the same format as the `schema` field
in `specd.yaml` — routed through `SchemaRegistry`:

- `@scope/name` — npm package
- `#workspace:name` — workspace-qualified
- `#name` or bare name — default workspace
- relative/absolute path — filesystem

Additionally, add `--file <path>` to `schema show` (which `validate` already has).

### User-confirmed decisions

- The positional argument is `[ref]` (optional), matching `<ref>` in `fork`/`extend`.
- `--file` is a separate flag (not the positional), consistent with `validate`'s existing design.
- When neither `[ref]` nor `--file` is provided, both commands fall back to the current
  behavior (active project schema).

## Affected areas

### CLI commands (packages/cli/src/commands/schema/)

- **show.ts** — add `[ref]` argument, add `--file <path>` option, dispatch to new use case modes
- **validate.ts** — add `[ref]` argument, dispatch to new `ref` mode in ValidateSchema
- **fork.ts** and **extend.ts** — reference only (they already use `<ref>`, pattern to follow)

### Core use cases (packages/core/src/application/use-cases/)

- **validate-schema.ts** — add `{ mode: 'ref', ref: string }` to `ValidateSchemaInput`.
  Implementation is similar to `file` mode but uses the ref directly with `SchemaRegistry.resolveRaw(ref)`.
  The registry already handles all ref formats.
- **get-active-schema.ts** — currently accepts no input and delegates to `ResolveSchema`.
  Needs to support an optional ref or file input. Options:
  1. Extend `GetActiveSchema` to accept optional input (ref or file)
  2. Create a new use case (e.g. `ShowSchema` or `ResolveSchemaByRef`)

  The first option is simpler but changes the spec's "accepts no input" requirement.
  The second is cleaner architecturally. Decision TBD during design phase.

### Core ports/infrastructure

- **SchemaRegistry** (port + FsSchemaRegistry) — no changes needed. `resolveRaw(ref)` and
  `resolve(ref)` already handle all ref formats. The infrastructure is ready.

## Key codebase observations

### Current command signatures

- `schema show` — no args, only `--format` and `--config`
- `schema validate` — no args, `--file <path>`, `--raw`, `--format`, `--config`
- `schema fork <ref> <name>` — ref is required positional
- `schema extend <ref> <name>` — ref is required positional

### ValidateSchema use case

- Already has 3 modes: `project`, `project-raw`, `file`
- `file` mode: resolveRaw(filePath) -> extends chain -> build -> return with warnings
- Adding `ref` mode follows the same pattern as `file` but with a ref string
- Extends chain warnings would say "extends '<ref>' resolved from <path>"

### GetActiveSchema use case

- Thin delegation to ResolveSchema — contains no logic
- ResolveSchema is constructed with schemaRef, plugins, overrides
- For a `[ref]` or `--file` case, we don't want plugins/overrides — just the schema itself
  with its extends chain resolved
- This means we need either a new use case or a different code path

### SchemaRegistry.resolve(ref) vs resolveRaw(ref)

- `resolve(ref)` returns a fully-built `Schema | null`
- `resolveRaw(ref)` returns `SchemaRawResult | null` (parsed data + templates + path)
- For `show`, we could use `resolve(ref)` directly — but it doesn't resolve extends chain.
  Actually, `resolve()` DOES build the schema but only from the single file, not the
  extends chain. The extends chain is handled by `ResolveSchema`.
- For show with ref, we likely need: resolveRaw -> extends chain -> buildSchema (same as
  validate-file mode but returning the Schema instead of a validation result)

### Mutual exclusivity

For `validate`: `[ref]`, `--file`, and `--raw` should be mutually exclusive in groups:

- `[ref]` vs `--file` — can't specify both
- `--raw` only applies to project mode (no ref, no file)
- So: `[ref]` + `--raw` = error, `[ref]` + `--file` = error

For `show`: `[ref]` vs `--file` — can't specify both

## Open questions for design phase

1. **New use case or extend GetActiveSchema?** For `schema show [ref]` / `schema show --file`,
   should we extend `GetActiveSchema` to accept optional input, or create a new use case?
   The current spec says "accepts no input" — changing this is a spec delta.

2. **Shared resolution logic between show and validate?** Both `show [ref]` and `validate [ref]`
   need: resolveRaw -> extends chain -> build. Currently validate-file does this. Should there
   be a shared internal helper, or should each use case duplicate the ~10 lines?

3. **`--raw` with `[ref]`?** Should `validate --raw` work with `[ref]`? Semantically it would
   mean "validate this ref without the project's plugins/overrides" — but `[ref]` already
   doesn't apply plugins/overrides. So `--raw` + `[ref]` would be redundant. Probably just
   make them mutually exclusive.
