# Exploration: studio-graph-health-warnings

## Problem statement

Studio Bell notifications show a **generic** stale-graph message when
`projectStatus.graph.stale === true`. CLI graph commands emit richer stderr via
`warnGraphStale` (VCS ref delta + derivation fingerprint mismatch). API/IPC DTOs
and `ProjectGraphSummaryDto` only expose `stale?: boolean` — not
`fingerprintMismatch` or structured warnings. Users miss fingerprint/config drift
that CLI already surfaces.

## Approach (agreed direction)

1. Extend wire DTOs (`GraphStatusDto`, `ProjectGraphSummaryDto`) with graph health
   fields needed for UI parity (at minimum `fingerprintMismatch`; consider
   `warnings: string[]` or typed `{ type, message }[]` built in presenters).
2. Map from `GetGraphHealthResult` / `buildProjectStatusSnapshot` in API presenters
   and IPC (same shapes via `@specd/client`).
3. Update `StudioTopBar` notifications per `ui:design-system` — show distinct
   cards for stale vs fingerprint mismatch (reuse CLI copy where sensible).
4. **Do not** change `GetGraphHealth` use case unless design requires a formal
   `warnings[]` on the domain result; presenters may derive warnings from
   `stale` + `fingerprintMismatch` + refs.

## Rejected / deferred

- **Full `GraphStatusDto.warnings[]` from SDK orchestration layer** — defer unless
  design needs shared warning builder in `@specd/sdk`; presenters acceptable for v1.
- **Terminal / bottom-panel** — separate future work; spec removed in `63bf9049`.
- **Implementation review stale symbols** — out of scope.
