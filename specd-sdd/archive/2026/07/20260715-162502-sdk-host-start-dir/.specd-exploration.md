# sdk-host-start-dir exploration

## Problem statement

The SDK host bootstrap currently exposes `configPath` and `kernelOptions` through `OpenSpecdHostInput`, but it does not expose a discovery-root input for hosts that select a project at runtime while keeping `process.cwd()` elsewhere. This forces non-CLI hosts to reimplement config loading and kernel bootstrap manually when they need normal discovery semantics instead of explicit forced-file semantics.

## Approach / solution outline

- Extend `OpenSpecdHostInput` with optional `startDir`.
- Keep `configPath` as forced mode.
- Use `startDir` for discovery mode.
- Fall back to `process.cwd()` only when neither is supplied.
- Reject simultaneous `configPath` and `startDir` with an actionable error.

## Key codebase observations

- `openSpecdHost` currently calls `createDefaultConfigLoader(input?.configPath !== undefined ? { configPath } : { startDir: process.cwd() })`.
- `createDefaultConfigLoader` already supports a discriminated union: `{ startDir }` or `{ configPath }`.
- `core:config-loader` already defines the semantic distinction between discovery mode and forced mode, including layered discovery and `specd.local.yaml` handling.
- `SpecdConfig` already carries `warnings?: readonly string[]`, so this change does not need new warning plumbing by itself.
- `resolveCliContext` only passes `configPath` plus `kernelOptions`; CLI is not the motivating host for this follow-up.
- Graph impact for `openSpecdHost` is `CRITICAL`, with broad CLI and SDK dependence, so bootstrap contract changes need tight scope.

## Design decisions

- This change is intentionally about discovery-root selection only.
- Do not mix in graph-provider customization or warning-contract changes here.
- Preserve current default behavior for callers that pass nothing.
- Preserve current forced-mode semantics for callers that pass `configPath`.

## Agreements reached

- Split the follow-ups into three separate changes instead of one grouped change.
- This change owns the explicit discovery-root API and mixed-input rejection.

4. Verify default `cwd` discovery, explicit `startDir` discovery, explicit `configPath` forced mode, and invalid mixed-input behavior.

## Rejected alternatives

- Passing `<projectRoot>/specd.yaml` as `configPath` instead of adding `startDir`.
  - Rejected because forced mode is not equivalent to discovery mode; it follows only the explicit file and its `extends` chain and does not automatically attach later discoverable variants.
- Grouping this with the other two follow-ups.
  - Rejected to reduce review scope and avoid mixing bootstrap input semantics with provider composition and diagnostic contract questions.
