# sdk-graph-provider-factory exploration

## Problem statement

`createSdkContext` currently hardwires `createCodeGraphProvider(config)` from `@specd/code-graph`. Hosts that need a compatible provider implemented elsewhere cannot substitute that lifecycle boundary through the SDK and must wrap or rebuild host context externally.

## Approach / solution outline

- Add a host-supplied provider-factory option to SDK bootstrap.
- Keep the default factory as `createCodeGraphProvider(config)`.
- Ensure each `createGraphProvider()` invocation returns a fresh provider instance.
- Preserve source compatibility for existing callers that do not supply the new option.

- `packages/sdk/src/composition/host-context.ts`

## Key codebase observations

- `createSdkContext(config, options?: KernelOptions)` currently returns `{ kernel, createGraphProvider: () => createCodeGraphProvider(config) }`.
- `openSpecdHost` delegates to `createSdkContext(config, input?.kernelOptions)`.
- `withOpenGraphProvider` already treats `ctx.createGraphProvider()` as the lifecycle seam and requires a fresh provider plus open/close handling.
- `resolveSdkHostContext` in CLI manually builds `createGraphProvider` with `createCodeGraphProvider(config)` when a kernel already exists; this is an adjacent consumer pattern to keep in mind.
- Graph impact for `createSdkContext` is `CRITICAL`, so preserving backward compatibility matters.

## Design decisions

- Keep this change strictly about graph-provider composition.
- Do not couple SDK to any Electron/UI-specific package.
- Prefer an option shape that composes with existing kernel options cleanly.
- Maintain the invariant that the factory sees the same resolved `config` as the kernel.
