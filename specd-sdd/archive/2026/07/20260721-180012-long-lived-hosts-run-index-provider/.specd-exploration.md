# Exploration: long-lived-hosts-run-index-provider

Generated: 2026-07-21

## Problem statement

Long-lived graph hosts (Studio desktop IPC + HTTP API) keep an open `CodeGraphProvider`
across the process/session. Indexing previously conflicted with SDK `runIndexProjectGraph`,
which always wrapped work in `withOpenGraphProvider` (open → index → close). That forced
workarounds:

- **API:** `releaseGraphProviderForIndex()` → `runIndexProjectGraph(ctx)` (transient
  provider) → `refreshGraphProvider()`
- **Desktop:** call `createIndexProjectGraph().execute({ provider })` directly on the
  long-lived instance, bypassing SDK orchestration

Main archived `allow-existing-provider-in-run-index-project-graph`: optional
`input.provider` on `RunIndexProjectGraphInput`. When supplied, index runs on that open
instance; must NOT close it; must NOT combine with `beforeOpen`/`afterClose`
(`InvalidProviderLifecycleError`).

This change adapts UI-branch hosts to that API.

## Approach / solution outline

1. Merge `origin/main` into `feat/user-interface` (done: `af78ec40`) so SDK +
   `sdk:run-index-project-graph` workspace specs include the provider option.
2. **Do not modify** `sdk:run-index-project-graph` or `packages/sdk` orchestration —
   consume as dependency only.
3. Update host specs + code:
   - API index route: pass long-lived provider into `runIndexProjectGraph`; remove
     release/refresh cycle around index (unless design finds a remaining need).
   - Desktop `indexGraph` IPC: use `runIndexProjectGraph` / SDK host with `{ provider }`
     instead of manual `createIndexProjectGraph`.
4. Keep long-lived accessors (`withGraphProvider` / `withHealthyGraphProvider`, peek vs
   healthy) from prior archive `merge-main-adapt-ui-branch`.
