# Exploration: host-controlled list limits

## Problem statement

Recent repository cache normalization (`20260716-083618-normalize-repository-caches`) introduced shared list pagination with **`limit` defaulting to 100** on every listable repository port. That breaks structural consumers that need the full catalog in one call — notably the Studio/API spec tree on branch `feat/user-interface` (`GET /v1/workspaces/{ws}/specs`), which builds a workspace tree from all spec paths.

Agents and discovery UIs should receive the **complete spec listing** by default. Tabular CLI browsing of changes (active, drafts, archive, discarded) should remain paginated with a sensible default, but that default belongs to the **host (CLI)**, not the repository port.

## Approach / solution outline

1. **Repository ports (`ListOptions`)**: No default limit. Pagination applies **only when `limit` is explicitly provided**. When omitted, `list()` returns the full filtered set in canonical order.
2. **`paginateList` / fs index caches**: Treat missing `limit` as unlimited (return all items after filter/sort).
3. **`ListMeta`**: When unpaginated, `count === total` and `limit` reflects actual count (or equals `total` — to be decided in design).
4. **CLI change list commands** (`changes list`, `drafts list`, `archive list`, `discarded list`): Host applies **default `--limit 100`** via `parseListPaginationFlags` / command action when user omits `--limit`.
5. **CLI spec list** (`specs list` / `spec list`): **No default limit** — forwards pagination flags only when user passes them; agents get full listing.
6. **Internal use cases**: Remove `Number.MAX_SAFE_INTEGER` hacks where they exist (`validate-specs`, `spec-pattern-matching`, `spec-repository.search`, `index-code-graph`, etc.) — plain `list()` or `list(undefined, {})` suffices.

## Design decisions (user-confirmed)

| Decision                         | Detail                                                                                            |
| -------------------------------- | ------------------------------------------------------------------------------------------------- |
| Port default                     | **No limit** — full catalog unless caller passes `limit`                                          |
| CLI changes                      | **Default limit 100** for active changes, drafts, archive, discarded                              |
| CLI specs                        | **No default limit** — full listing for agents                                                    |
| Rationale                        | Host decides pagination; repos expose complete data; tree/navigation use cases work without hacks |
| Rejected: `all: true` flag       | User preferred inverting default instead of opt-in unlimited                                      |
| Rejected: default 100 everywhere | Blocks UI tree and forces `MAX_SAFE_INTEGER` workarounds                                          |

## Rejected alternatives

- **`all: true` opt-in on `ListOptions`**: Explicit but adds another flag; user chose "no default limit" instead.
- **Separate `GetWorkspaceSpecTree` use case**: Valid but heavier; unnecessary if port returns full list by default.
- **Keep port default 100 + API passes high limit**: Perpetuates magic numbers and wrong layer for defaults.

## Key codebase observations

- Current default enforced in `paginateList`: `const limit = options?.limit ?? 100` (`packages/core/src/infrastructure/fs/list-pagination.ts`).
- `ListSpecs` forwards `limit ?? 100` (`packages/core/src/application/use-cases/list-specs.ts`).
- Internal callers already hack unlimited via `{ limit: Number.MAX_SAFE_INTEGER }` in search, validate-specs, spec-pattern-matching, index-code-graph.
- Prior change `20260716-083618-normalize-repository-caches` **deliberately** broke "return all" with default 100 — this change **partially reverts** that decision at the port layer only.
- `feat/user-interface` handler (`packages/api/.../handler-workspaces.ts`) calls `kernel.specs.list.execute({ includeSummary: true })` expecting full array — incompatible with current paginated `ListSpecs` return shape until merge + this fix.

## Resolved design decisions (2026-07-22 follow-up)

### Port / core

| Rule                    | Behavior                                                                     |
| ----------------------- | ---------------------------------------------------------------------------- |
| No `limit`              | Return full filtered set                                                     |
| `meta.limit`            | **`=== meta.total`** when unpaginated                                        |
| `page` without `limit`  | **Error**                                                                    |
| `after` without `limit` | **Valid** — return all items strictly after cursor (no final slice by limit) |
| Use cases               | **No defaults** — forward options to repos; return what repos return         |
| Host passes `limit`     | CLI/API inject `limit` when paginating                                       |

### CLI

| Command                                             | Default                                                                  | Truncation hint                           |
| --------------------------------------------------- | ------------------------------------------------------------------------ | ----------------------------------------- |
| `changes` / `drafts` / `archive` / `discarded list` | `--limit 100` when omitted                                               | Only when `limit` set and `count < total` |
| `spec list`                                         | No default limit                                                         | Only when user passed numeric `--limit`   |
| **`--limit all`**                                   | On change list commands — omit `limit` from use-case call (full listing) | N/A                                       |

### `--limit all` (confirmed over `none`)

- Special CLI value alongside positive integers.
- **Change list commands**: omitted → `{ limit: 100 }`; `--limit all` → `{}` (no limit to use case).
- **`--limit all` + `--page`** → error.
- **`--limit all` + `--after-key`** → OK (cursor without ceiling).
- **Spec list**: no default; `--limit all` optional synonym for unpaginated (may document or accept silently).
- **Not supported today** — current parser uses `parseInt`; must add explicit handling in `list-pagination.ts`.

### Docs

- Update `docs/cli/cli-reference.md` and `docs/core/use-cases.md` in this change.
- CLI docs describe default 100 for change lists and `--limit all` escape hatch.

### `paginateList` behavior matrix

```
no limit, no after  → all items
no limit, with after → all items from cursor onward
with limit, with/sin after → existing pagination
with limit, page     → offset pagination
page without limit   → error
```
