# sdk-host-warning-contract exploration

## Problem statement

Configuration loading can emit non-fatal warnings through `SpecdConfig.warnings`, but reusable host bootstrap does not yet define one normalized contract for how hosts should receive and report those warnings. Without a clear contract, hosts may drop warnings silently or emit them multiple times when config loading and host bootstrap are composed separately.

## Approach / solution outline

- Define one main SDK-owned contract for warning propagation during host bootstrap.
- Preserve the rule that SDK itself performs no stdout/stderr I/O.
- Decide whether the contract is:
  - formalizing `config.warnings` as the bootstrap source of truth, or
  - adding a normalized `warnings` field on `OpenSpecdHostResult` while retaining `config.warnings` for compatibility.

## Key codebase observations

- `SpecdConfig` already has `warnings?: readonly string[]`.
- `resolveCliContext` currently reads `host.config.warnings` and prints each one once with `console.warn(...)`.
- `openSpecdHost` currently returns `{ config, configFilePath, ...ctx }` and does not normalize warnings separately.
- `MAIN_FOLLOW_UPS.md` explicitly requires no SDK console I/O.
- This is more a contract/ownership clarification than a deep bootstrap mechanics change.

## Design decisions

- Keep this change separate from discovery-root and provider-factory changes.
- Preserve host responsibility for formatting/output.
- Avoid changing warning severity; warnings must remain non-fatal diagnostics.

## Agreements reached

- This follow-up should be its own change even though it shares `sdk:host-context`.
- The current CLI behavior is important evidence and likely either becomes the reference consumer or gets slightly adjusted to the final contract.

## Steps / plan defined

1. Update `sdk:host-context` to define host bootstrap warning propagation explicitly.
2. Update `cli:host-context` so CLI consumption matches the chosen SDK contract.
3. Update `core:config` only as needed to clarify where warnings originate and their stability.
4. Verify warnings survive bootstrap unchanged, are emitted once by hosts, and SDK performs no console I/O.

## Rejected alternatives

- Leaving warning propagation implicit.
  - Rejected because hosts beyond CLI need a single contract and duplicate/drop behavior is easy otherwise.
- Merging this with the start-dir or provider-factory changes.
  - Rejected because the main decision here is semantic ownership of diagnostics, not bootstrap wiring.

- If both fields exist, what is the canonical field hosts should use?
- Should CLI continue reading `host.config.warnings`, or should it switch to a new normalized host result field?
