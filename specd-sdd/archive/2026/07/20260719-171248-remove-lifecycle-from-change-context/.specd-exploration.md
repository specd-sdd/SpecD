# Exploration context

Generated: 2026-07-19

## Problem statement

`change context` currently includes lifecycle-derived information. That information
changes when the change progresses through its lifecycle even when the working
context itself has not changed. Because the context fingerprint is calculated from
the compiled result, these lifecycle-only changes invalidate the fingerprint and
cause agents to receive the context again unnecessarily.

The user considers the lifecycle information redundant because `change status`
already provides lifecycle state. They want `change status` to be the single source
of lifecycle information.

## Agreed solution

Remove lifecycle information from `change context` rather than only excluding it
from its fingerprint. This includes the requested step's availability, blocking
artifacts, and the workflow-step availability collection. The resulting context
fingerprint should represent only material context supplied to an agent, so a
lifecycle transition alone does not change it.

## Scope

- Change name: `remove-lifecycle-from-change-context`.
- Initial specs: `core:compile-context`, `cli:change-context`.
- Both workspaces are `owned`.
- Known change dependency: `cli:change-context` depends on `core:compile-context`.

## Relevant observations

- `core:compile-context` currently defines `CompileContextResult` with
  `stepAvailable`, `blockingArtifacts`, and `availableSteps`.
- The `AvailableStep` type is defined in
  `packages/core/src/application/use-cases/compile-context.ts` around line 173;
  `CompileContextResult` follows around line 189.
- `CompileContext` currently evaluates lifecycle readiness through
  `LifecycleEngine`. Its spec requires that the fingerprint change when step
  availability and blocking artifacts change.
- `cli:change-context` currently renders a warning when the requested lifecycle
  step is unavailable and requires CLI output to project those lifecycle fields.
- `compileContextFingerprint` is in
  `packages/core/src/application/use-cases/_shared/compile-context-fingerprint.ts`
  around line 34. Graph analysis reports CRITICAL risk: 6 direct dependents,
  36 indirect dependents, and 232 transitive dependents. Design and implementation
  should make the result-contract removal intentional and cover callers/tests.
- The graph was fresh at discovery time (`2026-07-19T16:45:02.484Z`).

## Required behavioral direction

- `change context` must not output lifecycle state, requested-step availability,
  lifecycle blockers, or per-step availability.
- `change status` remains the command used to obtain lifecycle state.
- Fingerprint short-circuiting must remain effective across lifecycle-only changes
  when all actual context inputs are unchanged.
- Context content, warnings that are genuinely emitted as context diagnostics, and
  result-shaping flags remain outside this specific removal unless design discovers
  a necessary contract adjustment.

## Rejected alternative

Only omit lifecycle fields from the fingerprint while continuing to return them in
`change context`. Rejected because it leaves redundant lifecycle output in the
context response, while an unchanged fingerprint would no longer mean that the
entire response is unchanged.

## Open questions for design

- Confirm whether any non-CLI consumer relies on the removed `CompileContextResult`
  fields and define the migration/update path.
- Confirm the intended structured-result shape after removal and update its
  contracts/tests accordingly.
- Confirm whether a `change context` request should still validate that its named
  lifecycle step exists, even though it no longer reports whether the step is
  currently available.

## Conversation flow

1. The user identified that lifecycle state in change context changes the context
   fingerprint despite no meaningful context change, and noted that lifecycle state
   is already available from change status.
2. Discovery found the existing requirements in `core:compile-context` and
   `cli:change-context`, including the current explicit requirement that a
   fingerprint changes when step availability changes.
3. The initial proposal was to exclude lifecycle data from the fingerprint.
4. The user asked whether it would be better to remove lifecycle information from
   change context entirely.
5. The agreed scope became removal of lifecycle information from change context,
   using change status as the sole lifecycle-information source.
