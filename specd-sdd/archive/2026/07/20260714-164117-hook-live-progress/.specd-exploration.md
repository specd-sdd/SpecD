# Problem statement

The user wants hook execution to feel observable during long-running commands instead of appearing stalled until the subprocess exits. The concrete pain point is that both humans and agents currently see little or no signal while a hook is running, which makes it hard to know whether progress is happening or whether the process is hung.

# Approach / solution outline

Design hook execution so that long-running hooks emit live progress instead of only returning buffered stdout/stderr at the end.

Core design direction discussed:

- move from "capture all output and report at process end" to streaming execution progress
- preserve final per-hook results and complete output capture for failure reporting
- expose progress in a way that works for both humans and agents
- keep human-readable history visible instead of constantly repainting and erasing prior hook output

Proposed three-layer approach:

1. `NodeHookRunner` emits live execution events while still returning final `HookResult`
2. `RunStepHooks` propagates hook lifecycle and output progress upward
3. CLI commands render differently for human text mode versus agent-oriented structured modes

# Design decisions

- Long-running hooks must produce periodic observable progress, not just a final buffered result.
- The design must serve both humans and agents.
- Human-facing output should keep prior hook history visible; completed hooks should remain on screen rather than being overwritten.
- A hook currently running should show what command is executing and the latest output lines.
- If a hook fails, the full output of that hook should be shown.
- The default implementation should avoid aggressive terminal repainting if it would make agent consumption worse.
- Heartbeats are valuable when a hook is alive but not currently writing output.

# Agreements reached

- The user wants visibility into "what is executing" and "the last lines" while the hook runs.
- After a successful hook, the UI should move to the next hook while keeping the previous hook's visible history.
- The design must account for agent consumption, not just human terminal UX.
- The real issue is that for long-running hooks the agent currently infers that nothing is happening, so the command model needs to expose ongoing activity.
- The recommended split is:
  - `text` for human-readable append-only progress
  - `json|toon` for structured, agent-safe progress

# Steps / plan defined

High-level design plan discussed:

1. Replace end-only buffered hook execution with streaming execution events from the runner.
2. Extend hook progress events beyond `hook-start` and `hook-done`.
3. Propagate output/tail/heartbeat events through `RunStepHooks`.
4. Render human text mode as persistent per-hook blocks with visible tail.
5. Provide structured progress for agent-oriented formats so long-running work is observable without terminal noise.
6. Preserve full hook output for failure reporting.

# Rejected alternatives

- Purely decorative spinner-only UX was considered insufficient because it does not solve the agent observability problem by itself.
- A mode that continuously rewrites or overwrites previously rendered hook output was considered undesirable because the user explicitly wants earlier executed hooks to remain visible.
- Mixing noisy human progress output directly into structured agent formats was rejected because it would likely degrade agent parsing and reasoning.

# Open questions

- Whether structured progress for agents should be emitted as incremental NDJSON, incremental toon events, or another transport pattern.
- Whether live progress should be default in `text` mode or gated behind an explicit `--live` flag.
- Whether `change transition` should reuse the exact same per-hook rendering blocks as `change run-hooks` or use a more compact transition-oriented view.
- Whether hook progress should include elapsed time only on heartbeat or on every update.
- Whether the runner should emit line-level events, chunk-level events, or maintain tail windows centrally.
