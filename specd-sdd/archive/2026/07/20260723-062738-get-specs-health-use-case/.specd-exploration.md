# Exploration Context: Spec Health Use Case

Generated: 2026-07-23

## Problem Statement

The user wants a programmatic use case (and its corresponding `create` factory functions) to query the overall health of specifications in the project. Currently, the only way to validate specifications is through the `ValidateSpecs` use case, which returns a detailed result containing every single spec validation entry (`SpecValidationEntry`). For large codebases, this response payload can be very large and contains mostly successful entries. The user needs a lightweight and filtered outcome that reports general metrics and only lists specifications containing actual failures or warnings.

## Approach / Solution Outline

1. **New Use Case:** Create `GetSpecsHealth` in `packages/core/src/application/use-cases/get-specs-health.ts`.
2. **Reutilisation:** Inject `ValidateSpecs` into `GetSpecsHealth` via dependency injection.
3. **Filtering:** In `GetSpecsHealth.execute()`, call `ValidateSpecs.execute()` and filter the entries.
   - Compile only entries where `passed === false` or `warnings.length > 0`.
   - Aggregate statistics: `totalSpecs`, `passed`, `failed`, and `warningCount` (number of specs with warnings).
4. **Composition:** Define dependency resolution and factory function `createGetSpecsHealth` in `packages/core/src/composition/use-cases/get-specs-health.ts`.

## Design Decisions

- The new usecase must reuse `ValidateSpecs` rather than rewriting validation/caching logic.
- The returned result `GetSpecsHealthResult` will not contain the full entries list. Instead, it will contain maps or arrays of failing specs (with their errors) and specs with warnings (with their warnings).

## Agreements Reached

- Reusing `ValidateSpecs` is the correct, DRY, and cache-preserving way to accomplish this.
- The change is scoped to the `core` package as it is a pure business logic use case.
