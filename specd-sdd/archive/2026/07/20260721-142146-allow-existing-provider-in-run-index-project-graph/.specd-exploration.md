# Exploration Context — allow-existing-provider-in-run-index-project-graph

Generated: 2026-07-21

## Problem Statement

`runIndexProjectGraph` in `@specd/sdk` currently wraps project indexing in `withOpenGraphProvider`, which instantiates, opens, and closes a `CodeGraphProvider` for each execution.

While this is ideal for short-lived CLI commands, long-lived hosts (such as MCP servers, API servers, or background daemons) maintain a persistent `CodeGraphProvider` instance in memory. If a long-lived host calls `runIndexProjectGraph`, `withOpenGraphProvider` closes the provider connection at the end of the run. Conversely, calling `provider.index()` directly forces the host to manually duplicate project orchestration boilerplate (workspace discovery, Git VCS ref/root detection, graph config building, and versioning).

## Approach / Solution Outline

Extend `RunIndexProjectGraphInput` in `@specd/sdk` to accept an optional `provider?: CodeGraphProvider` field.

1. **When `input.provider` is supplied**: Execute `IndexProjectGraph` directly using the provided `CodeGraphProvider` instance without wrapping it in `withOpenGraphProvider` (avoiding `provider.close()`).
2. **When `input.provider` is omitted**: Retain the existing behavior of opening a transient provider via `withOpenGraphProvider(ctx, ...)`.

## Design Decisions & Agreements

- Add `readonly provider?: CodeGraphProvider` to `RunIndexProjectGraphInput`.
- Extract internal indexing execution callback `executeIndex(provider)` in `runIndexProjectGraph`.
- If `input.provider !== undefined`, return `executeIndex(input.provider)`.
- Otherwise, return `withOpenGraphProvider(ctx, executeIndex, providerOptions)`.
- Complete backward compatibility with zero breaking changes for existing CLI/script usages.
