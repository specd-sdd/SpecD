# Exploration Context

## Problem Statement

The conversation started from a concrete repository behavior question: whether `SpecRepository` treats `spec-lock.json` as just another artifact. Investigation confirmed that `spec-lock.json` currently appears in `spec.filenames` and can be read through the generic artifact surface, while at the same time it also has dedicated persistence APIs (`readPersisted*`, `updatePersisted*`, publication fields).

This led to a broader design discussion about the relationship between:

- `spec-lock.json`
- `metadata.json`
- spec context commands and use cases
- compile context behavior inside changes
- spec validation responsibilities

The core problem identified is not only repository surface leakage, but inconsistent treatment of `dependsOn` across canonical metadata, persisted sidecar state, and change snapshots.

## Approach / Solution Outline

The user clarified the intended model:

- `metadata.json` is not merely a convenience cache in the narrow sense; it is the canonical normalized form used to unify heterogeneous spec schemas for consumers.
- Different schemas may have different artifact layouts:
  - two-file schemas (`spec.md` + `verify.md`)
  - one-file schemas
  - schemas with different section naming
  - schemas such as OpenSpec-style ones that do not express dependencies in `spec.md`
- `spec-lock.json` exists in part to persist data that some schemas cannot express directly.

From that model, the intended direction is:

- `metadata.json` should expose a canonical normalized view for consumers.
- `spec-lock.json` should supply canonical data when the schema cannot derive it from spec artifacts.
- `change.specDependsOn` should remain a separate change-level snapshot, because a change needs to know the dependency baseline at the moment the spec enters the change so later dependency edits can be archived as a new snapshot.

## Key Design Agreements Reached

### 1. `spec-lock.json` is not conceptually a normal spec artifact

Even though the current repository implementation exposes it through `spec.filenames` and the generic artifact APIs, the conversation concluded this is a leaky abstraction. The durable model already treats it separately in `publish()` and the dedicated persistence methods.

### 2. `metadata.json` is the consumer-facing canonical form

The user explicitly reframed the metadata layer as the canonical normalization mechanism across multiple schema styles. This changes the earlier framing from "metadata is only a cache" to "metadata is a canonical normalized view, but still derived/stale-able."

### 3. `dependsOn` must be canonicalized into metadata

The current implementation leaves a gap:

- if the schema can extract `dependsOn`, metadata may contain it
- if the schema cannot extract `dependsOn`, but `spec-lock.json` contains it, metadata may omit it

That omission breaks the promise of metadata as the normalized consumer form.

The intended direction is that `metadata.json` should include canonical `dependsOn` even when its source is `spec-lock.json`.

### 4. `GetSpecContext` and `GetProjectContext` should be able to rely on metadata

The user and agent concluded that consumers should not need bespoke `spec-lock` logic if metadata is truly canonical. The current incorrectness is therefore in metadata construction / maintenance rather than in the CLI layer itself.

### 5. `CompileContext` is special because of `change.specDependsOn`

The user explained why `change.specDependsOn` exists:

- when a spec enters a change, the change records the dependency snapshot
- later, the change may modify those dependencies
- at archive time, the system needs both the baseline and the new snapshot to persist the updated dependency state

This means `CompileContext` should not simply ignore `change.specDependsOn`. It has a separate purpose from canonical metadata.

### 6. Validation must cover stale or inconsistent canonical metadata

The user stated that validation likely needs checks for:

- extraction-derived dependency data vs `spec-lock`-persisted dependency data not matching
- metadata being stale and therefore needing reconstruction

The conversation converged on the need for validation rules around metadata freshness and source mismatch, especially for fields like `dependsOn` whose source may be extraction or sidecar persistence.

## Key Codebase Observations

### Repository surface

- `FsSpecRepository.get()` and `list()` currently include all regular files in a spec directory, which causes `spec-lock.json` to appear in `spec.filenames`.
- `allowedSpecArtifactFilenames(spec)` adds `spec.filenames` into the allowed set for `artifact()` and `save()`, which means `spec-lock.json` is implicitly accepted there.
- `publish()` already treats `spec-lock` separately via `persistedSchema`, `persistedDependsOn`, and `persistedImplementation`.
- `_readSpecLock()` and `_saveSpecLock()` are dedicated private sidecar I/O paths.

### Context consumers

- `GetSpecContext` loads `repo.metadata(spec)` and, when `followDeps` is enabled, traverses dependency data derived from metadata, not persisted sidecar state.
- The CLI `spec context` command simply delegates to `kernel.specs.getContext.execute(...)`; it does not add dependency resolution logic of its own.

### Metadata generation

- `GenerateSpecMetadata` extracts metadata from schema-declared spec artifacts.
- It projects `implementation` from `readPersistedImplementation(spec)`.
- It does not project `dependsOn` from `readPersistedDependsOn(spec)`.

### Change snapshot semantics

- `create-change` and `edit-change` use `loadPersistedSpecDependsOn()` to seed `change.specDependsOn`.
- `loadPersistedSpecDependsOn()` reads `readPersistedDependsOn(spec)` first, then falls back to `metadata.dependsOn`.
- This confirms `change.specDependsOn` is intentionally a snapshot of the dependency baseline at scope entry time, not just a copy of current metadata.

### Compile context behavior

- `CompileContext` currently prioritizes `change.specDependsOn` for change-scoped specs.
- If no snapshot exists, it looks at `metadata.dependsOn`.
- If metadata is absent and the schema supports extraction of `dependsOn`, it can fall back to ad hoc extraction.

## Agreements About Desired Behavior

### Metadata semantics

- `metadata.json` should serve as the canonical normalized consumer form across heterogeneous spec schemas.
- It should not require downstream consumers to know whether a field came from extraction or persisted sidecar state.

### `dependsOn` semantics

- `dependsOn` should appear in canonical metadata even when it cannot be extracted from the schema’s artifacts.
- `spec-lock.json` is one of the legitimate canonical sources for that field.

### `change.specDependsOn` semantics

- `change.specDependsOn` must remain distinct from metadata because it represents the baseline snapshot at the time the spec entered the change.
- `CompileContext` should preserve that semantic distinction.

### Validation semantics

- Validation should likely detect stale metadata that must be rebuilt.
- Validation should likely detect mismatches between extraction-derived and sidecar-derived canonical fields, especially for `dependsOn`.

## Rejected Alternatives

### Rejected: fix only the CLI

The user asked for options "sin modificar cli". The conclusion was that the CLI should remain a thin consumer. The correct layer for the fix is `core`, especially metadata generation and validation.

### Rejected: treat `GetSpecContext` reading metadata as inherently wrong

At first the conversation moved toward saying the spec-context flow was wrong because it ignored `spec-lock`. After the user clarified the intended meaning of metadata, the conclusion changed:

- reading metadata in consumers is fine
- the bug is that metadata is not currently canonical enough

### Rejected: rely on metadata-only extraction for `dependsOn`

This is insufficient because some schemas cannot express dependencies in spec artifacts, so extraction alone cannot satisfy the canonical metadata promise.

### Rejected: collapse `change.specDependsOn` into metadata

The user clarified that `change.specDependsOn` exists to capture the dependency state at change creation/scope entry time, which is required for later archive behavior. Therefore it cannot simply be replaced by current metadata.

## Open Questions

1. What exact precedence should define canonical `metadata.dependsOn` when both extraction and `spec-lock` provide values?
2. Should mismatch between extraction and `spec-lock` be a hard validation failure, a warning, or a review-required state?
3. Should metadata freshness incorporate `spec-lock` changes for fields that are projected from sidecar state?
4. Should repository surface cleanup (`spec-lock.json` removed from `spec.filenames` / artifact APIs) be in scope of this same change, or should it be a separate follow-up once metadata canonicalization is in place?
5. Are there other fields beyond `dependsOn` and `implementation` that should be explicitly canonicalized from sidecar state into metadata?

## Likely Implementation Direction

The conversation points toward a change that:

1. clarifies in specs that `metadata.json` is the canonical normalized consumer representation
2. clarifies that `spec-lock.json` may supply canonical fields when schemas cannot derive them
3. updates metadata generation / update rules so `metadata.dependsOn` is populated consistently from canonical sources
4. preserves `change.specDependsOn` as a change snapshot
5. aligns `CompileContext` semantics with that snapshot-vs-canonical distinction
6. adds validation for stale or inconsistent canonical metadata

## User Preferences / Constraints

- The user wants the fix addressed through the specd workflow, not via a direct code-only escape hatch.
- The user prefers not to modify the CLI for the fix.
- The user is focused on long-term model correctness, not a narrow patch.
- The user repeatedly pushed on conceptual correctness: source-of-truth boundaries, canonical metadata semantics, and realistic schema heterogeneity.
