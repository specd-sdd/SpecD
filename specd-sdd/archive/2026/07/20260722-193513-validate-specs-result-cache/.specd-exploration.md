# Exploration: validate-specs-result-cache

Generated: 2026-07-22

## Problem statement

`ValidateSpecs` / `specd spec validate --all` revalidates every spec on every run: list → get → read artifacts → parse AST → `evaluateRules` / cross-artifact rules → metadata consistency (including possible `_extractDependsOn` re-extraction). With ~243 specs this is expensive when most specs have not changed.

User asked whether a cache similar to the FS list-index caches (`FsIndexCache` / `FsSpecIndexCache`) would be viable for this path.

## Approach / solution outline

Add a **result cache** for `ValidateSpecs`, sibling to list-caches under fs-cache, **not** mixed into `FsSpecIndexCache`.

- Location: `{configPath}/tmp/fs-cache/validate-specs/<workspace>/`
- Files: `.specd-index-meta.json` + `.specd-index.jsonl` (same family as other fs-cache buckets)
- Owned by / wired near `ValidateSpecs` (application + FS helper), **not** by `FsSpecRepository`
- Use case still reads specs only via `SpecRepository` port; cache is separate persistence for prior validation outcomes

### Freshness cascade (agreed)

1. **Meta OK?** — `!isInvalidated`, `schemaFingerprint` match, `engineVersion` match. Else full miss / rebuild bucket semantics.
2. **mtimes OK?** — compare `sourceFiles` stamps. If OK → **hard hit**: return cached `entry` (no hash, no validate).
3. **mtime differ → fingerprint** — compute `inputFingerprint`; if match → **soft hit**: return `entry` and **update stored mtimes** so next run wins at step 2.
4. **fingerprint miss** → run full `_validateSpec`, then upsert row (`entry` + new mtimes + new fingerprint).

### Meta contents (bucket-level)

- `totalCount`, `generatedAt`, `isInvalidated` (align with existing `IndexMeta` where possible)
- `schemaFingerprint` — hash of schema pieces that affect validation (spec-scoped validations + cross-artifact rules, schema identity/ref as needed). Schema change invalidates **entire** bucket.
- `engineVersion` — bump when core evaluation logic changes independently of schema YAML

### JSONL contents (per-spec row)

- `entry` — public payload ≈ `SpecValidationEntry` (`spec`, `passed`, `failures`, `warnings`)
- `sourceFiles` — helper-only mtime stamps (same role as list-cache freshness)
- `inputFingerprint` — hash of validation **inputs** (not of the result):
  - contents of `scope: 'spec'` artifacts that participate in validation
  - `metadata.json` (or fields that affect metadata consistency)
  - persisted `dependsOn` if not already covered by metadata
- `entryId` = qualified spec id `workspace:path`

### What fingerprint is for

Fingerprint is the **skip key** for expensive work (AST parse, rule eval, cross-rules, dependsOn re-extract). Hashing is cheap relative to that. mtime is the cheap first gate; fingerprint answers “did inputs actually change?” when mtime moved (e.g. touch).

### Soft-hit mtime refresh

When mtime fails but fingerprint OK: return cached entry **and** upsert row with updated `sourceFiles` mtimes; leave `inputFingerprint` and `entry` unchanged.

## Design decisions (confirmed)

1. Viable, but **not** by stuffing results into `FsSpecIndexCache` / `SpecListEntry`.
2. Bucket: `{configPath}/tmp/fs-cache/validate-specs/<workspace>/` (v1 FS adapter).
3. Cascade: meta → stamps (mtime) → fingerprint → validate.
4. Soft hit updates stamps/mtimes.
5. `ValidateSpecs` reads specs via `SpecRepository`; cache is a separate **application port**.
6. Schema/engine fingerprints in meta; per-spec `inputFingerprint` in jsonl.
7. Do not store ASTs or artifact contents in the cache.
8. Hosts MUST NOT know the cache exists.
9. Cache **all** `SpecValidationEntry` fields (pass/fail/warnings).
10. New meta type **extends** `IndexMeta` with `schemaFingerprint` + `engineVersion`.
11. `inputFingerprint` / stamps MUST include **`spec-lock.json`** (plus artifacts + metadata.json).
12. New spec: `core:validation-result-cache-port`.

## Agreements / preferences

- Fingerprints skip expensive checks; mtime/stamps first.
- Soft-hit refreshes stamps.
- Change name: `validate-specs-result-cache`.
- Scope: `core:validate-specs`, `core:storage`, `core:validation-result-cache-port`.
- CLI specs removed (host opacity).

## Key codebase observations

- `--all` path: for each workspace repo → `list(undefined)` (fs-cache list) → per row `get` → `_validateSpec` which **`get`s again**, then `artifact` per schema type, then `metadata` / `readPersistedDependsOn` / maybe re-read artifacts for dependsOn extract. Double `get` and possible double artifact reads today.
- List-cache already speeds discovery only; validation cost is content + rules.
