# Problem statement

The change investigates and likely refactors the public composition surface in `@specd/core`.

The initial suspicion was that use cases were coupled to filesystem storage. Investigation showed a more precise issue:

- `application/use-cases` are generally correctly decoupled and depend on ports such as `ChangeRepository` and `ActorResolver`.
- The real coupling is in `composition/use-cases` and repository composition factories, where many public `createX(...)` helpers explicitly construct fs-backed repositories and accept fs-shaped inputs (`changesPath`, `draftsPath`, `discardedPath`, `specsPath`, etc.).
- The kernel has evolved toward registry-based adapter resolution, but many standalone `createX(...)` factories still use direct `'fs'` adapter selection and explicit fs options.

The user wants to determine whether this will scale to future plugin-based adapters and concluded that the current surface should be redesigned more broadly, not just patched case by case.

# Approach / solution outline

The converged direction from the conversation is:

- Normalize all public composition factories around two signatures:
  - canonical: `createX(deps)`
  - convenience: `createX(config, options?)`
- Remove public `Fs*Options`-style explicit signatures from use-case composition factories.
- Introduce a shared resolver that reads `SpecdConfig`, combines built-ins with externally registered capabilities, resolves configured adapters, and returns a shared resolved dependency context.
- Make the kernel pure orchestration:
  - `createKernel(config, options?)` should resolve shared context once
  - kernel assembly should call `createX(deps)` rather than maintaining separate wiring logic
- Keep `kernel builder` as a fluent way to register capabilities/options for full kernel construction, not as a separate composition engine and not as a mechanism for composing individual use cases.

Working conceptual hierarchy agreed during exploration:

1. `createX(deps)` = canonical composition contract
2. `resolveCompositionContext(config, options?)` = single bootstrap/resolution path
3. `createX(config, options?)` = sugar over resolver + canonical `createX(deps)`
4. `createKernelFromContext(ctx)` = pure aggregator/orchestrator
5. `createKernel(config, options?)` = sugar over resolver + `createKernelFromContext`
6. `createKernelBuilder()` = fluent builder for composition options / registrations used by full-kernel bootstrap

# Design decisions discussed

## Decision: the real issue is not `application`, it is public composition

The user first raised concern about use cases being tied to fs.

Conclusion:

- `application/use-cases` are mostly correct
- public composition factories are where fs assumptions leak into the API

## Decision: the canonical public factory contract should be `createX(deps)`

This was explicitly agreed as the cleanest long-term contract.

`deps` should contain only already-resolved ports/collaborators:

- repositories
- actor/vcs/hook services
- schema provider / parsers / helpers
- supporting use cases when composition needs them

`deps` should not contain:

- filesystem paths
- adapter ids such as `'fs'`
- concrete infrastructure classes

## Decision: convenience `createX(config)` should remain

The user agreed that some consumers should still be able to pass config directly.

However, `createX(config)` should no longer hardcode fs wiring internally. It should delegate through the shared resolver and then call `createX(deps)`.

## Decision: future adapter/plugin support should flow through config + registry resolution

Important conclusion from the conversation:

- config should continue to say which adapter the project wants
- builder/options should register additional capabilities
- resolver should cross config selection with available registered capabilities
- builder should not silently override config by default

## Decision: kernel should not be a second source of truth

The user explicitly stated that the kernel should mostly be a “composer of createX”.

Conclusion:

- kernel should not duplicate separate wiring semantics
- kernel may share resolved instances, but should not invent a parallel composition path

## Decision: kernel builder remains only for full-kernel bootstrap

The user clarified that builder is for constructing a complete kernel, not for individual use cases.

Conclusion:

- builder should remain focused on registering capabilities/options for kernel bootstrap
- individual use cases should use `createX(deps)` or `createX(config, options?)`

# Agreements reached

- The refactor is broader than just the current `Fs*Options` cases.
- It should be applied consistently across the public composition surface for use cases.
- All public `createX(...)` factories should move to the same contract model.
- Kernel should become pure orchestration over shared resolved dependencies.
- The change should be created now and the proposal should be used to concretize the design.

# Steps / plan defined

High-level implementation direction discussed during exploration:

1. Define the target contract for all public `createX(...)` factories.
2. Introduce a shared `ResolvedCompositionContext`.
3. Introduce a shared resolver, tentatively referred to as `resolveCompositionContext(config, options?)`.
4. Convert each public composition factory to canonical `createX(deps)` plus convenience `createX(config, options?)`.
5. Add a pure `createKernelFromContext(ctx)` and make `createKernel(config, options?)` delegate to resolver + aggregator.
6. Keep builder as a fluent way to construct options/registrations for full kernel creation only.
7. Migrate incrementally, beginning with representative use cases (`draft-change`, `create-change`, `archive-change`, `get-status`) and then the rest.
8. Update specs/docs to make the new model explicit.

# Rejected alternatives

Rejected or disfavored ideas from the exploration:

- Keeping public `createX(context, Fs*Options)` contracts and just renaming types:
  - rejected as cosmetic; coupling would remain
- Hardcoding more adapter ids inside each `createX(...)`:
  - rejected as unscalable and duplicative
- Letting kernel and public `createX(config)` remain separate composition truths:
  - rejected due to drift risk
- Using builder as a composition path for individual use cases:
  - rejected; builder should stay focused on full-kernel construction
- Letting builder silently override config adapter selection by default:
  - rejected as ambiguous and likely to create two truths about adapter selection

# Key codebase observations

## Application layer is not directly importing fs

Graph and file inspection showed:

- `draft-change.ts`, `restore-change.ts`, and `discard-change.ts` in `application/use-cases` import only ports and application/domain types
- no direct imports from `node:fs` or `infrastructure/fs` in those files

`rg` over `packages/core/src/application/use-cases` found only references to `FsChangeRepository` in JSDoc/comments, not imports.

## Public composition use-case factories are explicitly fs-oriented

Examples:

- `packages/core/src/composition/use-cases/draft-change.ts`
- `packages/core/src/composition/use-cases/restore-change.ts`
- `packages/core/src/composition/use-cases/discard-change.ts`
- `packages/core/src/composition/use-cases/create-change.ts`

These files expose explicit `Fs*Options` interfaces and hardcode calls like:

- `createChangeRepository('fs', ...)`
- `createSpecRepository('fs', ...)`

## Kernel internals already use registry-based storage factories

`packages/core/src/composition/kernel-internals.ts` contains:

- `FS_SPEC_STORAGE_FACTORY`
- `FS_SCHEMA_STORAGE_FACTORY`
- `FS_CHANGE_STORAGE_FACTORY`
- `FS_ARCHIVE_STORAGE_FACTORY`

and a `createBuiltinKernelRegistry()` that registers built-in factories under names like `fs`.

This indicates the architecture already has a more general resolution model internally.

## The public composition surface and internal kernel composition are misaligned

`packages/core/src/composition/kernel.ts` and `kernel-internals.ts` resolve dependencies through a richer registry/bootstrap path.

Many standalone public composition factories still repeat fs-specific wiring manually.

This mismatch is the architectural seam the change is meant to close.

## Current specs/ADRs intentionally allow fs-oriented explicit composition in v1

Loaded specs and ADRs show:

- `core:composition` currently documents explicit `(context, Fs*Options)` signatures
- ADR-0006 confirms fs-only storage in v1
- ADR-0015 and ADR-0018 describe graduated composition levels and note that explicit composition is currently fs-oriented

Therefore, the issue is not “implementation drift against current spec”, but rather that the current spec now looks too concrete for the future adapter/plugin direction.
