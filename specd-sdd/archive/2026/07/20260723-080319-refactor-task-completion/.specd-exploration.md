## Problem Statement

The logic for counting pending tasks (using `safeRegex` on `incompletePattern` and optionally `completePattern` across files of an artifact) is currently duplicated across two separate use cases in the core package:

- `GetStatus` in [get-status.ts](file:///Users/monki/Documents/Proyectos/specd/packages/core/src/application/use-cases/get-status.ts)
- `TransitionChange` in [transition-change.ts](file:///Users/monki/Documents/Proyectos/specd/packages/core/src/application/use-cases/transition-change.ts)

This duplication is prone to drift, compromises maintainability, and increases complexity in two critical lifecycle-controlling areas.

## Proposed Approach

Extract the logic into a new, reusable application-layer use case or service class in the core package.

- **Name suggestion**: `CountTasks` or `GetTaskCompletion` (to be defined during the design phase).
- **Responsibilities**:
  - Take a `Change` object, `artifactId`, and `TaskCompletionCheck` config.
  - Access the [ChangeRepository](file:///Users/monki/Documents/Proyectos/specd/packages/core/src/application/ports/change-repository.ts) to retrieve file contents.
  - Compile the patterns using `safeRegex` and count occurrences.
  - Return structured task counts (e.g. `complete`, `incomplete`, `total`).
- **Dependency Injection**:
  - Register the new use case in the composition layer resolver [CompositionResolver](file:///Users/monki/Documents/Proyectos/specd/packages/core/src/composition/composition-resolver.ts).
  - Inject it into `GetStatus` and `TransitionChange`.

## Design Decisions & Agreements

- **SpecD Workflow**: The refactoring must proceed strictly under the `specd` workflow, keeping specifications up to date.
- **No Behavior Changes**: The external behavior of `GetStatus` and `TransitionChange` must remain completely intact. The refactor must not alter how tasks are parsed or when transitions are blocked, only how that calculation is structured internally.

## Open Questions for `/specd-design`

- Should the extracted logic be a separate use case class (e.g. `CountTasks`) or a shared private query helper? Since both existing classes are use cases and use dependency injection, a new use case class registered in `CompositionResolver` is the most consistent architectural approach.
- What naming convention is preferred for the return type? We can reuse/re-export `TaskCompletionStatus`.
