# Exploration: eliminate-git-hardcoding

## Problem

Several places in the codebase hardcode `git` commands instead of going through the `VcsAdapter` port. This prevents specd from working with non-git VCS (hg, svn) in those code paths.

## Approach / Solution

1. Add `identity(): Promise<ActorIdentity>` method to the `VcsAdapter` port interface
2. Implement `identity()` in `GitVcsAdapter` (reads `git config user.name` + `user.email`), `HgVcsAdapter` (reads `hg config ui.username`), `SvnVcsAdapter` (reads `svn info --show-item last-changed-author`)
3. Update `NullVcsAdapter` with a null identity fallback
4. Refactor `actor-resolver.ts` composition to receive a `VcsAdapter` instance instead of doing its own VCS detection
5. Eliminate the duplicate `git rev-parse` detection in `actor-resolver.ts`
6. Fix `config-loader.ts` to use `VcsAdapter.rootDir()` instead of calling `git rev-parse --show-toplevel`
7. Fix `discover-files.ts` in code-graph to use `VcsAdapter.rootDir()` instead of checking for `.git/` directory

## Design decisions

- `VcsAdapter` gets `identity(): Promise<ActorIdentity>` (not `vcsIdentity()`)
- `ActorResolver` port stays separate but its composition receives the already-detected `VcsAdapter`
- VCS implementations (`GitVcsAdapter`, `HgVcsAdapter`, `SvnVcsAdapter`) are the single source of identity for their respective VCS
- The `NullVcsAdapter` returns a null identity (name: 'unknown', email: '')

## Rejected alternatives

- Eliminating `ActorResolver` port entirely — rejected because identity resolution may come from non-VCS sources (SSO, env vars) in the future
- Keeping duplicate detection in `actor-resolver.ts` — rejected because it violates DRY and runs git detection twice

## Open questions

- What to do with `PrivacyActorResolver`? It wraps `ActorResolver` and anonymizes identity — probably stays as-is since it's a decorator
- Should `core:actor-resolver-privacy` spec be removed from the change? It probably doesn't need modification
