# Exploration Context

## Problem statement

The repository currently mixes read behavior with persisted lifecycle mutation.

In particular, `FsChangeRepository.get()` can detect artifact drift, invalidate the
change, append history events, and write `manifest.json` during an ordinary read.

That behavior was surfaced while investigating the separate change
`fix-project-status-artifact-drift`, where `project status` could push a healthy change
back to `designing`. During that investigation, it became clear that the false drift bug
and the more general "write-on-read" behavior are related but should be handled as
separate changes.

This new change exists to isolate and fix the read/write boundary problem:

- ordinary repository reads should not persist lifecycle invalidation or manifest updates
- if drift or other mutation-worthy state is discovered during a read, that should be
  surfaced explicitly rather than silently written as a side effect of `get()`

## Scope of this change

This change is for the `write-on-read` problem only.

It is intentionally split away from the sibling change:

- `fix-project-status-artifact-drift`

The sibling change is intended to fix repository initialization and factory/kernel
alignment issues, including missing `resolveArtifactTypes` and related composition bugs.

This change should focus on repository read semantics and the boundary between:

- read/projection behavior
- explicit mutation/persistence behavior

Initial attached specs:

- `core:storage`
- `core:change-repository-port`

These were selected because:

- `core:storage` defines fs-backed repository behavior and load-time artifact status
  derivation
- `core:change-repository-port` currently documents the existing auto-invalidation-on-get
  behavior and therefore must change if reads become side-effect free

`core:get-status` was considered but not attached initially, because the core issue here
is repository semantics rather than the `GetStatus` use case contract itself. It can be
added during design if the final solution requires changes at the status projection layer.

## Relationship to the sibling change

The sibling change `fix-project-status-artifact-drift` uncovered two distinct problem
families:

1. factories and direct composition paths building partially initialized repositories,
   leading to false drift
2. `FsChangeRepository.get()` performing persisted invalidation or normalization during
   reads

The user explicitly decided to split those apart:

- the current active change should keep the initialization/factory-alignment work
- this new change should take only the write-on-read behavior

The user also wants the exploration of the first change updated afterward so it clearly
states that write-on-read is no longer in scope there.

## Confirmed current behavior

From the prior investigation:

- `FsChangeRepository.get()` loads the manifest, derives artifact/file state, may sync or
  normalize artifact structure, and may auto-invalidate drifted validated files
- when auto-invalidation triggers, it appends invalidation/transition history and writes
  the manifest before returning

Relevant repository area:

- `packages/core/src/infrastructure/fs/change-repository.ts`

Relevant documented contract:

- `core:change-repository-port`
- `core:storage`

The current `core:change-repository-port` spec explicitly allows and describes
auto-invalidation on `get()`. That is the current source-of-truth behavior that this
change is expected to revise.

## Design direction discussed so far

The preferred direction discussed in conversation is:

- reads should detect and project state
- writes should happen only through explicit mutation flows

In practical terms:

- `get()` / `list()` / `getDraft()` / `getDiscarded()` should ideally be side-effect free
- if drift is discovered during a read, that drift can be surfaced in memory or via status
  projection, but should not silently persist lifecycle invalidation as part of a read
- any persisted invalidation should move to an explicit mutation path, ideally under a
  serialized repository mutation boundary such as `mutate()` or another deliberate
  repository/use-case operation

An important caution was raised during discussion:

- simply "stopping writes" may break existing flows if parts of the system implicitly rely
  on `get()` doing persistence work today

So the intended change is not a blind removal. It needs to identify which read-time side
effects are relied on and replace them with explicit behavior where necessary.

## Key codebase observations

- `packages/core/src/infrastructure/fs/change-repository.ts` is the main implementation
  target for `get()` behavior
- `core:change-repository-port` currently defines `get()` as a snapshot read that MAY
  auto-invalidate and persist drift before returning
- `core:storage` currently defines artifact status derivation at load time, which likely
  remains relevant even if persisted invalidation moves elsewhere
- the sibling bug showed that repository read behavior is currently visible through
  user-facing commands such as `project status`
- there is already an explicit invalidation surface elsewhere in the system:
  `core:invalidate-change` and `cli:change-invalidate`, which may be relevant as a model
  for explicit persisted reopening behavior

## Rejected or deferred alternatives

- Keeping write-on-read inside `fix-project-status-artifact-drift`
  Rejected because the user wanted it split into a separate change with narrower intent.

- Blindly removing all read-time writes without replacement
  Rejected in discussion as too risky, because some current flows may implicitly depend on
  that behavior.

- Treating `GetStatus` itself as the primary root of the bug
  Deferred/rejected for this change's initial scope; the main issue is repository read
  semantics, not status projection alone.

## Open questions for /specd-design

1. Should `get()` become fully side-effect free, or are there narrow normalization cases
   that still need to persist outside explicit mutation paths?
2. If drift is detected during read, where should persisted invalidation move?
   Options include:
   - repository `mutate()`-based helper
   - a dedicated repository reconcile operation
   - a use-case-level explicit invalidation path
3. Which current callers rely on read-time persisted invalidation or manifest
   normalization?
4. Should spec changes explicitly distinguish:
   - derived read-time status/projection
   - persisted lifecycle invalidation
5. Does `core:get-status` need to be added to the change once the design is clearer?

## User preferences and constraints

- The user wants this separated cleanly from the sibling change
- The user wants the sibling change to focus on fixing bugs and aligning factories with
  kernel wiring
- The user does not want this change to be a casual patch; it should handle the read/write
  boundary deliberately

## Tangential topics discussed

- Whether kernel and direct factories should share helpers or whether factories should be
