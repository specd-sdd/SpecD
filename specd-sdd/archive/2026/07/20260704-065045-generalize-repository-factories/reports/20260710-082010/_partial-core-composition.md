## Global Architectural Consistency

### Hexagonal Architecture & ESM Conventions

1. **ESM Compliance**:
   - Every workspace package uses `"type": "module"` in `package.json`.
   - Local import paths explicitly resolve using relative `.js` extensions.
2. **Ports & Adapters Boundaries**:
   - Domain and Application modules do not reference Composition or Infrastructure components directly.
   - Domain entities (such as `Change` and `Spec`) strictly protect their invariants.
3. **Partitioned Barrel Exports**:
   - `packages/core/package.json` maps exports to separate subpaths: `"."` (Public factories), `"./ports"` (Core Interfaces), `"./extensions"` (Extension factory signatures), and `"./internal"` (internal logic).
   - Public entrypoint ([public.ts](file:///Users/monki/Documents/Proyectos/specd/packages/core/src/public.ts)) does _not_ leak concrete filesystem repository classes, exporting only factories and interfaces.

## Compliance Gaps Summary & Recommendations

### Gap 1: Ignored configuration options in `CompositionResolver` storage resolution

- **Problem**: In [composition-resolver.ts:L360-L390](file:///Users/monki/Documents/Proyectos/specd/packages/core/src/composition/composition-resolver.ts#L360-L390), `getChangeRepository()` only passes `{ path: config.storage.changesPath }` as configuration. It ignores `...config.storage.changesAdapter.config`, `drafts: config.storage.draftsAdapter.config`, and `discarded: config.storage.discardedAdapter.config`.
- **Problem**: In [composition-resolver.ts:L392-L414](file:///Users/monki/Documents/Proyectos/specd/packages/core/src/composition/composition-resolver.ts#L392-L414), `getArchiveRepository()` ignores `...config.storage.archiveAdapter.config`, `changes: config.storage.changesAdapter.config`, and `drafts: config.storage.draftsAdapter.config`.
- **Problem**: In [composition-resolver.ts:L416-L446](file:///Users/monki/Documents/Proyectos/specd/packages/core/src/composition/composition-resolver.ts#L416-L446), `getSpecRepositories()` ignores any additional properties in `workspace.specsAdapter.config` other than `path`.
- **Impact**: Callers attempting to configure custom storage adapters (or custom configuration parameters on built-in storage adapters) will find their options silently ignored when dependencies are constructed via the `CompositionResolver` (which backing the kernel and builder).
- **Fix**: Update the repository construction in `composition-resolver.ts` to mirror the logic previously defined in `createKernelInternals` by spreading the configuration blocks.
  ```typescript
  // getChangeRepository
  const repoConfig = {
    ...config.storage.changesAdapter.config,
    drafts: config.storage.draftsAdapter.config,
    discarded: config.storage.discardedAdapter.config,
    path: config.storage.changesPath,
  }
  ```

### Gap 2: Dead Code in `packages/core/src/composition/kernel-internals.ts`

- **Problem**: The module [kernel-internals.ts](file:///Users/monki/Documents/Proyectos/specd/packages/core/src/composition/kernel-internals.ts) is fully replaced by the `CompositionResolver` in all use case constructions. However, the file remains in the codebase with zero active dependents.
- **Impact**: Dead code adds to maintenance overhead and causes confusion.
- **Fix**: Safe to delete `kernel-internals.ts` and its test suite `kernel-internals.spec.ts` once the config options gap on `CompositionResolver` is resolved.
