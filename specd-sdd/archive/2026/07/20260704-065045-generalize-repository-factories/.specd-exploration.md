# Future Change Exploration

## Problem Statement

Track a future refactor for repository composition factories that still expose and implement hardcoded filesystem adapter dispatch such as:

- `createChangeRepository('fs', config, options)`
- `createSpecRepository('fs', config, options)`
- `createArchiveRepository('fs', config, options)`
- `createSchemaRepository('fs', config, options)`

The concern raised during implementation of `decouple-composition-factories` is that use-case composition is being decoupled from fs, but repository-level composition still keeps an adapter-discriminated public API and hardcoded `'fs'` dispatch in core composition. This may become the next blocker when external adapter plugins need to participate cleanly.

## Why This Exists As A Separate Change

- The current in-flight change `decouple-composition-factories` is focused on use-case composition factories, kernel orchestration, dual factory signatures, dependency resolution, and composition resolver behavior.
- During implementation/verification, a related but distinct issue was re-confirmed: repository factories are still not fully aligned with the future plugin-extensible composition model.
- This was intentionally split out as a future change to avoid expanding the current refactor's scope while preserving the architectural follow-up.

## Approach / Solution Outline

- Revisit repository-level composition factories so adapter resolution is not effectively hardcoded behind public `'fs'`-based factory entrypoints.
- Align implementation with the already-declared contract in `core:composition`, which says repository factories accept an adapter id and are designed for future plugin-registered adapters through `*StorageFactory` registration on `./extensions`.
- Decide whether public repository composition should remain:
  - adapter-id driven (`createChangeRepository(adapterId, config, options)` with real registry dispatch), or
  - resolver-driven / kernel-driven with adapter-specific creation hidden from the main public surface.
- Ensure any decision remains coherent with:
  - `createKernel`
  - `createKernelBuilder`
  - plugin adapter registration
  - `@specd/sdk` curated public re-exports for hosts

## Key Spec Observation

`core:composition` already states:

- repository factories must be exported on the public root
- each repository factory must accept an adapter id as its first argument
- builtin `'fs'` must be supported
- future plugin-registered adapter ids are resolved through `*StorageFactory` registration on `./extensions`
- full registry dispatch wiring in factories was explicitly out of scope for the initial barrel split, but public signatures must be designed for extensibility

This future change should likely close that implementation gap or revise the contract if the team decides the public adapter-id path is the wrong abstraction.

## Design Decisions Already Reached

- Do not mix this work into `decouple-composition-factories`.
- Treat this as a follow-up architectural refactor, not a bugfix inside the current change.
- Avoid attaching specs during creation to reduce unnecessary overlap with the current active change; add them during design once the exact scope is chosen.
- Preserve the distinction between:
  - use-case composition refactor now
  - repository/adapter dispatch refactor later

## Agreements Reached

- The user explicitly wanted a separate future change created so this concern is not lost.
- The future change is about the still-hardcoded repository factory adapter dispatch, not about redoing the just-implemented use-case composition contract.
- Kernel should remain a convenience/orchestration layer, not a second source of truth.
- Any future repository-factory design must remain coherent with plugin-based adapters and builder/kernel composition.

## Rejected / Deferred Alternatives

- Expanding `decouple-composition-factories` to also remove repository-factory `'fs'` dispatch.
  - Rejected for now because it would broaden scope significantly during implementation/verification of the current change.
- Forcing immediate spec scope overlap by attaching `core:composition` and `sdk:composition` now.
  - Deferred to design stage to keep the new change as a future placeholder without introducing unnecessary overlap noise during creation.

## Open Questions

- Should repository factories remain public adapter-id factories at all?
- If yes, should they dispatch through a real registry of `*StorageFactory` instances instead of switch branches?
- If not, should adapter-specific wiring move behind resolver/kernel/builder-only APIs while keeping only higher-level contracts public?
- How should adapter-specific options be modeled for non-fs plugins if the public factory remains adapter-id based?
- Should `@specd/sdk` continue re-exporting repository factories unchanged, or should host-facing guidance push consumers toward higher-level composition paths?
- Does this refactor warrant a new ADR if the public composition model changes materially?

## User Preferences

- Keep architecture coherent around one source of truth.
- Prefer kernel as orchestration/facility, with reusable `createX` functions holding the logic.
- Avoid duplicated truth between kernel and create paths.
- Surface future plugin adapter concerns early instead of leaving hidden fs assumptions in public composition APIs.
