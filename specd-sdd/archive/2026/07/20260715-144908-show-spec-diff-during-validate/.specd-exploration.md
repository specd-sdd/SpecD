# Exploration Context

## Problem statement

The user wants to improve `changes validate` now that core already has diff support around spec previews.

Requested behavior:

- when validating an artifact with `scope: spec`
- when the validation is an individual validation (not `--all`)
- when the target artifact is a delta against an existing spec artifact (not a new spec file)

`specd changes validate` should show the diff directly as part of validation output, so the reviewer can immediately inspect what changed.

The user explicitly does **not** want this for `--all`; batch validation should continue behaving as it does today.

The user also wants the skills in the `packages/skills` package updated so they no longer tell the agent to run `spec-preview` after validate in this case. Instead, skill guidance should instruct the agent to review the diff shown by `validate` to ensure prior contracts are not broken and nothing important was removed.

## Clarified intent

There was an initial ambiguity: the user first said this should apply when validating an artifact with `scope: change` and "it is a delta (not new)".

After technical review of the current system, this was clarified by the user:

- they actually meant `scope: spec`

This clarification is important because the current architecture already has diff/merged-preview semantics for spec-scoped deltas, while change-scoped artifacts do not have an equivalent preview use case today.

## Scope boundaries agreed in conversation

In scope:

- single validation flow only
- `scope: spec` artifacts only
- existing spec artifacts represented as deltas against a base artifact
- CLI output changes in `changes validate`
- supporting core behavior if needed to supply diff data cleanly
- skills/package workflow guidance updates so review happens from validate output

Out of scope:

- `--all` behavior changes
- change-scoped artifact preview/diff support
- introducing delta-backed behavior for `proposal`, `design`, or `tasks`
- autonomous lifecycle progression beyond creating this change

## Proposed solution outline

The intended product behavior is:

1. `specd changes validate <name> <specId> --artifact <artifactId> --format text`
2. validation still remains a structural/state gate
3. if the artifact is spec-scoped and is an existing delta-backed artifact, validate output should also surface the diff directly
4. that diff becomes the immediate review surface for the operator/agent
5. skills should tell the agent to inspect that diff for contract breakage or unintended removals, instead of requiring a follow-up `spec-preview` command in the common individual-validation path

Likely non-goals:

- do not reinterpret validate success as semantic approval
- do not remove `spec-preview` as a capability; it still remains useful outside this narrower path

### Current CLI validate behavior

From `packages/cli/src/commands/change/validate.ts`:

- validate already prints a structural-review warning:
  - `note: validation is structural; review artifact content separately before relying on it`
- for spec-scoped targets, it already emits guidance pointing to:
  - `specd changes spec-preview <name> <specId>`
- for spec-scoped single-artifact validation, it can already narrow the guidance to:
  - `specd changes spec-preview <name> <specId> --artifact <artifactId>`
- change-scoped validations intentionally do not emit preview guidance

This means the current CLI already recognizes the distinction between:

- structural validation
- separate merged-content review

The requested change is to tighten that UX by surfacing the diff directly during the individual validation path for spec deltas.

### Current preview architecture

From `packages/core/src/application/use-cases/preview-spec.ts`:

- `PreviewSpec` only works on `scope: spec` artifacts
- it walks schema artifacts and skips non-spec artifacts
- it understands:
  - base content
  - merged content
  - `no-op`
  - `missing`
  - optional unified diff generation via `includeDiff: true`

This is the strongest architectural fit for the requested feature. It already models the exact concept the user wants to expose during validate for spec-scoped delta artifacts.

### Current validate architecture

From `specs/cli/change-validate/spec.md` and `packages/cli/src/commands/change/validate.ts`:

- the CLI contract explicitly states that validate is structural/state validation only
- it currently recommends `spec-preview` as a follow-up review step for spec-scoped validation
- `--all` uses a batch driver and aggregates scheduled steps

This reinforces the requested scope boundary:

- individual validate path may gain richer review output
- batch path should remain unchanged

### Current skill guidance

From `packages/skills/templates/shared/shared.md.tpl`:

- skill guidance says validate is structural only

- current guidance points to `specd changes spec-preview`

This suggests skill updates must be precise rather than broad:

- replace the post-validate `spec-preview` requirement only where validate itself now shows the relevant diff
- preserve `spec-preview` guidance where the workflow still needs merged content outside the narrow validate path

### Important modeling constraint discovered

During investigation, it was verified that current schema-standard change-scoped artifacts are direct files:

- `proposal.md`
- `design.md`
- `tasks.md`

They are not delta-backed like spec-scoped `spec.md` / `verify.md`.

This is why the user clarification from `scope: change` to `scope: spec` materially simplified the feature and aligned it with existing architecture.

## Design decisions already made

These decisions were either explicit or strongly implied and accepted during discussion:

- the feature should apply only to single validation, not batch validation
- the feature should target `scope: spec`, not `scope: change`
- the diff should be shown when validating delta-backed spec artifacts against an existing base
- new spec artifacts should not be treated the same way as delta-backed existing artifacts
- skills should instruct agents to review the diff shown by validate for unintended removals or contract breakage
- validate remains structural; showing a diff does not turn it into semantic approval

## Rejected or corrected alternatives

### Initial `scope: change` interpretation

Rejected due to user clarification and architecture mismatch.

Reason:

- current change-scoped artifacts do not use the same delta-preview model
- existing diff support is centered on `PreviewSpec`, which is spec-scoped
- extending the same concept to change-scoped artifacts would be a deeper design change than the user intended

### Broad `--all` diff output

Rejected by user intent.

Reason:

- the user explicitly wants `--all` to remain as it is now
- batch output would likely become noisy and less readable if per-artifact diffs were injected into the aggregated validation flow

## Open questions for specd-design

These are the main questions the design artifact should settle:

1. Should `changes validate` render the diff itself in CLI text mode by calling preview-with-diff, or should core validation return enough structured review metadata that the CLI can render without a second use-case call?
2. What exact condition determines "is a delta (not new)" in the single-validation path?
   - likely tracked filename ends with `.delta.yaml`
   - likely requires existing base artifact in the spec repository
3. Should the diff be shown only on success, or also on failure when validation reaches a state where merged diff can still be computed?
4. Should the diff be full unified diff or artifact-filtered unified diff only for the requested artifact? Likely artifact-filtered only.
5. Should JSON/TOON output also carry diff fields for this path, or is the enhancement text-mode only?
6. Which skill templates should change, and which ones should retain `spec-preview` guidance because they are not specifically in the "just validated a single spec delta" path?
