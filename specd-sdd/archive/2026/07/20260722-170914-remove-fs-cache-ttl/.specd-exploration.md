# Exploration: remove-fs-cache-ttl

## Problem statement

The fs-cache list indexes (`FsIndexCache` and bucket wrappers) currently apply a fixed
5-minute TTL (`INDEX_TTL_MS = 300_000`) as a third freshness check after invalidation
flag and mtime comparison. The user believes this TTL is unnecessary given the current
freshness model and wants it removed.

## Approach / solution outline

Remove the TTL step from the freshness sequence. The model becomes:

1. If `isInvalidated` → rebuild
2. Else if mtime/sourceFiles stamps mismatch → rebuild
3. Else serve from index

Implementation targets:

- `packages/core/src/infrastructure/fs/fs-index-cache-base.ts` — remove `INDEX_TTL_MS`,
  remove TTL branch in `_ensureFresh()`, update class doc comment
- `packages/core/test/infrastructure/fs/fs-index-cache-base.spec.ts` — remove
  `'rebuilds when TTL expires'` test

- `core:fs-archive-repository` — same as spec repo

`core:storage` does NOT mention TTL explicitly; no delta needed unless design adds
a cross-cutting freshness note (optional).

## Design decisions (agreed / proposed)

- **Remove TTL entirely** — not make it configurable or increase duration
- **Keep `generatedAt` in meta** — still useful for debugging/observability; only the
  TTL check goes away
- **Keep invalidation + mtime checks** — these are sufficient for correctness
- **External changes** — covered by mtime scan on every list/count; explicit
  `invalidateCache()` remains for callers that know storage changed outside repo paths

## Rationale (why TTL is redundant)

1. Every `list()`/`count()` already runs `_isStale()` which scans disk and compares
   mtimes (`currentStamps()` callback per bucket).
2. Write paths maintain index via upsert/invalidate on save, create, delete, moves.
3. `invalidateCache()` exists on all fs repositories for external/manual workflows.
4. TTL only forces rebuild when stamps match but age > 5 min — costly and rare benefit
   (mtime-preserving edits like `touch -r`).

## Rejected alternatives

- **Increase TTL** — doesn't address redundancy; just reduces frequency of wasted rebuilds
- **Make TTL configurable** — explicitly rejected in original normalize-repository-caches
  change; still unnecessary complexity
- **Keep TTL as safety net** — user explicitly wants removal after analysis
