## Project Overview

**specd** (Spec-Driven Development) is a platform for AI-native teams that enforces spec-driven workflows. It compiles the specs an AI agent needs at every lifecycle step — so context is delivered, not discovered, and compliance is structural, not hoped for.

The project is in **active development** (March 2026) and fully usable from source in the monorepo. Core packages (`@specd/core`, `@specd/cli`, `@specd/code-graph`, `@specd/schema-std`) are functional. The MCP server, plugin packages, and `@specd/skills` are scaffolded or in redesign.

## Architecture Pattern

**Hexagonal Architecture (Ports & Adapters)** with DDD tactical patterns.

- `domain/` — pure TypeScript, zero I/O. Entities, value objects, domain services, errors.
- `application/` — use cases + port interfaces. No I/O; all external concerns go through ports.
- `infrastructure/` — concrete port implementations (fs, git, hg, svn, node, null).
- `composition/` — factory functions for DI wiring. Only entry points call these.
- Delivery mechanisms (`@specd/cli`, `@specd/mcp`, plugins) are independent adapters.

**Key enforced invariants (via ESLint `no-restricted-imports`):**

- `domain/` must not import from `application/`, `infrastructure/`, or `composition/`
- `application/` must not import from `infrastructure/` or `composition/`
- `infrastructure/` must not import from `composition/`

## Cross-Cutting Concerns

### Error Handling

- All domain and application errors extend `SpecdError` (abstract base with `code: string`)
- 21 concrete error classes in `core/src/domain/errors/`
- CLI maps errors to exit codes: `1` = domain/user error, `2` = hook failure, `3` = system/schema/unexpected
- Structured JSON errors on stdout for `--format json|toon`; plain text to stderr always

### Output and Formatting

- Three output formats: `text` (human-readable), `json` (machine-readable), `toon` (LLM-friendly Token-Oriented Object Notation)
- All output routed through `output()` in `cli/src/formatter.ts` (score: 221 — the most-called symbol in the CLI)
- Format parsed and validated via `parseFormat()` before every command

### Testing

- Domain and application layers: unit tests with mocked ports (no real filesystem)
- Infrastructure adapters: integration tests against temporary real filesystems
- Test files live in `packages/*/test/`, mirroring `src/` structure
- Pattern: `*.spec.ts` files using Vitest
- No CI pipeline found — tests run via `pnpm test` + lint-staged pre-commit

### Conventions (from ESLint config + specs)

- No default exports — named exports only (`ExportDefaultDeclaration` is forbidden)
- No public class fields — use methods or getters (`PropertyDefinition` without `private|protected` is forbidden)
- No `any` — `@typescript-eslint/no-explicit-any: error`
- Explicit module boundary types on all exports
- Kebab-case filenames for all source files
- JSDoc required on all functions, classes, methods, type aliases, and interfaces in `src/`
- All `@param` and `@returns` tags must have descriptions; `@throws` required where applicable

### Spec-Driven Workflow

- The project uses specd to govern its own development
- `specd.yaml` configures 4 workspaces: `default` (\_global specs), `core`, `cli`, `code-graph`
- Global specs (`default:_global/*`) are always included in context for every change
- Commit hooks run `pnpm test` and `pnpm lint` on transition to implementing and at archiving

## Security Model

- No authentication or authorisation mechanisms (specd is a local developer tool, not a server)
- `SpecPath` validates and rejects path traversal attacks at construction time (`..`, reserved chars)
- `PathTraversalError` is a dedicated domain error for traversal attempts
- Actor identity is resolved from VCS config (git config name/email) — no credentials stored by specd
- `read-only` workspace ownership enforces write prohibition at the use-case level

---

## Performance Constraints

- Schema loading: three-level lookup (project-local → user override → npm package); cached via `LazySchemaProvider`
- Spec context compilation: uses pattern matching + `dependsOn` traversal; lazy mode (default) limits full rendering to tier-1 specs only
- Code graph: LadybugDB embedded graph database; incremental indexing via content hashing; last indexed at `82a96a6`
- File writes: `writeFileAtomic()` for all critical writes (manifest, metadata) to prevent corruption
