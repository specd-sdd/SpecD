# Agent: specd Bootstrap

> Temporary working document. Final location TBD.

## What this agent does

Analyzes an existing repository and generates a `specd-bootstrap.md` file at the repo root.
This file gives specd and the LLM enough context to start working with the project without
existing specs — the cold-start problem when adding specd to an established codebase.

## When to run

- When adding specd to an existing project for the first time
- When specd has no specs and no metadata yet
- Analogous to `claude /init` or GitHub Copilot workspace initialization

## Instructions

When this agent is invoked:

1. Ask the user for the project root path if not already known.

2. Launch a subagent using the Task tool with:
   - `subagent_type: general-purpose`
   - `model: haiku`
   - The full prompt below, substituting `<project-root>` with the actual path.

3. The subagent has Write access — it will create `specd-bootstrap.md` at `<project-root>`.

## Subagent prompt

```
You are analyzing a repository to generate specd-bootstrap.md for the project at <project-root>.

This file is the cold-start context for specd — it gives the LLM enough understanding of the
project to begin creating specs without any existing ones. Aim for useful and accurate over
exhaustive. It is a starting point, not a finished product.

### Step 0 — Create progress tasks

Create these tasks before doing any other work:
- "Explore repo structure" (activeForm: "Exploring repository structure")
- "Read key files" (activeForm: "Reading key files")
- "Identify domains" (activeForm: "Identifying domains and modules")
- "Write specd-bootstrap.md" (activeForm: "Writing bootstrap file")

Mark each in_progress when you start it and completed when done.

### Step 1 — Explore repo structure

Mark "Explore repo structure" in_progress.

List the top-level directory structure. Identify:
- Is this a monorepo? If so, what are the packages/workspaces?
- What language/runtime? (look for package.json, Cargo.toml, pyproject.toml, go.mod, etc.)
- What build tooling? (turbo, nx, lerna, gradle, make, etc.)
- What test framework?
- Are there existing docs, ADRs, or architecture notes?

Mark "Explore repo structure" completed.

### Step 2 — Read key files

Mark "Read key files" in_progress.

Read the following files if they exist (do not error if absent):
- README.md (or README.rst, README.txt)
- package.json / Cargo.toml / pyproject.toml / go.mod (root level)
- Any file named ARCHITECTURE.md, CONTRIBUTING.md, DESIGN.md, AGENTS.md, CLAUDE.md
- .github/copilot-instructions.md
- Any existing specd.yaml or .specd/ directory contents

Also read 2–3 representative source files from the core domain (not tests, not config) to
understand naming conventions, patterns, and domain vocabulary.

Mark "Read key files" completed.

### Step 3 — Identify domains

Mark "Identify domains" in_progress.

From everything read, identify:
- The primary purpose of the project (1–2 sentences)
- The **architectural pattern** — hexagonal, layered, clean, MVC, CQRS, microservices,
  modular monolith, or mixed/unclear. Look for directory names like domain/, application/,
  infrastructure/, adapters/, ports/, controllers/, services/, repositories/, usecases/.
- The key domain concepts (the nouns that matter: entities, aggregates, services)
- The main modules or packages and what each is responsible for
- Existing conventions: naming style, file structure, test patterns, commit style
- External dependencies that are load-bearing (databases, queues, auth providers, etc.)
- Any explicit constraints or invariants found in docs or code comments

Mark "Identify domains" completed.

### Step 4 — Write specd-bootstrap.md

Mark "Write specd-bootstrap.md" in_progress.

Write <project-root>/specd-bootstrap.md with the following structure:

---

<2–3 sentences: what this project does, who it's for, why it exists>

- **Test**: <vitest/jest/pytest/etc>
- **Key deps**: <2–5 most important external dependencies>

## Architecture

**Pattern**: <hexagonal / layered / clean / MVC / CQRS / mixed / unclear>

<1–2 sentences on how the codebase is structured and what that means for specs>

## Repository structure

<monorepo or single package>

<for monorepo: one line per package with its responsibility>
<for single: top-level dirs with their purpose>

## Domain concepts

<The key nouns and what they mean in this project. One bullet per concept.>

## Conventions

- <naming convention for files/classes/functions>
- <import style>
- <commit style if detectable>
- <test location pattern>
- <any other patterns worth noting>

<Any hard rules found in docs, comments, or code. If none found, omit this section.>

## Suggested specd structure

<list of suggested spec paths, one per identified domain/module>

Example:
- `core/user` — user entity, registration, authentication
- `core/billing` — subscription lifecycle, invoicing
- `_global/conventions` — coding conventions for this repo

## External dependencies worth speccing

<Services, APIs, or infrastructure that have strong contracts worth capturing as specs.
If none obvious, omit this section.>

Mark "Write specd-bootstrap.md" completed.

## Notes
```
